# MiniApp 用户可见错误文案与恢复动作规范 v1（2026-03-09）

## 1. 文档目标
- 建立“错误码 -> 用户文案 -> 恢复动作”单一映射，禁止 message 分支。
- 强化降级文案：`degraded` 可解释但不可伪成功。

## 2. 强约束
1. 前端分支条件必须是 `errorCode` / `degraded` / `degradeReason`，不得解析 message。
2. 所有阻断错误必须给明确恢复动作：`重试` / `刷新` / `联系客服`。
3. `degraded=true` 场景禁止显示“成功”“已完成”等终态文案。
4. 成功动效只能在后端成功态确认后展示。

## 3. 错误码文案映射（执行版）

## 4. 降级文案规范（degraded 不伪成功）

| 降级信号 | 用户文案（示例） | 恢复动作 | 禁止文案 |
|---|---|---|---|
| `degraded=true` + `PAY_ORDER_NOT_FOUND` | `支付结果确认中，请稍后刷新。` | `刷新`（最多3次） -> `查看订单` -> `联系客服` | `支付成功`、`已完成支付` |
| `degraded=true` + `TICKET_SYNC_DEGRADED` | `当前服务记录同步中，不影响主要操作。` | `继续当前流程` + `稍后查看进度` | `全部处理完成` |
| `degraded=true`（通用） | `当前为降级展示，部分信息可能延迟。` | `刷新` + `返回上页` + `联系客服` | `操作已完全成功` |

## 5. RESERVED_DISABLED 错误码文案（若误返回）
- 适用：`1008009901/1008009902/1008009903/1004009901/1008009904/1011009901/1011009902/1013009901/1013009902/1030009901`
- 统一用户文案：`当前功能配置处理中，请稍后再试。`
- 恢复动作：`刷新` -> `联系客服`。
- 内部动作：按 P1 配置异常上报，不让用户承担定位成本。

## 6. 页面落地检查清单
- [ ] 每个错误态都可见错误码。
- [ ] 每个错误码至少一个主恢复动作。
- [ ] 每个降级态都不展示成功终态文案。
- [ ] 客服话术与本映射一致（不得另写 message 分支话术）。
