# MiniApp 社区门店私域专题：到店服务过程中的公转私触点 PRD v1（2026-03-23）

## 0. 文档定位
- 目标：定义用户到店前、到店中、到店后的关键触点，如何把交易用户转成可持续运营的私域用户。

## 1. 当前真值
- 已有：`Booking` 查询、创建、取消、加项主链；评价提交与结果页；订单详情等页面。
- 已有后台治理：booking review 的店长通知与后台治理链，但这是“门店治理”，不是“用户私域运营”。
- 当前缺口：没有用户侧“服务中/服务后加企微、入社群、领福利、做任务”的统一触点设计。

## 2. 触点设计

| 阶段 | 触点 | 当前状态 | 目标动作 |
|---|---|---|---|
| 到店前 | 预约成功页 / 到店提醒 | 部分存在 | 引导领取到店礼包、绑定门店归属 |
| 到店中 | 核销后服务页 | 未独立实现 | 展示门店福利、加企微、服务项目二次推荐 |
| 服务后 | 评价完成页 / 订单完成页 | 已有订单与评价页 | 引导加入社群、领取复购券、订阅下次服务 |
| 售后后 | 售后完成页 | 已有售后链 | 对不满意用户给修复与复购补救，而不是硬推私域 |

## 3. 推荐业务动作
1. 首次到店完成后展示“门店专属恢复包 / 次卡 / 复购券”。
2. 服务后评价完成，引导加入门店企微或社区群。
3. 针对高满意用户，优先引导社群与邀请裂变。
4. 针对低满意用户，优先引导客服修复，不做强私域转化。

## 4. 规则边界
- 公转私触点必须基于真实交易或服务行为，不做纯弹窗骚扰。
- 低评分/售后中的用户不能直接走“拉群促销”话术。
- 当前没有真实用户侧企微承接页前，不允许写成“到店即自动入私域”。

## 5. 开发建议
- P0：预约成功页 / 评价完成页补私域承接卡片。
- P0：订单完成页补“门店福利 + 下次预约 + 加企微”组合入口。
- P1：服务中触点页与核销后权益页。
- P1：满意度分层后的差异化公转私策略。

## 6. 验收口径
- [ ] 至少有 2 个真实交易后触点进入私域承接页。
- [ ] 差评/售后用户不会被误导进强促销链路。
- [ ] 触点必须能归因到门店和订单。
