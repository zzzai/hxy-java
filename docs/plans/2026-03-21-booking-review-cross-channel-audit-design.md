# Booking Review Cross-Channel Audit Design（2026-03-21）

## 1. 背景
- booking review 当前已经具备：
  - notify outbox 双通道出站
  - 阻断诊断与修复建议
  - 店长路由只读核查页
  - 台账 `notifyRiskSummary`
- 当前剩余的问题是：
  - 详情页虽然展示了 outbox 表，但“最新通知状态”只盯一条记录
  - 值班人员仍要自己把 `IN_APP / WECOM` 两条链路拼成整体判断
  - 不能一眼回答“这条差评整体是双通道阻断、单通道失败，还是双通道都已派发”

## 2. 目标
- 在不新增后端接口、不改变 outbox 状态机的前提下，为 booking review 详情页补一层跨通道审计聚合展示。
- 固定三个展示结果：
  - `双通道摘要`
  - `App 通道卡片`
  - `企微通道卡片`
- 继续保持结论：`Doc Closed / Can Develop / Cannot Release`

## 3. 不做的内容
1. 不新增后端聚合接口
2. 不改 notify outbox 派发逻辑
3. 不做自动补发
4. 不做区域负责人升级
5. 不把当前聚合块写成“真实送达闭环”

## 4. 方案比较

### 方案 A：继续只展示列表
- 优点：零改动
- 缺点：运营仍需要自己拼装双通道状态
- 结论：不采用

### 方案 B：前端 helper 聚合当前 outbox 列表
- 优点：
  - 范围最小
  - 不改后端
  - 规则可在详情页稳定复用
- 缺点：
  - 聚合逻辑需要前端测试兜底
- 结论：采用

### 方案 C：新增后端聚合接口
- 优点：接口更“正统”
- 缺点：当前收益不够，容易把简单展示做成重接口工程
- 结论：暂不采用

## 5. 核心设计

### 5.1 双通道摘要
- 聚合当前评价下的 outbox 列表，按 `channel` 取每个通道最新记录
- 摘要口径优先使用：
  - `review.notifyRiskSummary`
  - 若缺失，再由前端 helper 按通道状态推导
- 同时展示四类计数：
  - 已发送
  - 待派发
  - 发送失败
  - 路由阻断

### 5.2 通道卡片
每个通道固定显示：
- 通道名称
- 当前状态
- 接收账号
- 诊断结论
- 最近动作
- 动作人
- 修复建议

若当前通道没有记录，明确写：
- `未核出当前通道记录`

### 5.3 页面位置
- 放在详情页“通知观测”块内
- 结构为：
  - 顶部：双通道摘要
  - 下方：`App 通道`、`企微通道` 两张卡片
  - 再下方：原始 outbox 明细表

## 6. 测试策略
1. helper test：
- 双通道阻断聚合
- 单通道失败聚合
- 缺失通道记录聚合
2. page structure test：
- 详情页存在 `双通道摘要 / App 通道 / 企微通道`
- 页面引用聚合 helper

## 7. No-Go
1. 不得把双通道摘要写成真实送达回执
2. 不得把前端聚合块写成后端单一真值接口
3. 不得把当前 admin-only 审计增强写成 release-ready

## 8. 当前结论
- 这是详情页观测增强，不是消息系统升级。
- 当前口径继续固定为：`Doc Closed / Can Develop / Cannot Release / No-Go`
