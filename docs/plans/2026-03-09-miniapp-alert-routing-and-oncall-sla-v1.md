# MiniApp Alert Routing and Oncall SLA v1 (2026-03-09)

## 1. 目标
- 统一告警路由和值班SLA，确保“谁先接、谁处置、谁闭环”明确可执行。
- 适用告警：经营指标告警、发布门禁告警、DQ告警、合规/假成功告警。

## 2. 告警路由图（职责流）

```text
[监控系统]
   |
   |-- 交易/履约/售后类 -> [技术值班] -> [域负责人] -> [发布负责人]
   |
   |-- 经营KPI类 -> [运营值班] -> [经营Owner] -> [业务负责人]
   |
   |-- 合规/假成功类 -> [合规值班] -> [产品负责人] -> [技术负责人]
   |
   |-- 门店经营异常 -> [门店运营] -> [区域经理] -> [经营负责人]
```

## 3. P0/P1/P2 响应时限与升级链路

| 等级 | 典型场景 | 首响SLA | 缓解SLA | 关闭SLA | 升级链路 |
|---|---|---|---|---|---|
| P0 | 假成功动效、DQ BLOCK、关键发布门禁失败 | 5分钟 | 15分钟 | 4小时 | 值班 -> 域负责人 -> 技术/业务负责人 |
| P1 | 交易成功率/履约率/退款SLA跌破阈值 | 15分钟 | 1小时 | 24小时 | 值班 -> 域负责人 -> 发布负责人 |
| P2 | 趋势预警、TopN集中、轻微口径漂移 | 30分钟 | 4小时 | 72小时 | 指标Owner -> 域负责人 |

## 4. 告警分类与路由责任

| 告警类别 | 规则样例 | 路由主责 | 协同角色 |
|---|---|---|---|
| 交易链路 | 支付成功率下降、降级率上升 | 技术值班 | 运营/产品 |
| 履约售后 | 履约超时、退款SLA下滑 | 运营值班 | 技术/门店 |
| 经营异常 | ROI负值、毛利异常 | 经营Owner | 财务BP/运营 |
| 合规风险 | 风险词、误导营销、假成功 | 合规值班 | 产品/技术 |
| 数据质量 | 唯一性/关联/金额/时序/幂等异常 | 数据治理值班 | 域负责人 |

## 5. 工单闭环字段（强制）
- 每张工单必须包含：
  - `runId`
  - `orderId`
  - `payRefundId`
  - `sourceBizNo`
  - `errorCode`
- 附加建议字段：`storeId/route/experimentId/releaseId/riskLevel`。

## 6. 工单生命周期
1. `Open`：告警触发自动建单。
2. `Ack`：值班确认并分派。
3. `Mitigating`：执行限流/回滚/重试/配置修复。
4. `Resolved`：恢复指标至阈值内。
5. `Postmortem`：48小时内完成复盘结案。

## 7. 升级触发规则

| 触发规则 | 升级动作 |
|---|---|
| P0 超 15 分钟未缓解 | 自动升级到技术负责人+业务负责人 |
| P1 超 1 小时未缓解 | 升级到发布负责人 |
| 同类告警 24 小时内重复 >=3 次 | 升级为系统性问题，进入专项治理 |
| 工单缺关键字段 | 自动退回并标记 `INVALID_TICKET` |

## 8. 复盘标准
- 复盘必须回答：
  1. 告警是否在SLA内响应？
  2. 五键是否完整可检索？
  3. 是否命中发布门禁规则？
  4. 是否需要调整阈值或路由人群？

## 9. 验收标准
1. 所有告警可路由到唯一主责团队。
2. P0/P1/P2 响应与关闭时限可量化统计。
3. 工单均具备五键字段并可追溯。
4. 重复告警有升级和专项治理闭环。
