# MiniApp Booking Review Service Recovery Runbook v1（2026-03-17）

## 1. 目标与当前边界
- 目标：把 booking review 当前已落地的人工恢复链路固定成可执行 runbook。
- 当前仅覆盖：
  - 用户评价提交后的人工恢复
  - 后台回复、跟进、看板查看
- 当前不覆盖：
  - 自动通知店长 / 技师组长 / 客服
  - 自动补偿
  - 自动奖励
  - 自动 SLA 升级任务
- 当前发布结论：`Can Develop / Cannot Release`。

## 2. 当前系统能力真值

### 2.1 已有系统能力
1. 用户可从订单列表 / 订单详情进入评价。
2. 差评会在数据层自动派生为：
   - `reviewLevel=差评`
   - `riskLevel=紧急`
   - `displayStatus=review_pending`
   - `followStatus=待跟进`
   - `auditStatus=manual_review`
3. 后台已有台账、详情、回复、跟进状态更新、看板页面。
4. 差评当前已可进入后台“店长待办”池，系统会写入联系人快照与 10 分钟认领 / 30 分钟首次处理 / 24 小时闭环截止时间。
5. 历史差评如果尚未初始化 `managerTodo*` 字段，当前只有在首次执行店长待办写动作时才会 lazy-init；list / dashboard 不会在读链路自动补齐。

### 2.2 未有系统能力
1. 没有自动消息推送到店长。
2. 没有自动把问题路由到技师负责人或客服负责人。
3. 没有工单系统联动。
4. 没有自动计时器去推送超时提醒。
5. 当前“店长待办”目标对象只冻结到门店主数据 `contactName/contactMobile`，当前没有稳定 `managerUserId` 真值。

## 3. 推荐角色分工

| 角色 | 当前职责 | 系统是否自动承接 |
|---|---|---|
| 运营值班 / 服务恢复 owner | 每日查看评价台账、筛选差评、分配后续动作 | 否，需人工进入后台台账 |
| 店长 | 处理门店环境、接待、履约体验问题 | 部分，系统只承接联系人快照与待办 SLA，不自动外发通知 |
| 技师负责人 | 处理技师服务质量问题 | 否，当前只能人工通知 |
| 客服恢复 owner | 负责回访、致歉、人工补偿决策 | 否，当前只支持后台回复 / 跟进状态记录 |
| 区域负责人 | 处理重大或反复差评 | 否，当前无自动升级链路 |

补充说明：
- 当前“店长”在系统里最多只能稳定回溯到门店联系人 `contactName/contactMobile`。
- 不能把这层联系人真值误写成“店长后台账号已打通”。

## 4. 人工恢复分池规则

| 命中条件 | 当前分池 | 建议动作 |
|---|---|---|
| `overallScore <= 2` | 紧急恢复池 | 立即人工查看详情，并同步门店或服务负责人 |
| `overallScore == 3` | 关注池 | 结合内容与标签决定是否人工跟进 |
| `overallScore >= 4` | 常规池 | 进入统计，不触发即时人工升级 |
| 命中负向标签 + 差评 | 紧急恢复池 | 人工确认是否需要店长 / 技师负责人介入 |

## 5. 当前 SLA 真值
1. 差评店长待办创建后，系统会写入：
   - 10 分钟认领截止
   - 30 分钟首次处理截止
   - 24 小时闭环截止
2. 这些截止时间当前只用于后台台账、详情、看板观察。
3. 当前代码库没有已提交的自动提醒、自动升级、自动外发消息证据。
4. 因此当前 SLA 仍属于“后台治理真值”，不是“自动恢复系统真值”。
5. 当历史差评缺少 booking order 行时，系统仍会按 `submitTime` 回填触发类型与 SLA 截止时间；但店长联系人快照可能继续为空。

## 6. 后台操作流程

### 6.1 台账巡检
1. 进入 `/mall/booking/review`。
2. 按 `riskLevel=2`、`followStatus=1`、`replyStatus=false` 优先筛选。
3. 如需店长协同，可叠加：
   - `onlyManagerTodo=true`
   - `managerTodoStatus`
   - `managerSlaStatus`
4. 对空列表：
   - 只算合法空态
   - 不能写成“服务恢复成功”

### 6.2 详情研判
1. 打开 `/mall/booking/review/detail?id=`。
2. 查看：
   - `bookingOrderId`
   - `storeId`
   - `technicianId`
   - `overallScore`
   - `reviewLevel`
   - `riskLevel`
   - `content`
   - `tags`
3. 当前可直接查看店长待办联系人、SLA 截止时间与最近处理备注。
4. 当前如果需要联系店长 / 技师负责人，仍由人工外呼或人工同步完成。

### 6.3 回复用户
1. 在详情页填写 `replyContent`。
2. 提交前人工确认话术。
3. 提交成功后：
   - `replyStatus=true`
   - `replyUserId`、`replyTime` 由系统记录
4. 提交失败时：
   - 不自动重试
   - 需人工再次提交

### 6.4 更新跟进状态
1. 在详情页选择 `followStatus`：
   - `0 无需跟进`
   - `1 待跟进`
   - `2 跟进中`
   - `3 已解决`
   - `4 已关闭`
2. 填写 `followResult` 记录处理结论。
3. 当前 `followOwnerId` 记录的是执行更新动作的操作人，不是自动路由 owner。

### 6.5 看板复盘
1. 进入 `/mall/booking/review/dashboard`。
2. 关注：
   - `negativeCount`
   - `pendingFollowCount`
   - `urgentCount`
   - `repliedCount`
3. 看板只有统计，不代表自动联动已经存在。

### 6.6 店长待办动作
1. 在详情页可执行：
   - 认领待办
   - 记录首次处理
   - 标记闭环
2. 服务端当前已守住状态机：
   - 认领只允许从 `待认领` 进入
   - 首次处理只允许从 `已认领 / 处理中` 进入
   - 闭环只允许从 `已认领 / 处理中` 进入
   - `已闭环` 视为终态，不能再回头认领
3. 这些动作只更新后台待办状态，不会自动改写 `followStatus`。
4. 写失败时当前只有人工重试，没有自动重试。

## 7. 对“差评是否第一时间通知店长”的当前口径
1. 从服务恢复设计原则看，差评应优先被人工关注。
2. 但当前代码真值只能支持：
   - 差评进入后台恢复台账
   - 人工查看详情
   - 人工回复与跟进
3. 当前没有已提交的自动店长通知能力。
4. 因此当前正确表述只能是：
   - “应由运营或服务恢复 owner 第一时间研判，并按问题类型人工同步店长或相关负责人”
   - 不能写成“系统已自动通知店长”
5. 当前正确表述只能是“系统已把差评放入后台店长待办池，并沉淀联系人快照与 SLA 截止时间”；不能写成外部消息系统已经打通。
6. 当前也不能写成“系统已找到门店店长账号并自动分派”；最多只能写“系统沉淀了门店联系人快照，供后台人工协同”。

## 8. 奖励与补偿口径
1. 当前没有自动好评奖励。
2. 当前没有自动差评补偿。
3. 如需补偿，只能作为人工恢复动作记录在 `followResult` 或外部流程中。
4. 任何把当前能力写成“自动发券 / 自动退款 / 自动积分”的表述都属于误升。

## 9. 审计与样本字段
- 用户侧关键键：`bookingOrderId`、`reviewId`
- 后台恢复关键键：`reviewId`、`bookingOrderId`、`storeId`、`technicianId`、`memberId`、`followOwnerId`、`replyUserId`、`managerTodoStatus`、`managerClaimDeadlineAt`、`managerFirstActionDeadlineAt`、`managerCloseDeadlineAt`
- 当前没有独立 `runId`、自动通知 `sourceBizNo`、自动告警 `errorCode` 审计样本。

## 10. Runbook No-Go
1. 不得把后台台账存在写成“已自动闭环服务恢复”。
2. 不得把看板 0 值写成“恢复成功”。
3. 不得把人工通知店长写成系统通知店长。
4. 不得把人工补偿决策写成自动补偿能力。
5. 不得把历史差评尚未初始化待办字段时的 `待初始化 / 未计入统计` 状态写成系统已经自动修复全量旧数据。
