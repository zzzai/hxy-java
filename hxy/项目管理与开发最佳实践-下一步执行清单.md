# 项目管理与开发最佳实践-下一步执行清单

更新时间：2026-02-16

目标：在不打断当前交付节奏的前提下，把“可持续研发能力”搭起来。

## 一、项目管理最佳实践（立即执行）

1. 建立单一优先级队列（P0/P1/P2），所有需求/缺陷统一进一个看板。
2. 固定节奏：每周一次计划会、每天 15 分钟站会、每周一次复盘会。
3. 强制 DoR（需求可开发定义）：
   - 有业务目标
   - 有验收标准
   - 有风险说明
   - 有回滚预案
4. 强制 DoD（完成定义）：
   - 代码合并前通过 CI 上线拦截规则
   - 核心链路冒烟通过
   - 文档更新
   - 监控和告警已覆盖
5. 每周只承诺可完成容量，禁止超卖计划。

## 二、项目开发最佳实践（立即执行）

1. 分支策略：`main`（稳定）+ `develop`（集成）+ `feature/*`（短分支）。
2. PR 规范：
   - 变更说明
   - 风险点
   - 回滚方式
   - 验证截图/日志
   - 模板文件：`.github/pull_request_template.md`
3. 数据库变更必须脚本化（可重放、可回滚、幂等优先）。
4. 所有关键流程必须有幂等键（支付、退款、核销、分账）。
5. 所有异步流程必须有补偿机制和重试上限。
6. 日志标准化：统一 traceId，错误日志可检索。
7. 发布标准化：预检 -> 初始化 -> 部署 -> 健康检查 -> 冒烟。
8. 生产配置外置（密钥不入库）。

## 三、今天可继续做的 10 项

1. 跑 `ci_gate.sh --full --init-db` 一次，形成“当天可发布状态”。
2. 为支付核心接口补充失败重试和告警阈值文档。
3. 把现有 SQL 补丁统一命名版本号（v1/v2/v3）。
4. 新增 PR 模板（风险、回滚、验证清单）。
5. 给关键 API 增加契约测试清单（支付/治理/预约）。
6. 加一份上线前检查清单（上线拦截规则、配置、回滚开关）。
7. 补全故障演练脚本执行记录模板。
8. 明确“谁能改生产配置”的权限边界。
9. 建立每周质量指标看板（缺陷率、回滚率、交付准时率）。
10. 将 K8s 基线清单接入 dev 环境演练（先不生产切换）。

## 四、未来两周执行顺序（建议）

1. 第 1 周：工程纪律
   - CI 上线拦截规则稳定运行
   - `.gitignore` 与仓库卫生清理
   - PR 规范落地
   - 上线/回滚清单固化
2. 第 2 周：环境演进
   - K8s 清单渲染上线拦截规则（不部署）
   - K8s dev 演练
   - 配置与密钥管理规范化
   - 监控与告警指标统一

## 五、与你当前项目最相关的三条红线

1. 不允许“先上线再补幂等/补偿”。
2. 不允许“人工分账替代系统分账”。
3. 不允许“无回滚方案上线”。
