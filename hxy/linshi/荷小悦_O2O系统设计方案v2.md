# **一、核心设计哲学**

+-----------------------------------------------------------------------+
| **从\"总部控制门店\"到\"规则约束行为\"**                              |
|                                                                       |
| 传                                                                    |
| 统方案：总部直接操作门店数据，1万家门店同步调用，耦合严重、系统脆弱。 |
|                                                                       |
| 优化                                                                  |
| 方案：总部只发布\"规则\"，门店只做\"实例化\"，两者通过规则引擎解耦。  |
|                                                                       |
| 核心转变：总部管                                                      |
| 规则，不管数据；门店提交变更时，规则引擎前置校验；事件驱动异步同步。  |
+-----------------------------------------------------------------------+

## **1.1 八大设计原则**

所有设计决策均遵循以下八项原则，贯穿数据结构、权限体系、技术架构全链路：

  ---------------- --------------------------------------------------------------------------- -------------------------------------------------------------
  **设计原则**     **核心内容**                                                                **技术落地关键点**

  **分层管控**     总部维护规则和模板，门店执行实例化配置，互不越权                            规则引擎前置校验；总部改规则，门店提交时自动验证

  **分离计算**     服务类聚焦技师时间排班，实物类聚焦库存扣减，套餐拆解为原子子项              策略模式（InventoryStrategy接口），运行时按类型选择

  **数据隔离**     各门店运营数据完全独立，通过字段+索引+权限+查询四层隔离                     store_id全量索引；行级权限校验；查询强制带store_id条件

  **规则外置**     权限规则独立于代码，存入eb_sku_control_rule配置表，DSL描述约束              新增门店类型/权限变更只需改数据，无需发版上线

  **快照不可变**   订单写入时冻结所有价格维度+SKU版本+套餐子项组合，历史数据不受后续变更影响   写入订单时同步写入snapshot\_\*字段及combo_relation_snapshot

  **事件驱动**     总部操作发布领域事件，门店服务异步消费，最终一致性替代同步调用              RocketMQ/Kafka；强制类事件高优先级队列；普通变更普通队列

  **兜底策略**     门店未配置价格/库存时自动回滚总部指导价，前端友好提示                       is_custom_price字段判断；价格为0展示\"咨询门店\"

  **兼容复用**     实物商品直接复用原有电商表，服务/套餐单独设计，减少开发冗余                 积分兑换独立eb_points_order表，与资金流财务完全隔离
  ---------------- --------------------------------------------------------------------------- -------------------------------------------------------------

## **1.2 核心概念重新定义**

统一业务语言，避免开发、运营、设计之间的概念混淆：

  ------------------------- ----------------------------------------------------------------------------- ------------------------ ------------------------------------------------------
  **概念**                  **定义**                                                                      **归属方**               **荷小悦场景示例**

  **SPU**                   标准化产品/服务单元，品牌统一基础品项，无具体售卖规格，含服务/实物/套餐三类   总部（唯一）             中式足疗、薰衣草精油、养生套餐A

  **SKU**                   库存/服务量单位，门店实际售卖的具体规格，关联价格/时长/库存/闲时优惠          总部定规格，门店定运营   中式足疗\[60分钟+重力\]、薰衣草精油\[10ml\]、闲时¥98

  **商品模板**              总部维护的标准化基础数据，含SPU/SKU所有基础属性、版本信息、生效时间           总部（唯一维护）         SPU名称、SKU规格、指导价、版本号

  **商品实例（Listing）**   门店基于模板生成的个性化运营数据，仅对当前门店生效                            各门店（独立维护）       门店售价、会员价、库存、闲时优惠价

  **合约策略**              挂载在门店Profile上的权限配置，决定门店在各维度的操作边界                     总部配置，门店只读       直营合约、标准加盟合约、高级加盟合约

  **闲时优惠**              门店针对非高峰时段设置的SKU优惠价，独立存储，与排班解耦                       门店（独立维护）         足疗60分钟 14:00-16:00 ¥98

  **快照**                  下单时冻结的价格+SKU版本+套餐子项快照，只写不改，历史数据不可变               系统自动生成             订单子项中的snapshot_store_price等字段

  **露珠（积分）**          虚拟农场场景下的用户资产，可兑换指定SKU，独立于资金流管理                     总部定规则，门店选配     100露珠兑换基础油加项，走eb_points_order独立表
  ------------------------- ----------------------------------------------------------------------------- ------------------------ ------------------------------------------------------

## **2.2 关键数据流向规则**

  ------------------ --------------------------- -------------------- ------------------------------------------------
  **触发动作**       **事件类型**                **队列优先级**       **门店侧消费逻辑**

  总部强制下架SKU    template.force_delist       高优先级（秒级）     所有门店Listing立即标记下架，门店无法覆盖

  总部修改基础信息   template.rule_updated       普通队列（分钟级）   违规Listing标记\"待复审\"，不强制回滚

  总部新增SPU/SKU    template.published          普通队列             为所有门店生成默认Listing（默认下架，指导价）

  全国大促开始       campaign.platform_started   高优先级             符合条件Listing自动入促，直营店无法豁免

  总部修改规则配置   rule.config_updated         普通队列             规则引擎热加载，下次校验立即生效

  门店提交变更       无事件（前置校验）          同步校验             规则引擎实时校验，通过则直接生效，失败返回原因
  ------------------ --------------------------- -------------------- ------------------------------------------------

## **2.3 规则引擎工作流**

门店每次提交变更时，后端执行以下校验链路：

+-----------------------------------------------------------------------+
| 门店提交变更请求（store_id + sku_id + field_name + new_value）        |
|                                                                       |
| ↓                                                                     |
|                                                                       |
| 步骤1：加载门店合约策略（contract_type from Store.profile）           |
|                                                                       |
| 步骤2：查询 eb_sku_control_rule（匹配 contract_type + product_type +  |
| field_name）                                                          |
|                                                                       |
| 步骤3：校验 store_can_edit                                            |
|                                                                       |
| → 为0：直接 REJECT（\"此字段不允许门店修改\"）                        |
|                                                                       |
| 步骤4：校验 has_range                                                 |
|                                                                       |
| → 为1：计算允许区间（range_type=percent: base_price×range_min \~      |
| base_price×range_max）                                                |
|                                                                       |
| → 为1：检查 new_value 是否在区间内                                    |
|                                                                       |
| 步骤5：超出区间处理                                                   |
|                                                                       |
| → violation_action=reject：REJECT，返回 description + 允许区间提示    |
|                                                                       |
| → violation_action=warn：通过，同时发送预警通知到总部                 |
|                                                                       |
| → need_approval=1：进入审批队列，暂不生效                             |
|                                                                       |
| 步骤6：校验通过 → 写入 eb_store_service_sku + 记录 eb_operation_log   |
+-----------------------------------------------------------------------+

### **表4：eb_sku_control_rule --- 规则引擎配置表（新增核心表）**

作用：外置权限规则，替代代码里的硬编码权限判断，支持动态配置，无需发版上线。

  ---------------------- --------------- ------------ ---------------------------------------------- -------------- ---------------------------------
  **字段名**             **类型**        **默认值**   **说明**                                       **可改方**     **技术适配点**

  **id**                 int unsigned    自增         规则主键                                       系统           ---

  **contract_type**      varchar(50)     all          适用合约类型：all/direct/standard/premium      总部           与门店Profile.contract_type匹配

  **product_type**       varchar(20)     all          适用商品类型：all/service/product/combo        总部           区分服务类/实物类规则

  **field_name**         varchar(64)     ---          被控字段名：price/shelf_status/stock等         总部           与门店提交字段名对应

  **store_can_edit**     tinyint(1)      1            门店是否可编辑此字段                           总部           为0则直接REJECT

  **has_range**          tinyint(1)      0            是否有值域限制                                 总部           为1时检查range\_\*字段

  **range_type**         varchar(20)     ---          percent（百分比浮动）/absolute（绝对值区间）   总部           计算允许的min/max值

  **range_min**          decimal(10,2)   ---          最小值或最小比例（如0.8=允许打八折）           总部           低于此值触发violation_action

  **range_max**          decimal(10,2)   ---          最大值或最大比例（如1.3=允许加价30%）          总部           高于此值触发violation_action

  **need_approval**      tinyint(1)      0            超出范围是否走审批流而非直接拒绝               总部           1=进审批队列，0=直接REJECT

## **6.2 关键技术落地决策**

  ---------------------- ------------------------------------ ------------------------------------------------------------------------------
  **技术问题**           **采用方案**                         **实施要点**

  **规则校验性能**       规则配置热加载 + 本地缓存            规则配置写入DB后，推送失效事件，各服务节点热加载缓存，避免每次校验查DB

  **万店同步**           消息队列异步消费 + 最终一致性        强制类事件用高优先级队列，普通变更用普通队列，消费失败自动重试，不阻塞主流程

  **库存超卖防护**       Redis原子操作 + DB乐观锁双重保障     用Redis incr/decr做预扣减，异步落盘DB，高并发下DB用version字段做乐观锁

  **快照写入一致性**     同一事务写入订单主表+所有子项+快照   使用数据库事务，下单时所有快照字段一次性写入，失败则全部回滚

  **多租户数据隔离**     行级隔离 + 应用层强制过滤            所有门店相关查询在SQL层强制加 store_id 条件，应用层二次校验，DB索引保障性能

  **操作日志写入**       异步写入，不影响主流程               主流程完成后异步发送日志事件，独立消费写入，失败不影响业务，可重放

  **闲时优惠实时计算**   JOIN查询 + Redis缓存活跃时段         高频查询的时段优惠数据用Redis缓存（TTL 5分钟），减少DB查询压力

  **积分流水一致性**     积分扣减 + 兑换订单同一事务          用分布式事务（Seata）或TCC模式，确保露珠扣减和兑换订单创建的原子性

# **八、方案总结与演进路径**

## **8.1 本方案 vs 传统方案核心差异**

  ------------------ ------------------------------------------------ -------------------------------------------------------------
  **对比维度**       **传统中心化方案**                               **本方案（规则引擎 + 事件驱动）**

  **权限控制**       代码里硬编码 if/else                             规则引擎外置（eb_sku_control_rule），纯配置变更，实时生效
                     权限判断，新增类型需改代码发版                   

  **总部同步门店**   同步调用1万个门店接口，高延迟、单点故障风险      消息队列异步消费，最终一致性，系统高韧性

  **价格快照**       退款/对账时查当前价格，与成交价不一致            下单时写入全量price快照，历史数据绝对不变

  **闲时优惠**       与排班表耦合，改排班可能影响优惠价格             独立 eb_store_flash_pricing 表，JOIN时合并，数据完全隔离

  **套餐核销**       核销时实时查关联表，套餐子项变更会影响历史订单   combo_relation_snapshot冻结下单时的子项组合，核销以快照为准

  **积分与资金**     共用订单表，财务对账混淆                         积分走 eb_points_order 独立表，资金流完全隔离

  **门店类型扩展**   新增门店类型需修改权限代码                       新增合约策略配置即可，不改代码

  **合规审计**       难以追溯历史操作和规则执行情况                   全链路操作日志 + 规则执行结果记录，完整可追溯
