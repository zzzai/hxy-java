# 荷小悦O2O多门店（服务+实物）SPU/SKU多门店完整结构说明（最新完整版）

本次最新完整版结构设计，围绕荷小悦「新中式·疗愈·社区」品牌定位，适配「服务项目（技师时间）+实物商品（库存）+套餐组合」混合O2O多门店模式，整合「业务管控、数据结构、技术落地、场景适配」四大维度，解决「总部统一品控、门店千店千价、库存/时间分离计算、多门店数据隔离、技术落地高效、UI场景适配」六大核心问题。

核心升级点：新增订单子项关联逻辑、闲时优惠定价适配、虚拟农场积分联动、总部版本控制优化，补充所有技术落地细节，衔接UI设计中的服务定制、技师排班等场景，实现「业务设计-数据结构-技术开发-前端展示」全闭环，所有设计可直接落地且兼容后续业务扩展。

# 一、核心设计原则（最新优化版，贯穿全结构）

在原有「分层管控、分离计算、数据隔离、自动兜底、兼容复用」五大原则基础上，新增「技术落地适配、场景化适配」两大原则，确保结构既符合业务逻辑，又能快速转化为代码，适配前端UI场景。

1. **分层管控**：总部仅维护标准化基础信息（SPU/SKU名称、规格、指导价、时长等），门店仅操作运营信息（价格、库存、上下架、排序、闲时优惠等），互不越权，总部保留超管兜底权限；

2. **分离计算**：服务类聚焦「技师时间排班」，实物类聚焦「实际库存扣减」，套餐类通过独立表拆解为子项，不污染核心SPU/SKU结构，结合策略模式实现库存扣减逻辑解耦；

3. **数据隔离**：各门店运营数据完全独立，通过「字段+索引+权限+查询」四层隔离，门店仅能查询/修改自身数据，无法操作其他门店及总部数据；

4. **自动兜底**：门店未配置价格/库存时，系统自动回滚至总部默认值，价格采用「写入时冗余」方案，避免N+1查询，提升性能，同时前端补充友好提示；

5. **兼容复用**：实物商品直接复用原有电商系统表结构，服务/套餐单独设计专用表，减少开发冗余；订单子项表衔接套餐核销场景，虚拟农场积分体系联动SKU兑换，兼容现有系统；

6. **技术落地适配**：所有数据表结构、关联规则均对接技术落地细则（策略模式、版本控制、原子化核销），新增必要字段（价格标识、版本、生效时间等），无需二次修改即可建表编码；

7. **场景化适配**：适配UI设计中的服务定制（时长/精油/力度）、技师排班（时间槽定价）、虚拟农场（积分兑换SKU）等场景，确保数据结构能支撑前端所有交互需求。

# 三、整体架构分层（总部+门店，最新优化版）

整体为「双层架构+单向数据流向+技术落地适配」，总部主数据是所有门店运营数据的“数据源”，门店基于总部主数据做个性化配置，数据仅从总部同步至门店，门店无法反向修改总部数据，同时新增订单子项表、操作日志表、积分兑换表，完善架构闭环，适配技术落地和UI场景。

## （一）架构分层图（最新完整版）

```plain text
【总部主数据层（全局唯一）】
├─ SPU主表（eb_service）：所有服务/套餐的标准化基础信息（含版本、生效时间）
├─ SKU规格表（eb_service_sku）：所有服务/实物的具体规格信息（含版本、生效时间）
├─ 套餐关联表（eb_service_combo_relation）：总部定义的套餐子项绑定规则（全局）
├─ 总部实物SPU/SKU表（原有电商表）：实物商品的总部基础信息
└─ 积分兑换规则表（eb_points_exchange）：总部定义的积分兑换基础规则（全局）
        ↓（单向同步：总部新增/修改→自动同步至所有门店，支持版本生效时间）
【门店运营数据层（各门店独立）】
├─ 门店SPU引用表（eb_store_service）：门店选择上架的SPU，记录上下架/排序
├─ 门店SKU定价表（eb_store_service_sku）：门店对SKU的定价/库存/会员价/闲时优惠（核心，含价格冗余字段）
├─ 原有电商实物门店表（eb_store_product*）：复用原有结构，存储实物商品运营数据
├─ 门店积分兑换表（eb_store_points_exchange）：门店选择的可兑换SKU及所需积分
└─ 技师排班表（eb_technician_schedule）：服务类SKU的技师时间槽，关联SKU和技师信息
【技术落地支撑层（全局/门店共用）】
├─ 订单主表（eb_order）：存储订单整体信息，关联门店、用户、套餐/单品SKU
├─ 订单子项明细表（eb_order_item）：套餐拆解子项、单品明细，关联核销状态（核心，原子化核销）
├─ 操作日志表（eb_operation_log）：记录门店/总部的所有修改操作，用于溯源
├─ 库存流水表（eb_stock_flow）：实物类SKU的库存变动记录，用于对账
└─ 用户积分表（eb_user_points）：用户虚拟农场露珠（积分）记录，关联兑换记录
```

## （二）数据流向规则（最新优化，含版本控制）

1. 总部新增SPU/SKU → 系统自动为所有门店生成默认的门店SPU/SKU记录（默认下架、价格为总部指导价、库存为默认值、is_custom_price=0，关联初始版本）；

2. 总部修改SPU/SKU基础信息（名称/图片/时长/指导价/规格）→ 生成新的版本（version+1），设置生效时间，所有门店的关联基础信息在生效时间后同步更新，历史订单/排班关联旧版本；

3. 总部停用SPU/SKU → 系统强制将所有门店的对应SPU/SKU设为下架，门店无法手动恢复，同时标记版本为“停用”，禁止新订单关联；

4. 门店仅能对自身的门店SPU/SKU记录做「上下架、改价、改库存、调排序、设置闲时优惠」操作，无法修改关联的总部ID、版本信息；

5. 门店设置闲时优惠 → 关联技师排班表的时间槽，标记闲时时段（如14:00-16:00），前端技师排班页自动展示优惠价格；

6. 门店配置积分兑换 → 从总部积分兑换规则中选择SKU，设置所需露珠数量，前端虚拟农场展示兑换提示；

7. 门店删除门店SPU/SKU记录 → 仅对当前门店生效，不影响其他门店及总部主数据，同时删除关联的闲时优惠、积分兑换记录；

8. 所有数据同步/修改操作 → 自动记录操作日志表，总部可追溯所有门店的操作，实物库存变动同时记录库存流水表。

# 四、核心数据表结构（最新完整版，含字段说明+核心索引+业务约束）

所有表均以`eb_`为前缀，服务/套餐用全新设计的专用表，实物商品复用原有电商表，新增订单子项表、积分兑换相关表，补充价格冗余、版本控制、闲时优惠等字段，所有字段均标注「是否可改」「技术适配点」，可直接用于MySQL建表，对接技术落地细则和前端UI场景。

## （一）总部主数据层（5张，全局唯一）
