# **一、核心设计原则（7大原则）**

本系统围绕\"总部管规则、门店管数据、规则引擎校验、事件总线同步\"四大核心机制，形成以下7大设计原则，贯穿全部数据结构与流程设计。

## **1. 分层管控原则**

总部仅维护标准化基础信息（SPU/SKU名称、规格、指导价、时长等基础主数据），门店仅操作运营信息（价格、库存、上下架、排序、闲时优惠等）。两者通过\"模板与实例\"机制解耦，互不越权，总部保留超管兜底权限（强制上下架、全局限价）。

## **2. 规则引擎外置原则（新增核心原则）**

所有权限边界、价格范围、操作约束，均通过 eb_sku_control_rule
规则配置表定义，而非硬编码在业务逻辑中。门店提交任何变更请求，后端先查规则表做合法性校验，校验不通过返回具体原因，不是笼统的\"无权限\"。规则调整只需改数据，不需改代码上线。

## **3. 事件驱动解耦原则（新增核心原则）**

总部的每一个业务操作（改价、上下架、新增商品、发起全国活动）都发布领域事件至消息队列，门店服务异步消费事件，实现最终一致性。彻底消除\"总部操作同步调用万家门店接口\"的耦合瓶颈，系统整体韧性大幅提升。强制类事件（如全局强制下架）走高优先级队列，秒级消费；通知类事件走低优先级队列，允许分钟级延迟。

## **4. 分离计算原则**

服务类商品聚焦\"技师时间排班\"（时间锁定与核销），实物类商品聚焦\"实际库存扣减\"，套餐类通过独立的订单子项表拆解为子项，不同类型的库存/时间计算策略通过策略模式（Strategy
Pattern）实现，互不污染，便于后续扩展新商品类型。

## **5. 数据隔离原则**

各门店运营数据完全独立，通过\"字段+索引+权限+查询\"四层隔离机制实现。所有门店运营表必须带
store_id 字段，接口层强制注入当前门店ID，SQL层强制带 store_id
条件，门店无法查询或修改其他门店数据。

## **6. 历史数据不可变原则（快照机制）**

订单创建时，必须完整冻结当时的价格快照（含5个价格维度）、SKU版本号、套餐子项快照。此后无论总部如何修改价格、规格、套餐组合，历史订单的展示与核销均以快照为准，与当前数据完全解耦。这是支撑退款、对账、客诉处理的关键基础。

## **7. 兼容复用与自动兜底原则**

实物商品直接复用原有电商系统表结构，服务/套餐单独设计专用表，减少开发冗余。门店未配置价格/库存时，系统自动回滚至总部默认值（指导价/默认库存），前端展示友好提示，确保业务连续性。

## **（三）商品可见性：双开关机制**

用户端商品是否展示，由两个独立开关共同决定，两个字段都为 true
才对用户可见：

> 商品可见 = 总部开关(eb_service.status = 1)
>
> AND 门店开关(eb_store_service.store_enabled = 1)
>
> 场景说明：
>
> 总部 status=0（停用）→ 全国所有门店同步下架，门店无法覆盖
>
> 总部 status=1（启用）→ 仅表示\"允许上架\"，门店需自行开启
>
> 门店 store_enabled=0 → 仅影响本店，其他门店不受影响
>
> 门店 store_enabled=1 → 需总部也为启用状态才真正可见

# **四、规则引擎设计（外置化权限管控）**

规则引擎是本次最核心的架构升级。传统方案把\"哪些字段能改、改的范围是多少\"硬编码在业务逻辑中，每次调整权限都要改代码上线。规则引擎外置后，总部只需在后台修改规则配置，系统自动执行新规则，完全无需开发介入。

## **（三）校验流程（伪代码）**

> // 门店提交变更请求
>
> function validateStoreChange(storeId, fieldName, newValue, skuId) {
>
> const store = getStoreProfile(storeId);
>
> const sku = getSkuTemplate(skuId);
>
> // 1. 查规则配置表
>
> const rule = eb_sku_control_rule.find({
>
> template_type: sku.type, // service/product/combo
>
> contract_type: store.contract_type, // direct/franchise/premium
>
> field_name: fieldName,
>
> status: 1,
>
> effective_time \<= now()
>
> });
>
> // 2. 基础可编辑校验
>
> if (!rule \|\| rule.store_can_edit === 0) {
>
> return { ok: false, reason: \"此字段不允许门店修改\" };
>
> }
>
> // 3. 范围校验
>
> if (rule.has_range) {
>
> const { min, max } = calcRange(rule, sku.guide_price);
>
> if (newValue \< min \|\| newValue \> max) {
>
> if (rule.need_approval) {
>
> return { ok: false, needApproval: true, reason: rule.error_message };
>
> }
>
> const msg = rule.error_message
>
> .replace(\"{min}\", min).replace(\"{max}\", max);
>
> return { ok: false, reason: msg };
>
> }
>
> }
>
> // 4. 校验通过，直接写入
>
> return { ok: true };
>
> }

# **五、事件驱动设计（总部与门店解耦）**

传统方案的最大耦合点：总部操作需同步调用万家门店的接口，一旦某家门店服务异常，整个操作失败。事件驱动方案：总部只发布一个领域事件，消息队列保证投递，门店服务独立消费，彻底解耦。

## **（一）核心领域事件清单**

  --------------------------- ------------------------- --------------- ---------------- --------------
  **事件名称**                **触发操作**              **消费方**      **队列优先级**   **允许延迟**

  template.published          总部新增SPU/SKU           所有门店服务    普通             分钟级

  template.updated            总部修改SPU/SKU基础信息   所有门店服务    普通             分钟级

  template.force_delist       总部强制下架              所有门店服务    高优先           秒级

  template.deprecated         总部停用SKU               所有门店服务    高优先           秒级

  rule.updated                总部更新规则配置          规则引擎服务    普通             分钟级

  campaign.platform_started   总部发起全国大促          符合条件门店    普通             分钟级

  campaign.platform_ended     总部结束全国大促          参与门店服务    普通             分钟级

  stock.low_warning           门店库存低于阈值          门店/总部运营   低               小时级

  order.created               用户下单成功              库存服务/排班   高优先           秒级

  order.completed             服务/商品核销完成         财务/积分服务   普通             分钟级

  approval.approved           总部审批通过              门店服务        普通             分钟级

## **（三）幂等性与失败重试**

+-----------------------------------------------------------------------+
| **关键设计要点**                                                      |
+-----------------------------------------------------------------------+
| 每个事件携带唯一 event_id，消费前检查 eb_event_log                    |
| 表，防止重复消费（幂等）                                              |
|                                                                       |
| 消费失败自动重试3次，间隔指数退避（1s → 4s →                          |
| 16s），超过3次进入死信队列                                            |
|                                                                       |
| 死信队列由运维人工处理，消费失败不影响其他事件的正常流转              |
|                                                                       |
| 强制类事件                                                            |
| （force_delist/order.created）在消费成功前阻塞后续处理，确保状态一致  |
|                                                                       |
| 普通事件允许最终一致性，门店状态允许秒级延迟，不影响业务正确性        |

| 门                                                                    |

## **（三）收益分账模型**

> 资金流向：用户付款 → 平台统一收款 → 按结算周期划拨给门店
>
> （推荐T+1或T+7结算，不允许门店直接收款后总部事后抽成）
>
> 分账结构：
>
> 用户实付金额
>
> ├── 平台服务费（3\~5%，按合约）
>
> ├── 技术使用费（固定/按单）
>
> ├── 大促活动补贴（总部承担）
>
> └── 门店实收金额（剩余部分）
>
> 积分兑换单独结算（eb_points_order），不计入普通订单GMV
>
> 门店可在App中实时查看\"待结算金额\"和\"历史结算明细\"

# **八、关键场景完整流程**

## **场景1：用户预约足疗（服务类下单完整链路）**

> 1\. 用户选择门店 → 前端查询 eb_store_service（store_enabled=1 AND
> eb_service.status=1）
>
> 2\. 选择\"足疗60分钟\" → 展示SKU规格（eb_store_service_sku）
>
> \- 展示 store_price（门店价）+
> guide_price_cache（划线价，冗余字段，无需JOIN）
>
> \- 检查当前时间是否在闲时段（查 eb_store_flash_pricing）
>
> \- 如在闲时段：展示 flash_price（绿色标签）
>
> 3\. 选择技师和时间 → 查询 eb_technician_schedule（status=1
> 且时长覆盖）
>
> \- 找出空闲时间槽（status=1）
>
> \- 结合 eb_store_flash_pricing 判断该时段是否有闲时价
>
> 4\. 确认下单 → 事务内原子操作：
>
> a\. 锁定排班时间槽（UPDATE status=2, locked_order_id=xxx）
>
> b\. 创建 eb_order（记录 appoint_time、store_id）
>
> c\. 创建 eb_order_item（写入5个价格快照 + price_type + sku版本快照）
>
> d\. 发布 order.created 事件
>
> 全部成功则提交事务，任一失败则回滚（排班时间槽自动释放）
>
> 5\. 到店服务 → 门店端扫码/点击核销 → UPDATE
> eb_order_item.verify_status=1
>
> \- 记录 verify_time, verify_staff_id
>
> \- 发布 order.completed 事件 → 触发财务结算 + 用户积分发放

## **场景2：套餐下单与原子化核销**

> // 养生套餐A = 足疗60分钟 + 艾草贴×2 + 养生茶
>
> 1\. 下单时，查 eb_service_combo_relation 获取当时的子项组合
>
> 2\. 序列化子项快照至 eb_order_item.combo_relation_snapshot：
>
> \[
>
> {item_type:1, item_id:101, item_num:1, sku_name:\"足疗60分钟\",
>
> snapshot_store_price:190.00, snapshot_guide_price:200.00},
>
> {item_type:2, item_id:205, item_num:2, sku_name:\"艾草贴\",
>
> snapshot_store_price:0.00, snapshot_guide_price:0.00},
>
> {item_type:2, item_id:301, item_num:1, sku_name:\"养生茶\",
>
> snapshot_store_price:0.00, snapshot_guide_price:0.00}
>
> \]
>
> 3\. 核销时以快照为准，不再查 eb_service_combo_relation
>
> \- 子项独立核销：服务类走技师排班核销，实物类走库存扣减
>
> \- 支持分次核销（先做足疗，下次再领艾草贴）
>
> \- 每个子项独立记录 verify_status，全部完成则订单完成
>
> 4\. 即使总部后来修改套餐子项，历史订单核销不受影响（快照保护）

## **场景3：总部全国大促发起流程**

> 1\. 总部在管理后台创建大促活动，设置：
>
> \- 活动时间、折扣力度、适用SKU范围、合约类型范围
>
> \- mandatory=true（强制参与）or false（可豁免）
>
> 2\. 系统发布 campaign.platform_started 事件
>
> 3\. 各门店服务消费事件：
>
> \- 合约类型在 campaign.scope 内 → 自动将符合条件的SKU标记入促
>
> \- 更新 eb_store_service_sku 的活动价格（临时字段）
>
> \- 对 mandatory=true 的活动：门店标记 force_join=1，无法退出
>
> 4\. 直营店：全部强制参与
>
> 加盟店（standard）：强制参与，可申请豁免（需总部审批）
>
> 加盟店（premium）：可选择参与，申请退出无需审批
>
> 5\. 活动结束 → 发布 campaign.platform_ended 事件 → 各门店恢复正常价格

## **场景4：规则变更的平滑过渡**

> 总部决定将加盟店价格浮动范围从±20%收窄到±15%：
>
> 1\. 总部更新 eb_sku_control_rule：range_min=0.85, range_max=1.15
>
> 设置 effective_time = 7天后（给门店调整窗口）
>
> 2\. 系统发布 rule.updated 事件（effective_time后）
>
> 3\. 各门店消费事件：扫描本店所有SKU，找出价格超出新范围的实例
>
> 标记 eb_store_service_sku.review_status=1（待复审）
>
> 发送通知给门店店长：哪些商品需要在7天内调整价格
>
> 4\. 门店调整价格 → 规则引擎用新规则校验 → 通过后清除待复审标记
>
> 5\. 7天后仍未处理 → 系统自动将超范围价格回滚至范围边界值
>
> 记录操作日志（operator_type=3 系统自动）
>
> 再次通知门店

## **场景5：虚拟农场积分兑换**

> 1\. 用户在虚拟农场页查看可兑换项目：
>
> 查 eb_store_points_exchange（当前门店）
>
> JOIN eb_points_exchange（总部规则）获取基础信息
>
> 2\. 用户选择\"100露珠兑换基础油加项\"，确认兑换：
>
> a\. 检查用户露珠余额（eb_user_points）
>
> b\. 检查门店兑换库存（eb_store_points_exchange.store_stock）
>
> c\. 检查每日兑换限制（eb_points_exchange.daily_limit）
>
> 3\. 原子操作：
>
> a\. 扣减用户露珠（INSERT eb_user_points流水 + UPDATE余额）
>
> b\. 扣减门店兑换库存
>
> c\. 创建 eb_points_order（兑换单，与 eb_order 完全隔离）
>
> d\. 发送兑换码/核销凭证给用户
>
> 4\. 到店使用兑换品 → 技师在门店端核销兑换单
>
> UPDATE eb_points_order.exchange_status=1
>
> 财务上独立对账，不计入普通订单GMV
