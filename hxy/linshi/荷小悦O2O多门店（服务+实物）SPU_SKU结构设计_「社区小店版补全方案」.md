# 荷小悦O2O多门店（服务+实物）SPU/SKU结构设计 「社区小店版补全方案」

基于荷小悦**社区小店模型**（弱化房间/床位管理、排队等位为常态、运营轻量化），针对此前提出的5大深层次盲区，**剔除复杂的房间资源管理**，聚焦补全**财务价格分摊/逆向退款、履约BOM物料消耗、营销价格优先级、技师技能矩阵**四大核心盲区，所有设计均做**轻量化适配**，贴合小店低复杂度运营需求，同时保持原有架构的扎实性，补全后形成**商业闭环+财务合规+履约落地+营销适配**的完整结构，仍适配MVP阶段快速开发，且能支撑社区小店的日常运营。

本次补全遵循**「最小改动、最大适配」**原则：不重构原有核心表结构，仅通过**新增轻量化子表、原有表补充关键字段**实现盲区补全，与原有SPU/SKU架构完全兼容，开发成本低，贴合社区小店的运营和开发实际。

## 核心适配说明（荷小悦社区小店专属）

1. **资源维度**：完全弱化房间/床位管理，无需新增`eb_store_room`/`eb_room_schedule`表，预约仅锁定**技师时间槽**，到店排队等位为常态，系统仅展示「技师忙/闲」状态，不做房间资源强校验，贴合小店轻量化运营；

2. **财务维度**：极简设计**价格分摊/逆向退款**逻辑，仅在订单子项表新增1个核心字段，无复杂退款规则表，小店财务可快速对账；

3. **履约维度**：新增**轻量化服务物料BOM表**，仅关联服务SPU与消耗实物SKU，无复杂版本/批次管理，适配小店服务端的物料自动扣减；

4. **营销维度**：设计**全局+门店级价格计算流水线**，新增极简定价策略配置表，门店可继承总部规则，无需为每个SKU单独配置，降低运营成本；

5. **技师维度**：新增**轻量化技师技能矩阵表**，仅标记技师可承接的服务SPU，无复杂的技能分级/认证，贴合小店技师团队规模小、技能边界清晰的特点。

所有补全内容均融入原有SPU/SKU核心架构，形成**「总部管控+门店自主+小店轻量化+商业/财务/履约闭环」**的完整结构，仍达到P7/P8级架构师设计水准，且完全适配荷小悦社区小店的业务现实。

## 一、财务维度补全：价格分摊与逆向退款（核心刚需）

### 核心痛点

套餐打折后，子项部分核销的退款金额无明确依据，导致小店财务对账混乱、总部与门店扯皮，是**必须补全的核心点**。

### 补全原则

**轻量化设计**：不新增独立退款规则表，仅在**原有订单子项表** **`eb_order_item`** **新增核心字段，下单时自动按权重分摊实付金额，退款时直接按未核销子项的分摊金额**退款，小店财务可直接按字段对账，无复杂计算。

### 1. 原有表字段新增（`eb_order_item`）

在原有订单子项明细表中新增3个关键字段，实现价格分摊与退款金额锁定，**无侵入式修改，不影响原有逻辑**。

|字段名|类型|默认值|注释|业务约束|可改方|技术适配点|
|---|---|---|---|---|---|---|
|`real_split_price`|decimal(10,2)|0.00|子项实付分摊金额（核心）|非负，下单时自动计算，不可改|无（系统计算）|退款金额的核心依据，财务对账主键|
|`split_weight`|decimal(10,4)|0.0000|子项价格权重（用于计算分摊金额）|非负，下单时自动计算|无（系统计算）|分摊金额的计算依据|
|`refunded_amount`|decimal(10,2)|0.00|该子项已退款金额|非负，≤real_split_price|系统（退款时）|记录子项退款明细，避免重复退款|
### 2. 核心计算逻辑（贴合小店，极简权重算法）

#### （1）套餐实付金额分摊算法（下单时自动执行）

小店无需复杂的成本分摊，直接按**「子项原价/套餐所有子项原价总和」**计算权重，适配套餐打折场景，计算简单、财务易核对：

```Plain Text

步骤1：计算套餐所有子项的原价总和 = ∑(子项SKU的门店售价/指导价)
步骤2：单个子项的权重 split_weight = 该子项原价 / 套餐子项原价总和
步骤3：单个子项的实付分摊金额 real_split_price = 套餐实际支付金额 × 该子项权重
步骤4：尾差处理（如角分）：将尾差计入最后一个子项的real_split_price，确保所有子项分摊金额总和=套餐实付金额
```

**示例**：养生套餐A实付100元（足疗原价80元+养生茶原价40元，原价总和120元）

- 足疗权重 = 80/120 ≈ 0.6667，分摊金额 = 100×0.6667 = 66.67元

- 养生茶权重 = 40/120 ≈ 0.3333，分摊金额 = 100×0.3333 = 33.33元

- 总和=100元，与套餐实付一致，财务直接按字段对账。

#### （2）逆向退款逻辑（申请退款时自动执行）

**核心规则**：**仅退未核销子项的** **`real_split_price`** **，已核销子项不退款**，小店财务无需复杂计算，直接按订单子项的核销状态+分摊金额对账：

1. 退款时，系统自动筛选该订单下**未核销（`verify_status=0`）**的子项；

2. 退款总金额 = 所有未核销子项的`real_split_price`总和 - 已退款金额（`refunded_amount`）；

3. 退款完成后，更新未核销子项的`refunded_amount`为实际退款金额，同时标记子项状态为**已退款（** **`verify_status=3`** **）**；

4. 若套餐子项**部分核销**（如茶已核销、足疗未核销），则仅退未核销子项的分摊金额（如66.67元），无争议。

### 3. 业务约束（小店财务合规）

1. `real_split_price`和`split_weight`为**下单时系统一次性计算**，后续无论门店改价、总部调指导价，均不修改，确保退款依据唯一、固定；

2. 退款总金额**不得超过**订单实付金额，且**不得超过**未核销子项的分摊金额总和；

3. 已核销子项（`verify_status=1`）的`real_split_price`不可退款，避免客诉和财务混乱；

4. 订单全额退款后，所有子项的`refunded_amount`=各自的`real_split_price`，并标记订单状态为**已全额退款**。

## 二、履约维度补全：服务物料BOM消耗（足疗行业刚需）

### 核心痛点

服务项目（如精油开背）伴随**内部实物物料消耗**（如10ml精油），当前设计无自动扣减逻辑，导致小店实物库存账实不符（系统显示有库存，实际已用完），是足疗行业的**核心履约盲区**。

### 补全原则

**轻量化BOM设计**：新增独立的**服务物料清单表** **`eb_service_bom`**，仅关联服务SPU与消耗的实物SKU，无复杂的版本/批次/损耗管理，服务完成时**自动扣减实物库存**，记入小店内部领料成本，不影响前端销售，贴合小店运营和库存管理能力。

### 1. 新增核心表：`eb_service_bom`（服务物料清单表，总部维护）

**作用**：定义服务SPU对应的**内部消耗实物SKU**及消耗数量，总部统一配置，所有门店同步，小店无需修改，适配社区小店的标准化运营。

|字段名|类型|默认值|注释|业务约束|可改方|技术适配点|
|---|---|---|---|---|---|---|
|id|int(11) unsigned|自增|唯一BOM ID（主键）|非空，唯一|无（主键）|关联服务SPU和实物SKU|
|service_id|int(11)|0|关联总部服务SPU ID（`eb_service.id`）|非空，仅关联type=1的服务SPU|总部|仅服务项目有物料消耗，套餐/实物无|
|product_sku_id|int(11)|0|关联实物SKU ID（原有电商实物SKU/`eb_service_sku`）|非空，仅关联实物类SKU|总部|消耗的实物SKU，支持复用原有电商表|
|quantity|decimal(6,2)|1.00|单位服务消耗的实物数量（如1.00瓶、10.00ml）|＞0，精确到小数点后2位|总部|库存自动扣减的数量依据|
|is_mandatory|tinyint(1)|1|是否为必耗物料：0=可选 1=必耗|必耗物料必须扣减，可选按需扣|总部|区分核心消耗和可选消耗|
|remark|varchar(128)|空|物料备注（如“精油开背专用薰衣草精油”）|可选，小店运营参考|总部|辅助小店技师领料/库存管理|
#### 核心索引

```SQL

PRIMARY KEY (id);
UNIQUE KEY `idx_service_product` (service_id, product_sku_id); -- 避免同一服务重复关联同一物料
KEY `idx_service_id` (service_id); -- 按服务SPU快速查询物料清单
```

#### 业务约束

1. 仅关联**服务类SPU（type=1）**，套餐/实物SPU无物料消耗，套餐的物料消耗由其下的服务子项各自的BOM表决定；

2. 同一服务SPU+同一实物SKU仅能存在一条记录，避免重复扣减库存；

3. 必耗物料（`is_mandatory=1`）在服务完成时**强制自动扣减**，可选物料（`is_mandatory=0`）由技师在服务完成时手动选择是否扣减（如用户加项的物料）；

4. 总部修改BOM表后，**新订单按新版本执行，历史未完成服务按旧版本执行**，避免影响已预约业务。

### 2. 核心履约逻辑（服务完成自动扣减，轻量化）

贴合小店运营，无需人工介入，**服务核销/完成时自动触发库存扣减**，记入内部领料成本，与前端销售出库分离，库存流水单独标记，财务可区分“销售出库”和“服务消耗出库”。

```Plain Text

步骤1：技师在门店端标记「服务完成」（即用户核销服务，`verify_status=1`）；
步骤2：系统自动查询该服务SPU对应的`eb_service_bom`表，获取必耗物料清单；
步骤3：调用**实物库存扣减策略**（原有`RealProductStockStrategy`），按`quantity`扣减对应门店的实物SKU库存；
步骤4：在**库存流水表`eb_stock_flow`**中记录，**出库类型标记为「服务消耗」**，备注关联服务订单号/技师ID；
步骤5：可选物料由技师手动勾选后，系统执行上述扣减逻辑，未勾选则不扣减。
```

### 3. 库存流水表适配（`eb_stock_flow`）

在原有库存流水表中**新增1个字段**，区分出库类型，小店财务可快速对账“销售出库”和“服务消耗出库”：

|字段名|类型|默认值|注释|
|---|---|---|---|
|`flow_type`|tinyint(1)|1|流水类型：1=销售出库 2=服务消耗出库 3=采购入库 4=退货入库|
## 三、营销维度补全：价格优先级与多重营销冲突处理

### 核心痛点

闲时优惠、会员价、优惠券叠加时，价格计算无明确规则，导致用户实付价格争议、小店运营混乱，需明确定义**价格计算流水线**和**定价互斥策略**，且设计轻量化，小店运营可快速配置。

### 补全原则

**「全局规则+门店微调」**：总部定义**全局价格计算优先级**和默认互斥策略，门店可继承或微调（仅针对闲时优惠是否与会员价叠加），无需为每个SKU单独配置，降低小店运营成本；通过**新增全局定价策略表`eb_pricing_strategy`**实现，原有表仅新增1个关键字段。

### 1. 新增全局表：`eb_pricing_strategy`（定价策略表，总部/门店分级维护）

**作用**：定义价格计算流水线、营销互斥规则，总部配置全局默认规则，门店可创建自己的规则（继承总部，仅微调），适配小店的个性化营销需求。

|字段名|类型|默认值|注释|业务约束|可改方|
|---|---|---|---|---|---|
|id|int(11) unsigned|自增|唯一策略ID（主键）|非空，唯一|无（主键）|
|level|tinyint(1)|1|策略级别：1=总部全局 2=门店自定义|总部规则所有门店可继承|总部/门店|
|store_id|int(11)|0|门店ID，0=总部全局规则|级别2时非空|总部/门店|
|price_calc_pipeline|varchar(128)|"base->time->member->coupon"|价格计算流水线，用->分隔|按固定关键字配置|总部/门店|
|is_time_member_overlay|tinyint(1)|0|闲时价是否与会员价叠加：0=不叠加（默认）1=叠加|小店可按需开启|总部/门店|
|coupon_deduct_base|tinyint(1)|2|优惠券抵扣基数：1=原价 2=闲时价/会员价（最终价）|贴合用户认知，默认按最终价抵扣|总部/门店|
|status|tinyint(1)|1|策略状态：0=停用 1=启用|启用后生效|总部/门店|
#### 核心索引

```SQL

PRIMARY KEY (id);
UNIQUE KEY `idx_level_store` (level, store_id); -- 一个门店仅能有一个自定义规则
KEY `idx_status` (status); -- 按状态查询生效规则
```

#### 核心配置说明（贴合小店，默认极简配置）

总部全局默认配置为**足疗行业通用规则**，小店无需修改即可使用，仅在有特殊营销需求时微调：

1. **价格计算流水线**：`base->time->member->coupon`

    - `base`：门店SKU基础价（`eb_store_service_sku.price`）

    - `time`：闲时优惠价（如有，覆盖基础价，不叠加）

    - `member`：会员价（基于上一步价格计算折扣，闲时价默认不叠加）

    - `coupon`：优惠券（基于上一步最终价抵扣，默认不按原价抵扣）

2. **互斥策略**：`is_time_member_overlay=0`（闲时价为特殊一口价，不与会员价叠加，避免小店利润损失）

3. **优惠券抵扣**：`coupon_deduct_base=2`（按闲时价/会员价抵扣，而非原价，贴合用户认知，减少客诉）

**示例**：SKU基础价100元，闲时价80元，V6会员7折，满100减20优惠券

- 按默认规则计算：闲时价80元（不叠加会员价）→ 优惠券满100减20不满足 → 实付80元；

- 若小店开启叠加：闲时价80元×7折=56元 → 优惠券不满足 → 实付56元；

- 若无闲时价：基础价100元×7折=70元 → 优惠券满100减20不满足 → 实付70元。

### 2. 原有表字段新增（`eb_store_service_sku`）

在门店SKU定价表中新增1个字段，标记该SKU**是否参与闲时优惠**，与定价策略表联动，实现闲时价的精准配置：

|字段名|类型|默认值|注释|业务约束|可改方|
|---|---|---|---|---|---|
|`is_time_discount`|tinyint(1)|0|是否参与闲时优惠：0=不参与 1=参与|参与则需配置闲时价和时段|门店|
|`time_discount_price`|decimal(10,2)|0.00|闲时优惠价|≥0，is_time_discount=1时非空|门店|
|`time_discount_period`|varchar(255)|空|闲时优惠时段（如14:00-16:00,22:00-24:00）|用逗号分隔，24小时制|门店|
### 3. 核心价格计算逻辑（系统自动执行，小店无感知）

前端展示/下单时，系统自动根据**门店生效的定价策略**计算实付价格，全程无需人工介入，且价格计算结果**前端实时展示**，避免用户付款时争议：

1. 下单/展示时，系统自动查询该门店的定价策略（无自定义则用总部全局规则）；

2. 按`price_calc_pipeline`依次计算价格，根据`is_time_member_overlay`判断是否叠加；

3. 按`coupon_deduct_base`判断优惠券抵扣基数，计算最终实付价格；

4. 所有价格计算过程**前端透明展示**（如「闲时价80元（原价100元）→ 不满足优惠券使用条件」），小店无需解释。

## 四、技师维度补全：技师技能矩阵（门店运营刚需）

### 核心痛点

技师技能有边界（如初级技师仅能做基础足疗，高级技师可做正骨），当前设计无技能限制，可能导致**错派订单**（如正骨订单派给初级技师），引发客诉，贴合小店技师团队规模小、技能边界清晰的特点。

### 补全原则

**轻量化技能绑定**：新增独立的**技师技能关联表** **`eb_technician_skill`**，仅关联技师ID与服务SPU ID，无复杂的技能分级/认证/评分，总部统一维护服务技能分类，门店仅需标记技师可承接的服务SPU，操作简单。

### 1. 新增核心表：`eb_technician_skill`（技师技能关联表，门店维护）

**作用**：定义技师可承接的服务SPU，门店根据技师实际技能配置，预约时系统自动过滤**仅展示可承接该服务的技师**，避免错派订单，贴合小店运营。
