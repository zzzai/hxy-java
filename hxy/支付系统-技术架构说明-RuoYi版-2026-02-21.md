# 支付系统技术架构说明（RuoYi版）

- 日期：2026-02-21
- 基座：`ruoyi-vue-pro-master`
- 核心模块：`yudao-module-pay`、`yudao-module-mall/yudao-module-trade`

## 1. 架构目标

1. 建立“交易模块 -> 支付中台 -> 渠道”的单向架构，避免业务直连渠道。
2. 保证支付与退款在异常场景下可收敛、可补偿、可审计。
3. 为后续微服务拆分和 K8s 部署保留清晰边界。

## 2. 模块边界

1. 交易域（Trade）：
   - 创建交易订单、履约、售后、退款申请。
   - 通过 `PayOrderApi` / `PayRefundApi` 与支付域交互。
2. 支付域（Pay）：
   - 管理支付应用、支付渠道、支付订单、退款订单、通知任务。
   - 统一处理渠道 SDK、回调验签、状态机、补偿任务。
3. 渠道适配层（Client）：
   - `PayClient` 统一接口。
   - 微信/支付宝/钱包/Mock 由工厂按 `channelCode` 装配。

## 3. 关键代码映射

1. 支付订单服务：`yudao-module-pay/.../service/order/PayOrderServiceImpl.java`
2. 退款服务：`yudao-module-pay/.../service/refund/PayRefundServiceImpl.java`
3. 回调入口：`yudao-module-pay/.../controller/admin/notify/PayNotifyController.java`
4. 通知任务：`yudao-module-pay/.../service/notify/PayNotifyServiceImpl.java`
5. 渠道工厂：`yudao-module-pay/.../framework/pay/core/client/impl/PayClientFactoryImpl.java`
6. 交易调用支付：`yudao-module-mall/.../service/order/TradeOrderUpdateServiceImpl.java`
7. 售后调用退款：`yudao-module-mall/.../service/aftersale/AfterSaleServiceImpl.java`

## 4. 核心技术机制

### 4.1 幂等与状态机

1. 下单幂等：
   - 基于 `appId + merchantOrderId` 防重复创建。
2. 回调幂等：
   - 通过状态前置校验与条件更新，防止重复入账。
3. 状态机单向流转：
   - 支付：`WAITING -> SUCCESS/REFUND/CLOSED`
   - 退款：`WAITING -> SUCCESS/FAILURE`

### 4.2 回调处理策略

1. 统一回调入口：`/pay/notify/order|refund|transfer/{channelId}`。
2. 处理原则：先应答渠道，再异步执行业务处理。
3. 业务通知采用任务重试表（`pay_notify_task` + `pay_notify_log`）。

### 4.3 补偿与定时任务

1. 支付状态同步任务：补偿回调丢失。
2. 支付过期任务：查单、关单、本地关闭。
3. 退款同步任务：补偿“受理成功但终态未达”。
4. 通知任务重试：按频次退避直到成功或失败终态。

## 5. 数据模型（核心表）

1. `pay_app`：支付应用配置。
2. `pay_channel`：渠道配置与密钥参数。
3. `pay_order`：支付主订单。
4. `pay_order_extension`：每次渠道下单尝试记录。
5. `pay_refund`：退款订单。
6. `pay_notify_task` / `pay_notify_log`：业务通知任务与日志。
7. `pay_transfer`：转账能力（后续可扩）。

## 6. 服务商模式架构约束（当前重点）

1. 微信服务商模式参数必须成对：
   - `sp_mchid + sub_mchid`
2. 在 `sub_mchid` 未到位前：
   - 完成所有不依赖真实交易的能力（规则、补偿、门禁、监控）。
3. `sub_mchid` 到位后：
   - 执行真实联调（下单、回调、退款、对账）并沉淀证据。

## 7. 安全与合规

1. 渠道密钥、证书路径、敏感配置不得写入仓库明文。
2. 回调验签失败必须拒绝并告警。
3. 退款执行必须保留审计字段（操作者、规则命中、结果）。
4. 权限原则：门店无资金操作权限，总部统一支付治理。

## 8. 可观测性与上线拦截规则

1. 核心指标：
   - 支付成功率、回调延迟、退款收敛时长、对账差异量。
2. 告警规则：
   - 回调失败突增、补偿任务积压、对账差异超阈值。
3. 上线拦截规则（Gate）：
   - 无回滚方案、无验收证据、任务未跑、监控过期即阻断。

## 9. 演进路线

1. 当前：单体模块化（按领域解耦）。
2. 下一步：支付中台独立服务化（先读写分离，再服务拆分）。
3. K8s 方向：先做“仅渲染校验”CI，再推进灰度发布与回滚编排。
