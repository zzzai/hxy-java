# 用户数据-高争议标签治理规则 V1

> 状态：执行版  
> 日期：2026-02-16  
> 策略：先保留定义，再加上线拦截规则；先人工复核，再逐步自动化

## 1. 目标

1. 满足“高争议标签先保留”的业务诉求。
2. 以合法、公平、可解释、可申诉为前提使用标签。
3. 防止高争议标签直接驱动拒单、歧视定价、差别服务。

## 2. 标签风险分级

### G0 低风险标签（可自动化）

示例：
1. 到店频次、预约时段偏好、服务时长偏好、复购间隔。
2. 满意度分层、优惠券领取响应、活动参与度。

策略：
1. 可用于推荐、提醒、内容排序。
2. 可自动触达，但需频控和退订入口。

### G1 高争议标签（保留定义，严格上线拦截规则）

示例：
1. `income_level_inferred`（收入层级推断）。
2. `price_sensitivity`（价格敏感度）。
3. `complaint_risk_level`（投诉风险等级）。
4. `churn_risk_level`（流失风险等级）。
5. `neg_feedback_trend`（负面反馈趋势）。

策略：
1. 默认关闭，须走策略审批后开通。
2. 只能用于“服务改善、关怀提醒、运营实验”。
3. 不得用于拒绝服务、抬价、降级权益、差别排队。
4. 上线前必须通过离线偏差评估和法务/合规评审。
5. 必须记录“标签来源、更新时间、置信度、解释信息”。
6. 触达前增加人工复核阈值（如高价值客群、异常判断客群）。

### G2 禁止标签（仅保留概念，不得实现）

示例：
1. `marital_status_inferred`（婚姻状态推断）。
2. `rights_awareness_level`（维权意识等级）。
3. `negotiation_difficulty`（协商难度）。
4. `social_profile_external`（外部社交画像抓取）。

策略：
1. 禁止采集、禁止计算、禁止展示、禁止触达使用。
2. 若未来要启用，需重新立项并完成DPIA后再评审。

## 3. 使用白名单与黑名单

## 3.1 白名单（允许）

1. 服务建议排序（例如推荐更匹配的时段和技师）。
2. 关怀类触达（复访提醒、满意度回访、健康建议）。
3. 实验分组（A/B测试），且不得影响用户基本权益。

## 3.2 黑名单（禁止）

1. 基于争议标签做差别定价。
2. 基于争议标签拒绝预约/限制服务。
3. 基于争议标签降低会员权益或限制活动参与。
4. 把争议标签透传给第三方用于营销。

## 4. 审批与审计机制

1. 每个G1标签必须有 `label_owner`、`purpose_code`、`expiry_days`。
2. 每次调用G1标签必须记录 `request_id`、`use_case`、`decision_output`。
3. 每月输出一次偏差报告：按性别、年龄段、门店检查差异影响。
4. 用户可在隐私页申请解释：展示“为什么看到该推荐/触达”。

## 5. 运行上线拦截规则（必须同时满足）

1. 标签已在 `eb_label_policy` 中配置为启用。
2. 对应数据授权有效，且未撤回。
3. 模型版本在灰度白名单中。
4. 场景属于白名单用途。
5. 风险阈值触发人工复核时，人工复核已完成。

## 6. 最小技术落地

1. 在标签计算任务中加入 `policy_check` 阶段。
2. 触达引擎读取 `eb_label_policy` 做实时拦截。
3. 记录标签快照，保留可追溯版本（含置信度和解释）。
4. 提供 `label_disable` 一键熔断开关。

## 7. 首批可启用标签（建议）

1. `price_sensitivity`：仅用于券面力度建议，不用于基础价格差异。
2. `churn_risk_level`：仅用于召回触达，不用于权益限制。
3. `complaint_risk_level`：仅用于人工服务介入，不用于拒单。

## 8. 验收标准

1. G1标签越权使用次数 = 0。
2. G1标签每次调用均有审计记录。
3. 用户解释请求可在 15 个工作日内闭环。
4. 争议标签相关投诉率不高于基线。
