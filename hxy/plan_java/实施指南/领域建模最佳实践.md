# 领域建模最佳实践

> **目标**: 从业务需求到领域模型的系统化方法  
> **适用**: CRMEB Java DDD架构

---

## 🎯 建模流程

### 五步建模法

```
业务需求 → 识别限界上下文 → 提取聚合根 → 定义值对象 → 设计领域服务
```

---

## 步骤1：识别限界上下文

### 方法：事件风暴（Event Storming）

**输入**：业务流程描述

**输出**：限界上下文列表

**示例：预约业务**

```
用户故事：
"用户打开小程序 → 选择门店 → 选择服务 → 选择技师 → 
 选择时间 → 支付 → 到店核销 → 服务完成 → 评价"

识别领域事件：
- 预约已创建
- 预约已支付
- 预约已确认
- 服务已开始
- 服务已完成
- 预约已取消

识别限界上下文：
✅ 预约上下文（Booking Context）
✅ 支付上下文（Payment Context）
✅ 评价上下文（Review Context）
```

---

## 步骤2：提取聚合根

### 识别标准

1. **有唯一标识**（ID）
2. **有完整生命周期**（创建→变更→结束）
3. **封装业务规则**
4. **是业务核心概念**

### 示例：预约域

```
候选对象：
- 预约（Booking）✅
- 用户（User）❌ 属于会员域
- 技师（Technician）❌ 属于员工域
- 时间槽（TimeSlot）⚠️ 实体，不是聚合根
```

**判断依据**：
- Booking：有独立生命周期，封装预约规则 → ✅ 聚合根
- TimeSlot：依附于Booking，无独立业务意义 → 实体

---

## 步骤3：定义值对象

### 识别标准

1. **无唯一标识**
2. **由属性决定相等性**
3. **不可变**
4. **可替换**

### 示例：预约域

```java
// ✅ 值对象：预约状态
public enum BookingStatus {
    PENDING, CONFIRMED, CANCELLED
}

// ✅ 值对象：服务时长
public class Duration {
    private final int minutes;  // 不可变
}

// ❌ 不是值对象：用户ID
private Integer userId;  // 有标识，是引用
```

## 步骤4：设计领域服务

### 使用场景

1. **跨聚合根的业务逻辑**
2. **复杂算法**
3. **无状态操作**

### 示例：技师排班

```java
@Service
public class TechnicianScheduleService {
    /**
     * 场景：生成时间槽（复杂算法）
     */
    public List<TimeSlot> generateTimeSlots(...) {
        // 算法逻辑
    }
    
    /**
     * 场景：智能推荐（跨聚合根）
     */
    public List<TimeSlot> recommendTimeSlots(...) {
        // 需要访问用户偏好、技师排班等
    }
}
```

## 步骤5：定义仓储

### 设计原则

1. **面向领域对象**（不是PO）
2. **隐藏持久化细节**
3. **只包含必要查询**

### 示例

```java
public interface BookingRepository {
    Booking save(Booking booking);
    Booking findById(Integer id);
    List<Booking> findByUserId(Integer userId);
}
```

## 🎯 实战案例：会员域建模

### 需求描述

```
会员系统需求：
1. 用户注册成为会员
2. 消费累积成长值
3. 成长值达到阈值升级
4. 不同等级享受不同折扣
5. 会员可以查看成长值和等级
```

### 建模过程

**Step 1：识别限界上下文**

```
会员上下文（Member Context）
- 核心：会员等级管理
- 边界：不包含订单、支付
```

**Step 2：提取聚合根**

```java
public class Member {
    private Integer id;
    private Integer userId;
    private MemberLevel level;      // 值对象
    private GrowthValue growthValue; // 值对象
    private Date createTime;
}
```

**Step 3：定义值对象**

```java
// 会员等级
public enum MemberLevel {
    BRONZE(0, "青铜", 0, 0.95),
    SILVER(1, "白银", 1000, 0.90),
    GOLD(2, "黄金", 5000, 0.85);
    
    private final int level;
    private final String name;
    private final int requiredGrowth;
    private final double discount;
    
    public boolean canUpgradeTo(MemberLevel target) {
        return target.level == this.level + 1;
    }
}

// 成长值
public class GrowthValue {
    private final int value;
    
    public GrowthValue add(int amount) {
        return new GrowthValue(this.value + amount);
    }
    
    public boolean canUpgrade(MemberLevel currentLevel) {
        MemberLevel nextLevel = currentLevel.next();
        return value >= nextLevel.getRequiredGrowth();
    }
}
```

**Step 4：聚合根业务方法**

```java
public class Member {
    /**
     * 增加成长值（消费后调用）
     */
    public void addGrowthValue(int amount) {
        this.growthValue = this.growthValue.add(amount);
        
        // 自动升级检查
        if (this.growthValue.canUpgrade(this.level)) {
            this.upgrade();
        }
    }
    
    /**
     * 升级
     */
    private void upgrade() {
        MemberLevel nextLevel = this.level.next();
        if (nextLevel != null) {
            this.level = nextLevel;
            this.upgradeTime = new Date();
            // 发布领域事件
            DomainEventPublisher.publish(
                new MemberUpgradedEvent(this.id, nextLevel)
            );
        }
    }
    
    /**
     * 计算折扣价
     */
    public BigDecimal calculateDiscountPrice(BigDecimal originalPrice) {
        return originalPrice.multiply(
            BigDecimal.valueOf(this.level.getDiscount())
        );
    }
}
```

**Step 5：定义仓储**

```java
public interface MemberRepository {
    Member save(Member member);
    Member findById(Integer id);
    Member findByUserId(Integer userId);
}
```

## 🎯 常见错误

### 错误1：贫血模型

```java
// ❌ 错误：只有数据，没有行为
public class Member {
    private Integer id;
    private Integer level;
    private Integer growthValue;
    
    // 只有getter/setter
}

// 业务逻辑在Service
@Service
public class MemberService {
    public void addGrowthValue(Member member, int amount) {
        member.setGrowthValue(member.getGrowthValue() + amount);
        if (member.getGrowthValue() >= 1000) {
            member.setLevel(1);
        }
    }
}
```

```java
// ✅ 正确：充血模型
public class Member {
    private MemberLevel level;
    private GrowthValue growthValue;
    
    public void addGrowthValue(int amount) {
        this.growthValue = this.growthValue.add(amount);
        if (this.growthValue.canUpgrade(this.level)) {
            this.upgrade();
        }
    }
}
```

### 错误2：过度设计

```java
// ❌ 错误：为简单逻辑创建值对象
public class UserName {
    private final String value;
    // 只是简单的字符串，不需要值对象
}

// ✅ 正确：直接使用String
private String userName;
```

### 错误3：聚合根过大

```java
// ❌ 错误：一个聚合根管理太多东西
public class Order {
    private List<OrderItem> items;
    private Payment payment;
    private Delivery delivery;
    private Invoice invoice;
    // 太多职责
}

// ✅ 正确：拆分聚合根
public class Order {
    private List<OrderItem> items;
    // 只管理订单项
}

public class Payment {
    private Integer orderId;  // 引用
    // 独立的支付聚合根
}
```

## 🎯 检查清单

### 聚合根检查

- [ ] 是否有唯一标识？
- [ ] 是否有完整生命周期？
- [ ] 是否封装了业务规则？
- [ ] 是否避免了贫血模型？
- [ ] 是否避免了过度设计？

### 值对象检查

- [ ] 是否不可变？
- [ ] 是否实现了equals/hashCode？
- [ ] 是否有业务含义？
- [ ] 是否避免了过度使用？

### 领域服务检查

- [ ] 是否无状态？
- [ ] 是否处理跨聚合根逻辑？
- [ ] 是否避免了贫血Service？
