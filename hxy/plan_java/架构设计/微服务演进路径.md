## 阶段2：模块化单体（6-12月）

### 目标

强化领域隔离，准备数据库拆分

### 关键改造

#### 1. 引入消息队列

```
领域间通信：方法调用 → 消息队列

booking域 --[MQ]--> member域
```

**技术选型**：RocketMQ / RabbitMQ

**示例**：

```java
// 预约域：发送消息
@Service
public class BookingDomainService {
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    public Booking create(...) {
        Booking booking = Booking.create(...);
        
        // 发送消息（异步）
        mqTemplate.convertAndSend(
            "booking-created-topic",
            new BookingCreatedMessage(booking)
        );
        
        return booking;
    }
}

// 会员域：消费消息
@Service
@RocketMQMessageListener(topic = "booking-created-topic")
public class MemberMessageListener {
    public void onMessage(BookingCreatedMessage msg) {
        memberService.addGrowthValue(msg.getUserId(), msg.getAmount());
    }
}
```

#### 2. 数据库逻辑隔离

```sql
-- 当前：共享数据库
crmeb_db
├── eb_booking
├── eb_member
└── eb_order

-- 改造：Schema隔离
crmeb_db
├── booking_schema
│   └── eb_booking
├── member_schema
│   └── eb_member
└── order_schema
    └── eb_order
```

**配置多数据源**：

```java
@Configuration
public class DataSourceConfig {
    @Bean
    @ConfigurationProperties("spring.datasource.booking")
    public DataSource bookingDataSource() { }
    
    @Bean
    @ConfigurationProperties("spring.datasource.member")
    public DataSource memberDataSource() { }
}
```

#### 3. 引入服务注册中心

**技术选型**：Nacos

**目的**：为微服务拆分做准备

```yaml
# application.yml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        service: crmeb-monolith
```

## 阶段3：微服务（12-18月）

### 目标

独立部署、独立扩展

### 拆分策略

#### 优先级排序

| 优先级 | 服务 | 理由 | 复杂度 |
|-------|------|------|--------|
| P0 | booking-service | 业务独立，无强依赖 | ⭐⭐ |
| P1 | member-service | 被多个服务依赖 | ⭐⭐⭐ |
| P2 | order-service | 依赖多，事务复杂 | ⭐⭐⭐⭐ |

#### 拆分步骤（以booking-service为例）

**Step 1：创建独立项目**

```
booking-service/
├── src/
│   └── main/
│       └── java/
│           └── com/zbkj/booking/
│               ├── domain/      # 复制domain/booking
│               ├── controller/
│               └── BookingApplication.java
├── pom.xml
└── application.yml
```

**Step 2：拆分数据库**

```sql
-- 创建独立数据库
CREATE DATABASE booking_db;

-- 迁移表
CREATE TABLE booking_db.eb_booking AS 
SELECT * FROM crmeb_db.eb_booking;
```

**Step 3：服务间通信改造**

```java
// 原来：直接调用
Member member = memberRepository.findById(userId);

// 改造：Feign调用
@FeignClient("member-service")
public interface MemberClient {
    @GetMapping("/api/member/{id}")
    MemberDTO getById(@PathVariable Integer id);
}

MemberDTO member = memberClient.getById(userId);
```

**Step 4：分布式事务处理**

```java
// 场景：创建预约 + 扣减会员积分

// 方案1：Saga模式（推荐）
@Service
public class BookingSaga {
    public void createBooking(...) {
        // 1. 创建预约
        Booking booking = bookingService.create(...);
        
        try {
            // 2. 扣减积分（远程调用）

        } catch (Exception e) {

// 方案2：最终一致性（消息队列）
@Service

    public Booking create(...) {
        Booking booking = Booking.create(...);
```

## 🎯 风险与挑战

### 挑战1：分布式事务

**问题**：跨服务的数据一致性

**解决方案**：
1. Saga模式（补偿）
2. 最终一致性（消息队列）
3. TCC（预留-确认-取消）

### 挑战2：性能下降

**问题**：网络调用 vs 本地调用

**解决方案**：
1. 缓存（Redis）
2. 批量查询
3. 异步化

### 挑战3：运维复杂度

**问题**：多服务部署、监控

**解决方案**：
1. Docker + K8s
2. 统一日志（ELK）
3. 统一监控（Prometheus）

## 🎯 决策树

```
是否需要拆分微服务？
│
├─ 团队规模 < 10人？ → ❌ 不拆，保持单体
├─ 日活 < 10万？ → ❌ 不拆，单体足够
├─ 业务变化频繁？ → ✅ 考虑拆分
├─ 需要独立扩展？ → ✅ 考虑拆分
└─ 技术债务严重？ → ⚠️ 先重构，再拆分
```
