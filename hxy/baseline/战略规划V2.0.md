## 📋 阶段1：MVP验证（4周）

### 目标

**用最小成本验证：用户是否愿意用小程序预约足疗**

### 核心功能（仅5个）

| 功能 | 优先级 | 开发周期 | 验证目标 |
|------|--------|----------|----------|
| 1. 门店列表（LBS） | P0 | 2天 | 用户能找到附近门店 |
| 2. 服务预约 | P0 | 5天 | 用户能完成预约 |
| 3. 订单管理 | P0 | 3天 | 用户能查看订单 |
| 4. 到店核销 | P0 | 2天 | 门店能核销订单 |
| 5. 微信支付 | P0 | 2天 | 用户能在线支付 |

### 技术方案

**不追求完美架构，追求快速上线**

```
前端：UniApp（复用CRMEB模板）
后端：ThinkPHP 6（改造CRMEB订单模块）
数据库：MySQL（扩展eb_store_order表）
部署：Docker（快速部署）
```

### 成功指标

- 注册转化率 > 20%
- 首单转化率 > 30%
- 订单完成率 > 90%
- 用户满意度 > 4.0/5.0

### 失败快速止损

如果4周后指标不达标：
1. 分析失败原因（产品/运营/技术）
2. 快速调整方向
3. 避免继续投入DDD重构

## 🎯 核心决策

### 决策1：先MVP后DDD

**理由**：
- 商业模式未验证，过度设计浪费资源
- DDD架构需要业务稳定后才能发挥价值
- 快速试错比完美架构更重要

### 决策2：前端优先

**理由**：
- 用户体验决定成败
- 后端已有CRMEB基础，可快速改造
- 前端是当前最大瓶颈

### 决策3：数据驱动

**理由**：
- 埋点先行，数据指导决策
- A/B测试验证假设
- 用户反馈快速迭代
