# HXY 全新工作计划（0-50店，RuoYi版）V1（2026-02-22）

## 1. 现状基线

1. 代码主线已切换到 `ruoyi-vue-pro-master`，不再以 CRMEB 为开发基座。
2. 支付方向已定版为“总部支付中台 + 服务商 + 特约商户 + 官方分账”。
3. 当前已具备支付主链路基础能力：幂等、状态机、回调、退款、补偿、上线拦截。
4. 本周新增已落地：服务/实物统一商品类型 `productType` 贯通到商品与交易链路，并有服务单支付后处理骨架。
5. 外部依赖已解除：`sub_mchid=1106655249`（到位时间：2026-02-24），真实服务商链路已可进入全量联调。

## 2. 目标（0-50店）

1. 在 2026-05-17 前完成 0-50 店可运营支付体系。
2. 达成“支付稳定 + 服务履约闭环 + 总部门店账务清晰 + 可灰度可回滚”。
3. 以“支付优先、业务闭环第二、规模治理第三”的顺序推进，不做超前复杂化。

## 3. 规划原则

1. 资金与合规优先于功能速度。
2. 所有关键链路必须可观测、可补偿、可审计。
3. 先做可上线最小闭环，再做高阶优化。
4. 预收款在 0-50 店阶段默认单店隔离，不做跨店通用。
5. “1人+AI”执行模式下，文档、脚本、门禁必须产品化。

## 4. 时间与阶段（12周）

1. 阶段 A（2026-02-23 至 2026-03-08）：P0 收口与真实联调准备。
2. 阶段 B（2026-03-09 至 2026-04-05）：服务商主路径联调上线与服务履约闭环。
3. 阶段 C（2026-04-06 至 2026-05-03）：30-50 店规模化治理与 K8s 运行治理。
4. 阶段 D（2026-05-04 至 2026-05-17）：生产灰度收口与非支付模块二开起跑。

## 5. 阶段 A 任务（0-10店可用）

1. 支付合规与门禁收口。  
   交付物：支付配置核验脚本、回放脚本、上线拦截规则 v1、回滚 Runbook。  
   验收：18类异常回放通过率 100%，门禁未通过禁止发布。
2. 服务/实物统一商品模型完成第一版。  
   交付物：SPU/订单项 `productType` 全链路、数据库增量脚本、编译与测试脚本。  
   验收：创建商品、下单、支付成功后类型字段不丢失。
3. 服务订单履约骨架。  
   交付物：支付后服务订单处理器、预约单创建接口占位、审计日志。  
   验收：服务型订单支付后能稳定进入服务履约队列（即使先空实现也必须有记录）。
4. 退款分层策略固化。  
   交付物：秒退规则表、人工复核规则表、工单升级规则。  
   验收：低风险退款自动执行，高风险退款自动进入总部复核工单。

## 6. 阶段 B 任务（10-30店可用）

1. 服务商真实链路联调。  
   交付物：`sp_mchid + sub_mchid` 下单、查单、关单、退款、退款查询联调报告。  
   验收：支付账、交易账、渠道账三账一致；关键接口成功率 >= 99.95%。
2. 特约商户进件流程自动化。  
   交付物：门店进件申请、状态跟踪、失败原因回填、重试机制。  
   验收：新门店接入平均时长可量化，且全流程可追溯。
3. 服务订单状态机与核销闭环。  
   交付物：待到店、服务中、已完成、售后中的状态机与门店核销操作。  
   验收：状态单向流转，幂等可重复调用，终态不可逆覆盖。
4. 总部结算与门店经营账分层。  
   交付物：支付账、交易账、结算账分层报表与差异对账入口。  
   验收：T+1 自动对账，差异自动建单，不依赖人工盯盘。

## 7. 阶段 C 任务（30-50店可用）

1. K8s 运行治理落地。  
   交付物：支付关键任务迁移到 K8s CronJob、队列消费、告警面板。  
   验收：补偿任务支持失败重试、任务幂等、可观测。
2. 灰度发布与快速回滚体系。  
   交付物：分批门店灰度策略、门店白名单、自动回滚脚本。  
   验收：灰度异常可在 10 分钟内回退到上一稳定版本。
3. 退款与补偿 SLA 运营化。  
   交付物：自动秒退比例、人工复核时长、补偿收敛时长仪表盘。  
   验收：退款终态收敛和补偿积压可按日监控，触发阈值自动告警。

## 8. 阶段 D 任务（收口与二开起跑）

1. 生产收口。  
   交付物：门店接入手册、运维值班手册、支付故障演练记录。  
   验收：连续 2 周无 P0 支付事故。
2. 非支付模块二开启动。  
   交付物：商品、会员、门店运营三个二开 Backlog 与 Sprint 计划。  
   验收：支付模块不被非支付需求反向破坏，形成独立发布节奏。

## 9. P0/P1/P2 优先级

1. P0：支付链路稳定、服务履约闭环、退款合规、上线门禁。
2. P1：进件自动化、结算账分层、K8s 任务治理、灰度策略。
3. P2：营销增强、用户增长玩法、跨店预收款策略评估。

## 10. 门店规模触发规则

1. 门店 <= 10：允许过渡策略，但必须具备主路径切换预案。
2. 门店 10-30：服务商主路径必须成为默认路径。
3. 门店 30-50：必须进入 K8s 任务治理与灰度发布模式。

## 11. 量化验收指标（统一）

1. 支付成功率 >= 99.95%。
2. 支付终态收敛时延 P95 <= 5 分钟。
3. 自动秒退命中单据成功率 >= 99%。
4. 对账差异率 <= 0.01%，差异单 24 小时内闭环。
5. 关键补偿任务成功率 >= 99.9%。
