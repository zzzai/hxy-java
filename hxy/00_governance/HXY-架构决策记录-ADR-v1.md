# HXY 架构决策记录（ADR v1）

- 日期：2026-03-01
- 说明：每条决策必须可追溯、可回滚、可验证。

## ADR-001：采用“并轨增量”而非“大重构替换”

- 背景：现网已运行在 RuoYi/Yudao 主链，支付与交易链路不可中断。
- 决策：保留 `product/trade/pay` 主链，新增模板/EAV/门店映射能力并轨接入。
- 影响范围：商品域、交易域、门店治理、发布流程。
- 备选方案：一次性迁移到全新模型。
- 否决原因：风险过高、回归面过大、上线窗口不可控。
- 回滚条件：并轨能力导致主链性能或稳定性下降时可回退到旧路径。

## ADR-002：商品采用“服务/实物/套餐”三模型并存

- 背景：品牌属于服务+实物混合业务，不适配单一电商模型。
- 决策：服务按时间容量履约，实物按库存容量履约，套餐按子项拆分履约。
- 影响范围：下单、履约、退款、提成、报表。
- 回滚条件：套餐拆分造成履约不可控时可降级为单品售卖。

## ADR-003：技师提成按履约完成计提

- 背景：支付成功即计提会导致退款倒挂与财务争议。
- 决策：服务履约完成后计提，退款触发冲正单。
- 影响范围：提成、结算、财务对账。
- 回滚条件：履约状态异常率过高时可临时切换为人工审核发放。

## ADR-004：价格决策顺序统一固化

- 背景：总部价、门店价、活动价叠加导致门店口径不一致。
- 决策：统一顺序为“总部->门店->活动->权益抵扣”，并记录价格来源快照。
- 影响范围：结算、退款、风控、客服解释口径。
- 回滚条件：活动系统异常时允许降级到“总部/门店价”。

## ADR-005：库存流水幂等唯一键

- 背景：重复回调和重试可能导致重复扣库存。
- 决策：库存流水唯一键 `biz_type + biz_no + store_id + sku_id`。
- 影响范围：交易、库存、对账、监控。
- 回滚条件：无（基础一致性约束，不可关闭）。

## ADR-006：门店治理采用“合约策略 + 规则引擎外置”

- 背景：万店场景下，直营/加盟门店权限差异大，硬编码权限不可持续。
- 决策：门店权限边界采用合约策略配置，字段级校验采用规则引擎配置驱动。
- 影响范围：门店调价、调库存、上下架、活动叠加、审批流。
- 备选方案：按门店类型写死权限分支。
- 否决原因：门店类型扩展需要频繁发版，运营响应慢且高风险。
- 回滚条件：规则引擎异常时可降级为“总部只读+门店最小运营权限”。

## ADR-007：订单项快照不可变（价格/规则/版本/套餐子项）

- 背景：总部价盘、套餐子项、规则会持续迭代，历史订单必须可追溯。
- 决策：下单写入订单项全量快照，核销与售后以快照为准，不反查当前配置。
- 影响范围：交易、履约、退款、提成、对账、客诉。
- 备选方案：运行时反查最新规则和套餐关系。
- 否决原因：会导致历史订单口径漂移，无法审计，财务风险高。
- 回滚条件：无（基础财务一致性约束，不可关闭）。

## ADR-008：总部-区域-门店采用角色化运行模型（RACI 固化）

- 背景：多门店运行中存在职责重叠与空白，导致问题难以闭环。
- 决策：在产品与流程层面固定角色边界（总部/区域/门店/客服/供应链/财务/合规/运维）。
- 影响范围：权限模型、审批流、工单流、运营看板。
- 备选方案：继续按组织习惯口头约定职责。
- 否决原因：无法形成可审计、可门禁的稳定运行机制。
- 回滚条件：无（组织运行基础约束，不建议回滚）。

## ADR-009：售后统一工单中台，争议处理单轨闭环

- 背景：退款、履约争议、提成争议分散处理，口径不一致。
- 决策：统一工单入口，按 SLA 分级与升级，最终仲裁结果写回订单/提成/结算链。
- 影响范围：客服、门店、区域、财务、合规。
- 备选方案：各模块自管售后与争议。
- 否决原因：会导致跨模块数据断裂和重复协同成本。
- 回滚条件：工单系统不可用时可临时降级人工台账，但需补录。

## ADR-010：门店生命周期纳入标准流程引擎

- 背景：开店、停业、闭店、迁店、加盟退出缺少统一流程规范。
- 决策：生命周期事件标准化并纳入流程引擎，强制触发库存、订单、结算、权限清理。
- 影响范围：门店治理、供应链、交易、财务、审计。
- 备选方案：按线下流程人工处理。
- 否决原因：规模化后错误率高且无法追溯责任。
- 回滚条件：流程引擎故障时可启动紧急手工流程并补录审计记录。

## ADR-011：线上配置权限收敛为“总部-门店”两级

- 背景：万店扩张中，线上多层配置链路会放大决策延迟与权限漂移风险。
- 决策：线上系统配置管理仅保留“总部、门店”两级角色；训练、巡检、突发处置等岗位作为线下运行职能，通过总部工单与台账协同，不新增线上配置层级。
- 影响范围：RBAC、审批流、工单流、门店治理流程。
- 备选方案：维持总部-区域-门店三层线上配置。
- 否决原因：配置路径变长、口径不一致、审计复杂度提升。
- 回滚条件：无（作为组织与权限治理基线长期生效）。

## ADR-012：库存流水执行改为“租约抢占 + 状态机重试”

- 背景：并发重试场景下，同一库存流水可能被重复执行，导致库存重复变更。
- 决策：引入 `PROCESSING` 状态；执行前 CAS 抢占，执行后显式落 `SUCCESS/FAILED`；超时 `PROCESSING` 自动回收为 `FAILED`。
- 影响范围：门店库存扣减、补偿任务、异常监控。
- 备选方案：维持 `PENDING/FAILED` 直接执行。
- 否决原因：无法避免并发重复生效，幂等一致性风险高。
- 回滚条件：仅在紧急故障下可临时降级为串行执行，后续必须恢复状态机模型。

## ADR-013：长记忆机制采用“Git 事实源 + 门禁强校验”，当前阶段不引入外部向量库

- 背景：多窗口并行开发时，聊天上下文不可作为稳定记忆载体。
- 决策：以三份治理文档 + `memory-index.yaml` 作为唯一事实源，CI/发布脚本强制门禁；窗口切换必须写 handoff 文档。
- 影响范围：研发流程、发布流程、跨窗口协同效率。
- 备选方案：引入外部向量数据库做长期记忆。
- 否决原因：当前项目阶段优先保证可审计与可回放，Git 内文档更直接、成本更低、治理边界更清晰。
- 回滚条件：无（流程治理基线）；若后续规模扩大可在不替换现有机制的前提下增配向量检索层。

## ADR-014：订单项模板快照采用“请求透传 + 订单固化”分阶段落地

- 背景：交易域目前缺少对模板版本、模板快照、价格来源的订单级固化，导致审计链路不完整。
- 决策：先以请求透传方式打通 `App 请求 -> 价格计算 -> trade_order_item` 的三字段固化；后续再升级为“仅允许后端从已发布模板版本自动取快照”。
- 影响范围：交易下单、订单查询、售后审计、结算追溯。
- 备选方案：在当前阶段直接强制后端反查模板发布快照并阻断缺失下单。
- 否决原因：现阶段商品主数据尚未建立 `SKU/SPU -> 模板版本` 稳定映射，直接强制会阻断存量下单链路。
- 回滚条件：若前端透传异常率高，可临时回退为字段可空写入，保持主链路稳定。

## ADR-015：模板版本与服务分流校验前置到价格计算阶段

- 背景：若仅在下单落库或发货阶段校验，会导致无效订单进入后续链路，增加支付前失败与客服成本。
- 决策：在 `TradePriceServiceImpl` 前置两类校验：一是服务商品禁止 `EXPRESS`；二是当携带 `templateVersionId` 时强制快照非空，并校验模板“存在、已发布、类目匹配”。
- 影响范围：交易下单前置校验、订单有效性、客服失败口径一致性。
- 备选方案：继续在发货或支付后阶段兜底校验。
- 否决原因：失败反馈过晚，且会放大异常订单沉淀与排障成本。
- 回滚条件：如出现误拦截，可临时放宽为仅记录告警不阻断；排查后恢复强校验。

## ADR-016：模板快照采用“客户端优先 + 服务端回填”双保险

- 背景：交易侧已透传 `templateVersionId/templateSnapshotJson`，但客户端偶发缺失快照会直接导致下单失败，影响稳定性。
- 决策：当请求携带 `templateVersionId` 且 `templateSnapshotJson` 为空时，价格计算阶段优先使用模板版本 `snapshotJson` 自动回填；若模板版本快照也为空，继续按强约束拒绝。
- 影响范围：交易下单稳定性、模板快照审计完整性、客户端容错。
- 备选方案：继续强制要求客户端必须传快照，不做服务端兜底。
- 否决原因：会把可恢复的数据缺失问题放大成硬失败，增加下单抖动和客服成本。
- 回滚条件：若发现模板快照回填导致审计口径混乱，可临时回退为“仅客户端透传”，并同步修复前端采集链路后再恢复。

## ADR-017：服务商品模板版本强绑定（交易前置必填）

- 背景：服务商品履约与售后强依赖模板快照，若允许缺失 `templateVersionId` 会导致价格与规则审计链路断裂。
- 决策：在价格计算阶段前置强约束：命中 `service` 商品且请求项缺失 `templateVersionId` 时，直接拒绝下单计算。
- 影响范围：服务商品下单入口、模板审计完整性、售后与结算追溯。
- 备选方案：继续仅在“已传 templateVersionId”时校验，缺失时放行。
- 否决原因：会形成不可追溯服务订单，后续补数与客诉成本高。
- 回滚条件：若出现大面积历史兼容问题，可临时降级为告警模式，并在完成 `SPU/SKU -> templateVersionId` 主数据补齐后恢复强约束。

## ADR-018：模板版本从“请求透传”升级为“主数据显式绑定 + 请求兜底”

- 背景：仅依赖前端透传 `templateVersionId` 存在缺参风险，无法保证服务商品下单链路稳定。
- 决策：在 `product_spu`、`product_sku` 增加 `templateVersionId` 主数据字段；保存服务商品时强校验模板“存在、已发布、类目匹配”；交易侧请求缺失时按 `SKU -> SPU` 顺序回填。
- 影响范围：商品保存、下单价格计算、模板快照审计。
- 备选方案：继续只依赖请求层透传，不建设主数据绑定。
- 否决原因：会持续出现链路偶发缺参，导致可恢复问题升级为硬阻断。
- 回滚条件：主数据迁移异常时，可临时仅保留交易侧兜底读取，待回填完成后恢复保存态强校验。

## ADR-019：预约缺失采用“自动占位 + 幂等回写”闭环

- 背景：支付成功后若 booking 侧未及时建单，服务履约单长期停留待预约，影响履约和客服处理。
- 决策：`ServerTradeServiceBookingGateway` 按 `payOrderId` 查 booking；查不到时自动创建占位预约单（幂等）；创建成功后统一调用 `markBooked` 回写服务履约单。
- 影响范围：支付后履约收口、预约单可追溯性、待预约积压治理。
- 备选方案：保持“仅查询不创建”，完全依赖异步补偿任务。
- 否决原因：补偿时效不可控，链路状态不收敛。
- 回滚条件：若占位策略引发异常创建，可临时降级为查询模式并保留告警。

## ADR-020：支付 P0 发布门禁新增 stageA-p0-23 专项 CI

- 背景：模板绑定与预约占位属于支付主链高风险改动，需要独立、可阻断的快速回归门禁。
- 决策：新增 `run_payment_stagea_p0_23.sh` 与 `payment-stagea-p0-23.yml`，默认执行命名门禁、记忆门禁及四组关键单测（Product/Trade/Booking/Server）。
- 影响范围：PR 检查、发布前质量门禁、required checks 自动配置。
- 备选方案：继续依赖通用模块测试或人工回归。
- 否决原因：缺少可复用、可追踪的专属阻断门禁，容易在并行开发中漏测。
- 回滚条件：若 CI 时长超标，可保留门禁框架并调整用例分层，不取消阻断检查。

## ADR-021：套餐售后退款上限采用“子项履约优先 + 显式值兜底”

- 背景：套餐快照中同时存在 `bundleRefundablePrice` 与 `bundleChildren` 时，若优先使用显式总额，会忽略子项履约状态，导致超退风险。
- 决策：退款上限计算优先读取 `bundleChildren` 并按子项履约/可退状态汇总；仅在子项不可计算时回退显式 `bundleRefundablePrice`。
- 影响范围：售后申请校验、退款金额上限、财务审计、客服争议处理。
- 备选方案：始终优先 `bundleRefundablePrice`，子项信息只做展示。
- 否决原因：无法保证“履约后不可退子项”被排除，审计证据不足。
- 回滚条件：若历史快照大量缺失子项字段，可临时降级显式值优先并同步补齐快照治理。

## ADR-022：服务履约单 FINISHED 采用“状态优先阻断退款”

- 背景：历史脏数据场景下，服务履约单已完成但快照缺失，旧逻辑会回退到订单项快照/实付金额，存在错误放行退款风险。
- 决策：当存在服务履约单且状态为 `FINISHED` 时，退款上限直接为 0；快照仅作为审计补充信息，不再决定是否阻断。
- 影响范围：售后申请校验、客服退款口径、履约完成态审计一致性。
- 备选方案：仅在存在有效快照时才阻断，快照缺失时继续回退。
- 否决原因：会把数据缺失问题放大为资金风险，无法保障“履约完成不可退”底线。
- 回滚条件：若业务策略调整为“履约完成后仍允许特批退款”，需引入审批白名单后再评估放开逻辑。

## ADR-023：售后列表保留“退款上限来源”可检索维度

- 背景：退款规则增强后，运营与客服需要快速按限退来源分组排查异常单（服务履约快照口径、订单项快照口径、实付兜底口径）。
- 决策：管理后台售后分页 `AfterSalePageReqVO` 增加 `refundLimitSource` 条件，并在 Mapper 侧按字段精确过滤。
- 影响范围：售后运营排障效率、客服问题定位、规则效果回溯。
- 备选方案：仅导出后离线筛选，不在线上列表提供条件。
- 否决原因：排障时效差，无法满足实时运营与客服联动需求。
- 回滚条件：若后续统一改为专门审计看板，可保留字段但移除分页入口筛选。
