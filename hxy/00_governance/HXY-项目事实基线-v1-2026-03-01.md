# HXY 项目事实基线（v1）

- 日期：2026-03-01
- 用途：作为开发与评审的唯一事实基线（Single Source of Truth）

## 1. 品牌与业务

1. 品牌：荷小悦（新中式·疗愈·社区）。
2. 业务形态：服务项目 + 实物商品 + 套餐组合混合 O2O。
3. 连锁目标：先跑通单店，再复制到 50 店，再区域化扩张。

## 2. 核心业务约束

1. 服务商品核心容量是技师时间，不以库存扣减为核心。
2. 实物商品核心容量是库存，必须具备幂等库存流水。
3. 套餐必须拆子项，按子项履约与退款。
4. 技师提成只针对服务子项，按履约完成计提。

## 3. 组织边界

1. 总部负责：主数据标准、价格边界、活动规则、结算治理。
2. 门店负责：上架、排班、门店价执行、现场履约。
3. 线上配置权限：仅保留总部与门店两级（不设区域配置层）。
4. 线下运行职能：训练、巡检、突发处置可存在，但通过总部工单/台账协同，不形成独立线上配置层。
5. 客服负责：统一售后工单受理、分级、升级与闭环。
6. 供应链负责：补货、调拨、盘点、损耗治理。
7. 财务与审计负责：月结、冲正、四账一致性与审计留痕。

## 4. 交易与支付

1. 支付架构：服务商模式 + 特约商户体系。
2. 资金原则：合规分账、可对账、可审计、可追溯。
3. 退款原则：与履约状态和子项状态绑定。

## 5. 当前工程并轨原则

1. 保留现有 `product_spu/product_sku/trade_order` 主链。
2. 通过增量改造补齐强约束，不做一次性推翻重写。
3. 所有新能力必须可回退。

## 6. 不可破坏项

1. 下单/支付主链稳定性。
2. 退款与对账可追溯性。
3. 多门店隔离与权限边界。
4. 价格与库存幂等一致性。

## 7. 2026-03-02 最新工程事实

1. 门店库存流水已采用抢占执行模型：`PENDING/FAILED -> PROCESSING -> SUCCESS|FAILED`，并带处理租约超时回收。
2. 库存幂等键冲突已强校验：同 `biz_type + biz_no + store_id + sku_id` 不允许出现不同 `incr_count`。
3. 库存业务标识字段不再截断，长度超限直接拒绝，避免幂等键碰撞。
4. SKU 提交链路已补并发幂等兜底：`DuplicateKeyException` 后回查幂等任务返回命中结果，不再返回 500。
5. 长记忆门禁升级为“路径映射 + 内容有效性”强制模式；核心领域改动需同步三份治理文档。
6. 长记忆落地采用 Git 文档事实源，不依赖聊天历史；交接文件统一落 `hxy/07_memory_archive/handoffs/`。
7. 交易下单链路已补模板快照字段透传：`templateVersionId/templateSnapshotJson/priceSourceSnapshotJson` 已打通 `App 下单请求 -> 价格计算 -> trade_order_item`。
8. `trade_order_item` 已补增量脚本 `2026-03-02-hxy-trade-order-item-template-snapshot.sql`，用于模板/价格来源快照落库。
9. 服务预约网关当前生效实现为 `com.hxy.server.config.booking.ServerTradeServiceBookingGateway`（按 `payOrderId` 对齐 booking 订单并幂等标记 `BOOKED`）。
10. 价格计算阶段已前置服务/实物分流强约束：`deliveryType=EXPRESS` 且命中服务商品时，直接拒绝下单计算。
11. 价格计算阶段已补模板版本强校验（当请求携带 `templateVersionId`）：若请求缺少 `templateSnapshotJson`，服务端优先使用模板版本 `snapshotJson` 自动回填；若仍为空则拒绝下单计算；并校验模板版本“存在、已发布、类目匹配 SPU”。
12. 交易域模板校验已改为通过 Product API (`ProductTemplateVersionApi`) 读取模板版本，避免跨域直接访问 DAL。
13. 价格计算阶段已补“服务商品模板版本必填”约束：命中 `service` 商品但缺失 `templateVersionId` 时，直接拒绝下单计算。
14. 商品主数据已补 `SPU/SKU.templateVersionId` 字段与迁移脚本：`product_spu`、`product_sku` 均可显式绑定模板版本，服务商品保存时强校验“存在、已发布、类目匹配”。
15. 服务商品模板版本读取已支持主数据兜底：下单请求未传 `templateVersionId` 时，价格计算阶段按 `SKU -> SPU` 顺序回填，并继续执行发布态与快照校验。
16. 预约网关已实接占位创建：`ServerTradeServiceBookingGateway` 查不到 `payOrderId` 对应 booking 时，自动调用 `BookingOrderService.createPlaceholderOrder` 并回写服务履约单 `markBooked`。
17. 预约模块测试基线已补齐 `booking_order.dispatch_mode/parent_order_id/is_addon/addon_type` 字段，确保占位链路单测与 H2 结构一致。
18. 支付 P0 收口 CI 已新增 `hxy-payment-stagea-p0-23`：默认执行命名门禁、记忆门禁、模板绑定单测、交易模板回填单测、预约占位单测、网关联调单测。
19. 套餐售后退款上限已改为“子项履约优先”：当快照存在 `bundleChildren` 时，按子项履约/可退状态汇总可退上限；仅在无有效子项结果时才回退 `bundleRefundablePrice` 显式值。
20. 售后退款上限审计明细已增强：统一记录 `snapshotField/serviceSnapshotField`，并固化 `bundleChildren` 证据（`childCode/refundCapPrice/fulfilled/refundable/included`）。
21. 售后退款守卫已补强：当存在服务履约单且状态为 `FINISHED` 时，退款上限直接收敛为 0（即使履约快照缺失），避免因脏数据回退到订单项实付金额而放行退款。
22. 管理后台售后分页已支持按 `refundLimitSource` 精确筛选，便于运营定位不同限退口径（`SERVICE_ORDER_SNAPSHOT/ORDER_ITEM_PRICE_SOURCE/ORDER_ITEM_PAY_PRICE`）的异常单。

24. 管理后台售后详情已补 `refundLimitSourceLabel/refundLimitRuleHint`：后端根据 `refundLimitSource` 输出可读标签与规则说明，减少前端枚举映射与文案维护成本。
25. 售后统一工单分页已补 SLA 可检索维度：新增 `overdue`（仅待处理状态按 `slaDeadlineTime` 判定）与 `lastActionTime` 时间窗筛选。
26. 售后统一工单已对齐审计字段：新增并落库 `lastActionCode/lastActionBizNo/lastActionTime`，覆盖创建、规则再触发、SLA 自动升级、人工/自动收口全动作。
