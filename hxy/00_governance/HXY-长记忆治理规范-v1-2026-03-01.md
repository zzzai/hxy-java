# HXY 长记忆治理规范（v1）

- 版本：v1
- 日期：2026-03-01
- 目标：让项目文档和代码长期同频，避免“有文档但开发偏航”。

## 1. 长记忆定义

本项目“长记忆”不是对话缓存，而是可审计、可门禁、可追溯的工程资产：

1. 事实基线（不可随意漂移）。
2. 架构决策（可追溯为何这样做）。
3. 执行状态（已完成/部分完成/未开始）。
4. 代码证据（脚本、SQL、测试、报告）。

## 2. 四层记忆模型

### 2.1 L1 事实基线
文件：`hxy/00_governance/HXY-项目事实基线-v1-2026-03-01.md`

记录长期稳定事实：
1. 品牌定位与目标人群。
2. 商品模型（服务/实物/套餐）。
3. 支付与分账主体关系。
4. 门店与总部权责边界。

### 2.2 L2 架构决策 ADR
文件：`hxy/00_governance/HXY-架构决策记录-ADR-v1.md`

每条 ADR 固定字段：
1. 背景。
2. 决策。
3. 影响范围。
4. 备选方案。
5. 回滚条件。
6. 生效日期。

### 2.3 L3 执行状态看板
文件：`hxy/06_roadmap/HXY-执行状态看板-v1-2026-03-01.md`

要求：
1. 每个需求项有状态（Done/Partial/Todo）。
2. 每项绑定代码路径和脚本证据。
3. 每周至少更新一次。

### 2.4 L4 变更证据
目录：`ruoyi-vue-pro-master/script/dev` 与 `sql/mysql/hxy`

要求：
1. 关键能力必须有校验脚本或测试。
2. 发布门禁必须可自动执行。
3. 脚本输出可落盘到发布证据包。

## 3. 更新规则（强制）

1. 任何影响交易、库存、价格、履约、支付、提成的改动，必须同步更新 L3。
2. 任何影响架构边界的改动，必须新增或更新 ADR（L2）。
3. 任何改变品牌定位、业务边界的改动，必须更新 L1。
4. 代码合并前，必须通过“记忆门禁脚本”。

## 4. 记忆门禁

门禁脚本：`ruoyi-vue-pro-master/script/dev/check_hxy_memory_guard.sh`

CI 工作流：
1. `ruoyi-vue-pro-master/.github/workflows/hxy-memory-guard.yml`（PR/Push 强制检查）。
2. `ruoyi-vue-pro-master/.github/workflows/payment-stagea-p0-19-20.yml`（发布入口联动检查）。

门禁逻辑：
1. 如果本次改动涉及核心业务代码/SQL（product/trade/pay/hxy sql）。
2. 但未更新以下任一记忆文件，则阻断：
   - `hxy/00_governance/HXY-项目事实基线-v1-2026-03-01.md`
   - `hxy/00_governance/HXY-架构决策记录-ADR-v1.md`
   - `hxy/06_roadmap/HXY-执行状态看板-v1-2026-03-01.md`

## 5. 角色分工

1. 产品负责人：维护 L1 与 PRD。
2. 技术负责人：维护 ADR 与门禁脚本。
3. 研发执行：每次改动更新 L3 并附证据路径。
4. 发布负责人：门禁通过后才允许发布。

## 6. 失真修复流程

当出现“文档与代码不一致”时：

1. 24小时内创建修复条目（L3）。
2. 标注受影响模块与风险级别。
3. 补充或回滚代码，更新 ADR。
4. 复盘后固化为新的门禁检查项。

## 7. 最小执行动作（从今天开始）

1. 每次合并前执行 `check_hxy_memory_guard.sh`。
2. 每个里程碑结束更新一次 `HXY-执行状态看板`。
3. 每周一固定审阅 L1/L2/L3 一致性。
