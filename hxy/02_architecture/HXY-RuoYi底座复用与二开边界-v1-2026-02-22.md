# HXY RuoYi 底座复用与二开边界 V1

- 日期：2026-02-22
- 目标：明确“可直接复用”和“必须自建”的边界，避免过度开发或错误依赖。
- 代码基线：`/root/crmeb-java/ruoyi-vue-pro-master`

## 1. 定位结论

1. RuoYi-Vue-Pro 是“后台管理与基础能力底座”，不是荷小悦完整 O2O 业务系统。
2. 正确做法不是“整体照搬”，而是“底座复用 + 业务域自建”。
3. 0-100 店阶段优先复用非差异化能力，把研发资源集中到预约、履约、健康数据与门店经营域。

## 2. 当前仓库实况（已验证）

1. 根 `pom.xml` 已启用：`system`、`infra`、`member`、`pay`、`mall`。
2. 根 `pom.xml` 已注释：`bpm`、`report`、`mp`、`crm`、`erp`、`iot`、`ai`。
3. `yudao-server/pom.xml` 同步保持上述取舍，属于“轻量可跑”状态。

## 3. 直接复用清单（不重复造轮子）

1. RBAC 权限、菜单、数据权限与操作日志。
2. 多租户基础设施（租户上下文、租户过滤、租户缓存）。
3. 微信支付/分账/退款基础能力（在 `yudao-module-pay` 上二开）。
4. 定时任务、消息通知、文件存储、短信与代码生成器。
5. WebSocket 与基础可观测能力。

## 4. 必须自建清单（荷小悦核心差异化）

1. 预约调度域：技师排班、时段容量、锁位与冲突处理。
2. 履约核销域：到店核销、服务完成、逆向冲正。
3. 服务 SPU/SKU 域：总部主数据 + 门店运营配置（价格、上下架、可售时段）。
4. 健康档案域：问卷、体质标签、服务记录、推荐策略输入。
5. 结算台账域：门店应结、技师收益、结算批次、对账与审计。
6. C 端用户小程序（预约、支付、报告、会员权益）业务前端。

## 5. 关键提醒（容易踩坑）

1. “支持小程序”不等于“现成可用的荷小悦 C 端小程序”。
2. 多租户是“租户隔离”框架，不等于“总部-门店-加盟”业务模型已落地。
3. 代码生成器用于提速，不可替代领域建模与状态机设计。
4. 支付回调是事实来源，前端支付结果只做交互提示。

## 6. 模块取舍建议（当前阶段）

1. 保留：`system`、`infra`、`member`、`pay`、`mall`。
2. 继续禁用：`bpm`、`crm`、`erp`、`mp`、`report`、`iot`、`ai`（待确有需求再启用）。
3. 原则：先稳定单体，后做服务化拆分；先跑通业务，后扩展能力。

## 7. 3 天启动清单（执行版）

1. Day 1：跑通本地后端 + 管理端，确认登录、租户、权限链路。
2. Day 2：建立荷小悦首批核心表（订单、履约、台账、排班），用代码生成器出基础 CRUD。
3. Day 3：接通支付下单与回调，把“支付成功 -> 写订单 -> 写台账”闭环跑通。

## 8. 验收标准

1. 能从管理端创建门店、技师、服务商品并发布可售。
2. 能完成一笔真实或沙箱支付并正确收敛订单终态。
3. 能在后台看到门店与技师台账，并生成结算批次记录。
4. 能回放支付异常（回调重复、超时、失败）并保持数据一致。
