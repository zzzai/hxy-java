# HXY 技术架构规划 V2（2026-02-22）

## 1. 架构目标

1. 以 RuoYi 模块化单体为主，建立清晰的领域边界。
2. 支付域与交易域解耦，渠道接入统一由支付模块管理。
3. 具备可灰度、可回滚、可审计、可追踪的上线能力。

## 2. 分层与边界

1. 接入层：`admin-api` / `app-api` / 小程序。
2. 应用层：Controller + Application Service（编排与鉴权）。
3. 领域层：Trade、Pay、Member、Product、Ops。
4. 基础设施层：MySQL、Redis、MQ（按需）、日志与监控。

## 3. 模块治理

1. 交易模块只通过支付 API 调用支付域，不允许直连渠道 SDK。
2. 支付回调必须“验签 -> 快速应答 -> 异步落库处理”。
3. 兼容层（含旧 CRMEB 语义）仅保留过渡能力，禁止新增业务逻辑。

## 4. 部署架构

1. 公网入口：Nginx + SSL，按域名分流。
2. 内网服务：RuoYi 后端 + MySQL/Redis。
3. 网络链路：FRP 作为公网到内网穿透层。
4. 开发形态：Windows（HBuilderX/微信开发者工具）+ Linux 后端联调。

## 5. 数据与命名

1. 新命名统一 `hxy` 体系（见命名规范）。
2. 数据库建议命名：`hxy_dev` / `hxy_stg` / `hxy_prod`。
3. 表命名按领域前缀：`pay_*`、`trade_*`、`member_*`。

## 6. 可靠性

1. 幂等：下单、回调、退款、关单全链路幂等。
2. 状态机：单向流转，终态不可逆覆盖。
3. 补偿：查单、关单、退款查询、通知重试由定时任务收敛。
4. 发布：上线拦截规则 + 回滚预案双保险。
