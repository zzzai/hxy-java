# HXY 万店连锁 K8s 微服务治理蓝图 V1（2026-02-22）

## 1. 设计前提（独立判断）

1. 荷小悦未来万店规模的关键瓶颈不是单点性能，而是组织治理与资金一致性。
2. 架构必须支持“总部统一策略 + 区域弹性部署 + 门店轻终端接入”。
3. 微服务与 K8s 不是目的，目标是提升可靠性、发布效率、隔离故障域。

## 2. 目标架构（3层）

1. 总部控制平面（Control Plane）
   - 身份权限、配置中心、策略中心、审计中心、发布中心。
2. 交易与资金数据平面（Data Plane）
   - 交易、支付、退款、对账、结算、风控等核心服务。
3. 门店执行平面（Store Edge）
   - 门店履约、库存、排班、核销、售后证据上报。

## 3. 微服务边界建议

1. `hxy-gateway`：统一接入、鉴权、限流、灰度路由。
2. `hxy-trade-service`：订单与售后主流程。
3. `hxy-pay-service`：支付下单、回调、退款、通知。
4. `hxy-settlement-service`：总部结算、门店分账、账期管理。
5. `hxy-risk-service`：支付与退款风控规则引擎。
6. `hxy-store-ops-service`：门店履约与运营域。
7. `hxy-member-service`：会员身份与权益。
8. `hxy-product-service`：商品/SPU/SKU/价格库存。
9. `hxy-observability-service`：统一指标、日志、链路追踪视图。

## 4. K8s 治理基线

1. 命名空间分层：`hxy-prod-core`、`hxy-prod-edge`、`hxy-stg`、`hxy-dev`。
2. 工作负载治理：HPA/VPA、PDB、优雅下线、就绪探针、存活探针。
3. 安全治理：NetworkPolicy、PodSecurity、Secret 管理（外置 KMS/Vault）。
4. 发布治理：Argo CD/GitOps、金丝雀发布、自动回滚门禁。
5. 流量治理：Service Mesh（建议 Istio/Linkerd）做熔断、重试、超时、流量镜像。

## 5. 数据一致性策略

1. 交易与支付采用事件驱动（Outbox + MQ）保证最终一致性。
2. 幂等键贯穿：下单、回调、退款、通知、对账。
3. 状态机强约束：终态不可逆，禁止跨服务逆向覆盖。
4. 补偿任务服务化：查单、关单、退款查询、通知重放独立运行。

## 6. 从当前到目标的迁移原则

1. 先“模块化单体内拆分边界”，再“服务化部署”。
2. 先拆支付与交易边界，再拆门店与会员等非资金域。
3. 迁移过程必须保持支付主链路可用，不允许业务停摆式重构。
