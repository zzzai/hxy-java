# 支付系统微信支付 v3 迁移执行清单

更新时间：2026-02-17

## API verification log（2026-02-17）

- 触发范围：微信小程序支付（`wx.requestPayment` + 微信支付 v3 小程序下单/查单/回调）。
- 官方核验地址：
  - https://developers.weixin.qq.com/miniprogram/dev/api/payment/wx.requestPayment.html
  - https://pay.weixin.qq.com/doc/v3/merchant/4012791911
  - https://pay.weixin.qq.com/doc/v3/partner/4012076732
- 核验约束（落地到代码）：
  - 后端先下单获取 `prepay_id`，前端再发起 `wx.requestPayment`。
  - 前端成功回调不是最终支付结果，需结合“支付回调 + 主动查单”确认。
  - 小程序调起参数使用 `RSA` 签名（`timeStamp/nonceStr/package/signType/paySign`）。
  - 服务商模式下需区分 `sp_mchid/sub_mchid` 与 `sp_appid/sub_appid` 对应关系。
  - `prepay_id` 有效期有限（官方文档约 2 小时），超时需重新下单。

## 1. 目标与边界

- 目标：把小程序支付从当前 `V2 unifiedorder` 迁移到 `V3 JSAPI/小程序下单`。
- 原则：先并行、后切换；先可观测、后放量；全程可回滚。
- 边界：本清单只覆盖支付主链路（下单、回调、查单、退款），不改业务订单域模型。

## 2. 当前状态（基于仓库）

- 前端已使用 `wx.requestPayment`，无需重做。
- 后端下单仍走 V2：`https://api.mch.weixin.qq.com/pay/unifiedorder`。
- 现有服务商/子商户映射、回调验签、查单补偿、监控脚本可复用。

## 3. 迁移阶段（按顺序执行）

### Phase A：配置与上线拦截规则先行（1-2 天）

1. 新增 v3 配置项（见 `crmeb_java/crmeb/sql/payment_wechat_v3_template.sql`）。
2. 跑 v3 就绪检查：
   - `crmeb_java/crmeb/shell/payment_v3_readiness_check.sh`
3. 验收标准：
   - 关键参数齐全（`apiv3_key/serial_no/private_key_path`）。
   - `api_url` 为公网 HTTPS。
   - `pay_weixin_open=1`（预发可开，生产按窗口开启）。

### Phase B：后端双栈实现（3-5 天）

1. 增加支付网关抽象：
   - `wechat-v2-gateway`（现有）
   - `wechat-v3-gateway`（新增）
2. 增加版本开关：
   - `pay_routine_api_version = v2|v3`
3. 下单链路改造：
   - v3 普通商户：`/v3/pay/transactions/jsapi`
   - v3 服务商：`/v3/pay/partner/transactions/jsapi`
4. 回调链路：
   - 新增 v3 回调解密与签名验真（平台证书轮换机制）。
   - 与现有状态机/幂等逻辑复用。
5. 验收标准：
   - 同一订单域可在 v2/v3 两模式跑通。
   - 回调失败可自动重试+查单补偿。

### Phase C：联调与灰度（2-3 天）

1. 先预发跑 `0.01` 真单闭环（下单->支付->回调->查单->退款）。
2. 生产按门店灰度（建议 5%->20%->50%->100%）。
3. 每个阶段观察 24h：
   - 下单成功率
   - 回调成功率
   - 差错单/补偿单
4. 验收标准：
   - 关键指标不低于 v2 基线，且无新增 P1 资损风险。

### Phase D：收口与下线（1-2 天）

1. 关闭新流量 v2 下单入口（保留查询与历史兼容）。
2. 文档与脚本全部切到 v3 主路径。
3. 保留 v2 只读应急开关（限时 30 天）。

## 4. 回滚策略（必须具备）

- 回滚触发条件（任一满足）：
  - 支付成功率连续 30 分钟低于阈值。
  - 回调失败率高于阈值且补偿未收敛。
  - 出现资损或重复扣款风险。
- 回滚动作：
  1. 配置切回 `pay_routine_api_version=v2`。
  2. 保持订单状态机与回调入库逻辑不变。
  3. 继续跑查单补偿直到未决单清零。

## 5. 每日执行检查（迁移期）

1. `payment_v3_readiness_check.sh`
2. `payment_preflight_check.sh`
3. `payment_monitor_quickcheck.sh`
4. `payment_reconcile_daily.sh`
5. `payment_go_nogo_decision.sh`

## 6. 最小落地清单（你当前就能做）

1. 先落 v3 配置骨架（SQL 模板）。
2. 先跑 v3 就绪检查，补齐配置缺口。
3. 先在预发加版本开关，不改生产流量。
4. 先补自动化测试（签名、回调、幂等、退款）。

## 7. 风险提醒

- v3 核心风险不在“前端调起”，而在“服务端签名、证书与回调验真”。
- 迁移期间禁止同时改订单域核心事务，避免问题定位混杂。
- 明文密钥禁止进代码库；只允许配置中心/服务器密文注入。
