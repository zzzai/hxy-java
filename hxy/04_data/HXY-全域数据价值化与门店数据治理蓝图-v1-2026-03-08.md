# HXY 全域数据价值化与门店数据治理蓝图 v1（2026-03-08）

## 1. 目标与边界

### 1.1 目标
- 把用户数据、运营数据、经营数据、门店数据统一纳入可审计、可复用、可闭环的数据体系。
- 从第一天起保证“采集即治理”：字段有口径、事件有版本、状态有真值、异常可追溯。
- 数据必须直接服务经营动作，而不是只做报表展示。

### 1.2 非目标
- 不建设“大而全”数据中台一期。
- 不先做复杂建模后找业务场景。
- 不采集无法落到动作的数据。

## 2. 第一性原理与价值公式
- 数据价值 = 决策频次 x 决策提升幅度 x 执行到位率 x 反馈速度。
- 执行优先级：
  1) 稳定真值（状态机、一致性、幂等）
  2) 可追溯（日志键、审计链）
  3) 可决策（指标口径、责任归属）
  4) 可优化（AB/策略回传）

## 3. 全域数据资产分层

### 3.1 业务域
1. 用户域：注册、登录、授权、行为事件、标签、会员权益。
2. 交易域：订单、支付、售后、退款、回调、重放补偿。
3. 履约域：预约、技师、服务完成、核销、超时。
4. 门店域：门店主数据、排班、库存、调拨、盘点、损耗。
5. 营销域：活动、券、积分、触达、转化。
6. 财务域：提成计提/冲正、结算、四账对账差异。
7. 审计域：工单、异常、门禁、回滚、操作日志。

### 3.2 数仓层（必须）
1. ODS：原始事件和原始业务快照（不可改写）。
2. DWD：标准明细事实（按统一口径清洗）。
3. DWS：主题汇总（门店/技师/用户/活动/财务日汇总）。
4. ADS：经营数据集（增长、履约、财务、门店经营看板）。

## 4. 统一主键、状态机与真值优先级

### 4.1 主键体系（冻结）
- `tenantId/storeId/userId/staffId/orderId/orderItemId/afterSaleId/payOrderId/payRefundId/sourceBizNo/runId/traceId`。
- 所有跨域关联必须可回溯到 `orderId` 与 `sourceBizNo`。

### 4.2 状态机（冻结）
- 订单、支付、退款、售后、预约、券、积分、提成状态由后端状态机定义。
- 前端不允许通过文案推断状态；只能读取状态码与聚合结果码。

### 4.3 真值优先级（冻结）
1. 后端持久化事实
2. 聚合接口计算结果
3. 端侧缓存（仅展示）

## 5. 门店相关“最小必采”清单（Day-0）

| 数据主题 | 必采字段（最小集） | 触发时机 | 用途 |
|---|---|---|---|
| 门店交易 | `storeId/orderId/orderItemId/payStatus/refundStatus/payAmount/refundAmount` | 下单/支付/退款 | 收入与退款分析 |
| 门店履约 | `storeId/serviceOrderId/bookingId/staffId/startTime/endTime/status` | 开始/完成服务 | 产能、利用率、服务时长 |
| 门店库存 | `storeId/skuId/bizType/quantityBefore/quantityDelta/quantityAfter/bizNo` | 入出库/调拨/盘点 | 库存健康与损耗 |
| 门店营销 | `storeId/activityId/couponTemplateId/eventResult` | 曝光/领取/核销 | 活动ROI与券成本 |
| 门店质量 | `storeId/ticketId/sourceBizNo/severity/status/slaDeadline` | 告警/工单创建 | 风险收口与SLA |
| 门店财务 | `storeId/commissionId/originCommissionId/reversalAmount/settlementStatus` | 计提/冲正/结算 | 毛利与提成准确性 |

## 6. 事件采集协议（统一）

### 6.1 事件头（必填）
- `eventId`：全局唯一
- `eventName`：事件名（例如 `coupon_take_success`）
- `occurredAt`：事件发生时间（UTC）
- `schemaVersion`：契约版本
- `traceId`：链路追踪ID
- `sourceApp`：miniapp/admin/job/api

### 6.2 业务键（至少其一）
- `orderId/afterSaleId/payRefundId/bookingId/storeId/activityId`

### 6.3 治理键（必填）
- `tenantId`
- `purposeCode`
- `consentId`（涉及个人数据时）
- `fieldSetVersion`

### 6.4 推荐事件命名规范
- 页面：`page_view_{route}`
- 动作：`action_{domain}_{verb}`
- 结果：`result_{domain}_{success|fail|degraded}`

## 7. 标签治理落地（用户与门店标签）

### 7.1 风险分级
- G0：可自动化（推荐/提醒）。
- G1：高争议，默认关闭，审批后开通。
- G2：禁止标签，不得实现。

### 7.2 运行拦截（必须）
1. 查询 `eb_label_policy` 是否启用。
2. 校验授权有效与用途白名单。
3. 命中人工复核阈值必须人工确认。
4. 每次使用写审计：`requestId/useCase/decisionOutput/modelVersion`。

### 7.3 门店标签要求
- 门店标签必须区分：经营属性、服务能力、客群属性、风险属性。
- 禁止用敏感推断标签做差别定价、拒单、降权。

## 8. 指标体系与口径冻结

### 8.1 北极星与一级指标
1. 单店净收入 = 实收 - 退款 - 券成本 - 积分成本。
2. 履约效率 = 完单数 / 可服务时段。
3. 技师利用率 = 服务时长 / 出勤时长。
4. 退款率 = 退款订单数 / 支付订单数。
5. 工单SLA达成率 = SLA内关闭工单数 / 总工单数。
6. 复购率 = 周期内二次及以上下单用户 / 下单用户。

### 8.2 口径治理要求
- 每个指标必须有：公式、口径说明、负责人、刷新频率、下钻维度。
- 任何口径变更必须版本化并公告生效日期。

## 9. 数据质量治理（DQ）

### 9.1 必配规则
1. 主键唯一性：`eventId/orderId/runId` 不重复。
2. 关联完整性：`orderId` 在交易主表可命中。
3. 状态可达性：状态转移符合状态机。
4. 金额平衡：支付、退款、提成冲正可对齐。
5. 时序合法：`createdAt <= paidAt <= refundedAt`。
6. 幂等一致：相同幂等键不重复落账。

### 9.2 质量分级
- BLOCK：影响财务一致性、主链路正确性。
- WARN：影响分析准确性，但不阻断主链路。

### 9.3 处置机制
- DQ失败自动生成工单，分配 owner 与 SLA。
- 连续3次失败触发发布门禁阻断。

## 10. 观测与审计

### 10.1 统一检索键（冻结）
- `runId/orderId/payRefundId/sourceBizNo/errorCode`

### 10.2 关键链路
1. 退款回调
2. 退款重放
3. 四账联动
4. 工单同步
5. 提成计提与冲正

### 10.3 审计要求
- 关键路径必须输出结构化日志（JSON可检索）。
- 字段缺失写 `finance-log-validate` 告警，不改变主链路结果。

## 11. 组织机制（RACI）
- Product Owner：指标定义与业务解释。
- Data Owner：口径冻结与变更审批。
- Domain Owner（booking/trade/pay/member/promotion）：事件与事实落库质量。
- Ops Owner：DQ告警、工单SLA、回滚执行。
- Compliance Owner：标签与隐私合规审查。

## 12. 90天执行路线图

### Day 0-30（打底）
1. 冻结 ID/状态机/指标字典。
2. 完成门店最小必采事件接入（交易、履约、库存、营销、财务）。
3. 建立 ODS+DWD 最小链路与每日 DQ 报告。

### Day 31-60（闭环）
1. 建 DWS/ADS 门店经营主题集。
2. 打通“告警 -> 工单 -> 修复 -> 回归”流程。
3. 上线标签策略拦截与审计报表。

### Day 61-90（增益）
1. 上线增长/经营策略实验（券策略、预约填充、退款预警）。
2. 评估策略增益并形成季度复盘。
3. 固化发布门禁：契约门禁 + DQ门禁 + 合规门禁。

## 13. 发布门禁建议（新增）
1. `check_miniapp_p0_contract_gate.sh`：继续保留契约锚点。
2. 新增 `check_miniapp_event_contract_gate.sh`：检查事件名与字段必填项。
3. 新增 `check_miniapp_dq_gate.sh`：检查核心DQ规则结果。
4. 新增 `check_label_policy_gate.sh`：检查 G1/G2 标签策略与审计覆盖。

## 14. 验收标准
1. 关键指标口径文档完备且有负责人。
2. 门店最小必采数据覆盖率 >= 95%。
3. 关键链路日志检索成功率 >= 99%。
4. G1标签越权调用次数 = 0。
5. DQ BLOCK 问题发布前清零。
