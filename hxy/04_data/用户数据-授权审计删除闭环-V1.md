# 用户数据-授权审计删除闭环 V1

> 状态：执行版  
> 日期：2026-02-16  
> 目标：让“采集-使用-删除”全链路可审计、可追溯、可撤回

## 1. 适用边界

1. 本方案覆盖健康相关数据、敏感图片、高争议标签的授权与使用闭环。
2. 本方案优先于任何“增长目标”或“运营便利”。
3. 无授权、无审计、无删除能力的数据功能不得上线。

## 2. 三条闭环

## 2.1 授权闭环（Consent Loop）

### 2.1.1 场景编码

| 场景编码 | 说明 | 数据范围 | 默认状态 |
|---|---|---|---|
| HEALTH_BASIC | 健康问询基础字段 | 主诉、评分、禁忌项、偏好 | 必配 |
| IOT_VITALS | 体测数据采集 | 身高体重、BMI、血压、心率 | 关闭 |
| PHOTO_TONGUE | 舌照采集与分析 | 舌照原图、特征、AI结果 | 关闭 |
| PHOTO_FACE | 面照采集与分析 | 面照原图、特征、AI结果 | 关闭 |
| AI_PERSONALIZATION | AI个性化推荐 | 标签、评分、推荐理由 | 关闭 |
| THIRD_PARTY_SHARE | 第三方共享 | 对外提供行为/健康数据 | 冻结 |

### 2.1.2 状态机

`PENDING -> GRANTED -> WITHDRAWN/EXPIRED`  
`PENDING -> DENIED`

约束：
1. 每次授权必须记录 `policy_version` 与 `consent_text_hash`。
2. 每次采集必须附带 `consent_id`，否则写入失败。
3. 撤回授权后，后续采集立即停止；历史数据按保留策略处理。

### 2.1.3 最小接口（后端）

1. `POST /api/privacy/consent/grant`
2. `POST /api/privacy/consent/withdraw`
3. `GET /api/privacy/consent/list`
4. `GET /api/privacy/consent/detail/{id}`

## 2.2 审计闭环（Access Audit Loop）

### 2.2.1 数据访问分级

| 数据级别 | 示例 | 访问策略 |
|---|---|---|
| L1 | 预约单号、满意度 | 角色权限控制 |
| L2 | 问题部位、生活习惯 | 角色权限 + 访问理由 |
| L3 | 血压心率、体测详情 | 工单审批 + 理由必填 + 限时 |
| L4 | 舌照/面照原图 | 双人审批 + 最短时效 + 全量审计 |

### 2.2.2 工单状态机

`SUBMITTED -> AUTO_PASS/APPROVED/REJECTED -> EXECUTED -> CLOSED`

约束：
1. 访问L3/L4必须先拿工单 `ticket_no`。
2. 工单超时自动失效，已失效不得继续访问。
3. 审计日志必须记录“谁、何时、为何、看了什么、结果如何”。

### 2.2.3 最小接口（后端）

1. `POST /api/privacy/access-ticket/create`
2. `POST /api/privacy/access-ticket/approve`
3. `POST /api/privacy/access-ticket/reject`
4. `POST /api/privacy/access-ticket/close`
5. `GET /api/privacy/access-log/query`

## 2.3 删除闭环（Deletion Loop）

### 2.3.1 删除范围

| scope_code | 说明 | 是否可立即删 |
|---|---|---|
| PROFILE_BASIC | 非交易型档案数据 | 可进入冷静期 |
| HEALTH_DATA | 健康问卷、体测、图片 | 可进入冷静期 |
| TAG_DATA | 标签与画像结果 | 可进入冷静期 |
| ACCOUNT_CLOSE | 账号注销关联数据 | 需联动法定保留策略 |

### 2.3.2 状态机

`SUBMITTED -> COOLING -> EXECUTING -> COMPLETED/FAILED`  
`SUBMITTED -> REJECTED`（法定义务不可删）

约束：
1. 默认 7 天冷静期，用户可撤销。
2. 涉及法定留存（如交易凭证）进入“逻辑隔离”，不可对外展示，但暂不物理删除。
3. 删除完成后回写“删除报告”，包含数据类别、条数、时间、执行人。

### 2.3.3 最小接口（后端）

1. `POST /api/privacy/deletion/request`
2. `POST /api/privacy/deletion/cancel`
3. `GET /api/privacy/deletion/status/{ticketNo}`
4. `POST /api/privacy/deletion/execute`（系统任务）

## 3. 角色职责（RACI）

| 事项 | 产品 | 开发 | 运维 | 法务/合规 | 门店 |
|---|---|---|---|---|---|
| 授权文案版本管理 | R | C | I | A | I |
| 授权接口与校验 | C | A | C | I | I |
| 访问工单与审计 | C | A | R | C | I |
| 删除流程与任务执行 | C | A | R | C | I |
| 高风险场景审批 | I | C | I | A | R |

注：R=执行，A=最终负责，C=协作，I=知会。

## 4. SLO与告警

1. 授权写入成功率 >= 99.9%。
2. 越权访问拦截率 = 100%。
3. 删除请求受理时延 <= 5分钟。
4. 删除执行时效（不含冷静期）<= 72小时。
5. 任何L4数据访问失败或越权，5分钟内触发告警机器人。

## 5. 两周实施拆解

### 第1周

1. 上线授权记录表、访问工单表、删除工单表。
2. 小程序增加授权管理入口与撤回入口。
3. 服务端在采集链路接入 `consent_id` 强校验。

### 第2周

1. 接入访问审批流（L3/L4）。
2. 接入删除任务与删除回执。
3. 上线审计看板（授权率、越权拦截、删除时效）。

## 6. 和当前系统的落地映射

1. 复用现有 `eb_sensitive_access_log`/`eb_sensitive_access_permission` 作为审计基础。
2. 新增工单与授权表，形成“授权前置 + 工单放行 + 删除回执”的完整闭环。
3. 高争议标签由 `eb_label_policy` 统一上线拦截规则，禁止绕过策略直接使用。
