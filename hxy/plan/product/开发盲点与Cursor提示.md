# 荷小悦小程序 · 开发盲点与 Cursor 提示

> **开工前必读**。以下三点在 PRD/数据库/接口/任务清单中未展开，但开发时一旦忽略会直接导致线上事故。请将本文档纳入 Context 或口头告知 Cursor。

---

## 盲点一：并发地狱（The "Double Booking" Trap）

### 问题

数据库设计了 `eb_store_technician_schedule`（技师排班表），但足疗预约最怕 **「两个用户在 20:00 同时点了同一个技师」**。CRMEB 原生的秒杀有并发处理，**新写的预约逻辑如果没有锁，上线第一天就会双抢。**

### 缺失逻辑

- **乐观锁（Optimistic Locking）** 或 **Redis 原子锁** 锁定时间槽（Time Slot）。

### Cursor 提示补丁

在写 `createBooking`（创建预约订单）接口时，必须要求 Cursor：

> 在锁定技师时间槽（Time Slot）时，使用 **原子更新**：  
> `Db::name('store_technician_schedule')->where('id', $scheduleId)->where('is_available', 1)->update(['is_available' => 0, 'order_id' => $orderId])`  
> 并判断 **影响行数**。如果影响行数为 0，说明该时间槽刚才已被别人抢占，应抛出业务异常，提示：「手慢了，该时段已被预约」。

## 盲点二：被遗忘的「B 端」（The Technician View）

### 问题

PRD 详细规划了用户端（C 端），但 **技师怎么看自己的排班？技师怎么确认「我上钟了」？** 若没有技师侧能力，用户预约后技师不知情，到店后就是灾难。

### 缺失逻辑

- 技师端（或管理员端）的 **核销与状态流转**。

### 最小可行性方案（MVP）

不要开发独立技师 App（过重）。建议：

1. 技师关注公众号 / 绑定小程序。
2. **新预约产生时**，用 **微信模板消息（订阅消息）** 推送到技师手机。
3. 技师在 **小程序「我的 - 店员中心」** 查看今日预约，并点击「开始服务」「结束服务」。

### Cursor 提示补丁

在开发预约模块时，需复用 CRMEB 的 **订阅模板/模板消息能力**，增加一个「新预约提醒」的模板消息，发送对象为 `eb_store_technician` 表中关联的技师（若技师绑定 `uid` 则按 `uid` 推送；否则按门店维度由管理员/店员接收）。

## 盲点三：CRMEB 的订单状态机陷阱

### 问题

任务清单里「订单表改造」预估 1 天，**风险很大**。  
CRMEB 订单流程是：`未支付 → 已支付 → 待发货 → 待收货 → 已完成`。  
足疗流程是：`未支付 → 已支付 → 待服务(锁库存) → 服务中 → 已完成`。  
若直接改核心订单状态含义，会破坏 CRMEB 的 **后台统计、分销结算、退款逻辑**（它们都依赖 `status`）。

### 建议策略：「欺骗」系统

- **预约成功（已支付）**：在 CRMEB 里标记为 **「待发货」**（分销佣金暂不结算，安全）。
- **核销 / 开始服务**：在 CRMEB 里标记为 **「已发货」**（虚拟发货）。
- **服务结束**：在 CRMEB 里标记为 **「已收货 / 已完成」**（触发分销佣金结算）。

### Cursor 提示补丁

在编写订单状态流转时，请建立 **Status Mapping（状态映射）**：

- 我们的「待服务」 ↔ CRMEB 的 `STATUS_UNSHIPPED`（待发货）
- 我们的「服务完成」 ↔ CRMEB 的 `STATUS_RECEIVED`（已收货）

**不要新增顶层状态码**，而是使用 **新增字段 `service_status`**（或设计文档中的等价字段）来标记服务业特有的细分状态（待服务 / 服务中 / 已完成等）。
