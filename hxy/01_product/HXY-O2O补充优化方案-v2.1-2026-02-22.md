# 二、卡项体系设计（足疗店金融灵魂）

## 2.1 卡项核心逻辑

【核心理念】\
卡项 = 预收款工具 + 客户锁定机制 + 现金流保障\
\
足疗店80%的稳定营收来自卡项会员的持续复购，\
而非单次散客的偶然消费。\
\
【三大卡项类型对比】\
\
类型 本质 用户购买 核销方式 适用场景\
─────────────────────────────────────────────────────\
次卡 购买次数 10次卡¥980 每次核销1次 高频用户锁定\
(单次¥98)\
\
期卡 购买时长 月卡¥599 无限次使用 超高频用户\
无限次 (1个月内)\
\
储值卡 购买余额 充¥1000 按实际消费 不确定频次\
送¥200 金额扣减 但想享优惠

## 2.3 核心业务流程

【流程1：用户购卡】\
\
用户选择卡项\
↓\
选择：10次卡 ¥980（原价¥1280，立省¥300）\
↓\
下单支付\
↓\
系统生成持卡记录：\
├─ card_no: HXY20250214001\
├─ total_times: 10\
├─ remaining_times: 10\
├─ expire_time: 当前时间 + 365天\
└─ status: 1（正常）\
↓\
用户收到电子卡券（小程序卡包）\
\
【流程2：次卡核销】\
\
用户到店消费 60分钟足疗\
↓\
前台/技师端扫码 or 输入卡号\
↓\
系统检查：\
├─ 卡状态是否正常？\
├─ 是否在有效期内？\
├─ 剩余次数是否充足？\
└─ 本次服务是否在适用范围内？\
↓ 全部通过\
执行核销：\
├─ used_times + 1\
├─ remaining_times - 1\
├─ usage_records 追加本次记录：\
│ {\
│ \"time\": \"2025-02-14 14:00\",\
│ \"service\": \"60分钟足疗\",\
│ \"technician\": \"技师小王\",\
│ \"order_id\": 10086\
│ }\
└─ 若 remaining_times = 0，status = 2（已用完）\
↓\
核销成功，开始服务\
\
【流程3：储值卡扣减】\
\
用户到店消费 60分钟足疗 + 精油 实付¥158\
↓\
选择支付方式：储值卡\
↓\
系统检查储值卡余额：¥520\
↓\
充足，执行扣减：\
├─ used_amount: ¥358 → ¥516\
├─ remaining_amount: ¥520 → ¥362\
└─ usage_records 追加本次记录\
↓\
扣减成功，支付完成\
\
【流程4：期卡无限次使用】\
\
用户到店消费（本月第5次）\
↓\
扫描月卡\
↓\
系统检查：\
├─ 是否在有效期内？（购买后30天内）\
└─ 卡状态是否正常？\
↓ 通过\
直接核销，不扣次数\
↓\
usage_records 记录使用时间和项目\
↓\
核销成功（月卡可无限次使用）

# 三、加钟/升级场景设计（动态订单变更）

## 3.1 业务场景

线下服务过程中，常见的动态变更场景：

-   【场景1：加钟】客人做到一半觉得舒服，要求延长30分钟

-   【场景2：升级】客人觉得基础油不够，要求升级精油

-   【场景3：加项】服务中途推销，客人同意增加肩颈按摩

-   【场景4：换人】客人对当前技师不满意，要求更换技师

## 3.2 核心挑战

❌ 原订单已支付，如何追加金额？\
❌ 技师后续排期如何自动顺延？\
❌ 房间占用时间如何自动延长？\
❌ 加钟订单如何与主订单关联？\
❌ 技师提成如何计算（基础服务 + 加钟）？

## 3.3 数据表设计

**订单主表（eb_order）补充字段：**

  ----------------------------------------------------------------------------
  **新增字段**                        **说明**
  ----------------------------------- ----------------------------------------
  parent_order_id                     bigint(20)，关联主订单（加钟订单专用）

  is_addon                            tinyint(1)，是否加钟订单：0=否 1=是

  addon_type                          tinyint(1)，加钟类型：1=延长时长

  original_end_time                   int(11)，原计划结束时间

  actual_end_time                     int(11)，实际结束时间（加钟后）
  ----------------------------------------------------------------------------
