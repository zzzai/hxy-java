# 项目级产品设计总纲（RuoYi版）

- 日期：2026-02-21
- 基座：`/root/crmeb-java/ruoyi-vue-pro-master`
- 适用范围：项目全局（支付仅为其中一个核心子域）

## 1. 项目目标

1. 建立“总部统一治理 + 门店标准履约 + 用户端稳定体验”的连锁业务系统。
2. 以 RuoYi 模块化能力为底座，优先打通交易与支付主链路，再扩展非支付模块。
3. 形成“可上线、可回滚、可审计、可持续迭代”的工程体系。

## 2. 用户与角色

1. C 端用户：浏览、下单、支付、售后、退款进度查询。
2. 门店运营：商品与服务管理、履约、售后证据上传。
3. 总部运营：活动与策略配置、门店治理、数据复盘。
4. 总部客服/财务/风控：退款审批、对账差异处理、风险管控。
5. 系统管理员：配置管理、上线发布、监控与故障响应。

## 3. 业务域边界（项目级）

1. 交易域：订单创建、履约、售后申请与状态流转。
2. 支付域：支付下单、回调、退款、对账、审计。
3. 商品域：SPU/SKU、价格、库存、上下架。
4. 会员域：账号、身份、权益、积分（按阶段逐步收敛）。
5. 营销域：优惠券/活动（仅保留与交易闭环强相关能力）。
6. 运维治理域：监控、日志、上线拦截规则、发布回滚。

## 4. 版本范围（轻量）

### 4.1 P0（必须先完成）

1. 下单-支付-回调-退款-对账主链路可稳定运行。
2. 门店无资金操作权限，总部统一支付治理。
3. 异常补偿可自动收敛（回调丢失、超时、重试、乱序）。
4. 上线拦截规则生效（无验收/无回滚/监控失效即阻断）。

### 4.2 P1（在 P0 后推进）

1. 监控指标标准化与告警降噪。
2. 非支付模块（商品、会员、营销）按业务优先级二开。
3. 统一数据口径与经营报表能力增强。

## 5. 核心业务流程（项目视角）

1. 用户下单：商品校验 -> 交易单创建 -> 支付单创建。
2. 用户支付：前端拉起支付 -> 渠道回调 -> 业务状态更新。
3. 门店履约：接单 -> 交付 -> 完结/售后。
4. 用户退款：申请 -> 规则分流（自动/人工）-> 终态收敛。
5. 资金闭环：交易账、支付账、对账差异、工单处理。

## 6. 关键产品规则（项目级）

1. 资金规则：支付与退款执行只允许总部入口。
2. 权限规则：门店仅处理履约与售后材料，不执行退款打款。
3. 状态规则：交易与支付状态机单向流转，终态不可逆覆盖。
4. 合规规则：敏感配置不入库明文，不进代码仓库明文。

## 7. 项目级指标（KPI）

1. 支付成功率与回调收敛时延（P95）。
2. 退款终态收敛率与平均收敛时长。
3. 订单履约及时率与售后闭环时长。
4. 上线质量：拦截命中率、发布回滚率、P0 故障次数。
