## 🎯 核心设计原则（7大原则）

### 1. 分层管控
- **总部**: 仅维护标准化基础信息（SPU/SKU名称、规格、指导价、时长等）
- **门店**: 仅操作运营信息（价格、库存、上下架、排序、闲时优惠等）
- **权限**: 互不越权，总部保留超管兜底权限

### 2. 分离计算
- **服务类**: 聚焦「技师时间排班」
- **实物类**: 聚焦「实际库存扣减」
- **套餐类**: 通过独立表拆解为子项，不污染核心SPU/SKU结构
- **技术**: 结合策略模式实现库存扣减逻辑解耦

### 3. 数据隔离
- **四层隔离**: 字段 + 索引 + 权限 + 查询
- **门店权限**: 仅能查询/修改自身数据，无法操作其他门店及总部数据

### 4. 自动兜底
- **价格兜底**: 门店未配置价格时，自动回滚至总部默认值
- **性能优化**: 价格采用「写入时冗余」方案，避免N+1查询
- **用户体验**: 前端补充友好提示

### 5. 兼容复用
- **实物商品**: 直接复用原有电商系统表结构
- **服务/套餐**: 单独设计专用表，减少开发冗余
- **扩展性**: 订单子项表衔接套餐核销场景，虚拟农场积分体系联动SKU兑换

### 6. 技术落地适配
- **策略模式**: 库存扣减逻辑解耦
- **版本控制**: 支持总部规格变更的版本管控
- **原子化核销**: 套餐拆解为子项，独立核销
- **字段完整**: 新增必要字段（价格标识、版本、生效时间等）

### 7. 场景化适配
- **服务定制**: 时长/精油/力度选择
- **技师排班**: 时间槽定价、闲时优惠
- **虚拟农场**: 积分兑换SKU
- **UI联动**: 确保数据结构能支撑前端所有交互需求

## 📊 核心概念定义

| 概念 | 定义 | 归属方 | 荷小悦业务场景示例 | 核心作用 |
|------|------|--------|-------------------|---------|
| **SPU** | 标准化产品/服务单元，品牌统一的"基础品项标识"，无具体售卖规格 | 总部（唯一） | 中式足疗、薰衣草精油、养生套餐A | 总部统一品控、品牌展示 |
| **SKU** | 库存量/服务量单位，门店实际售卖的具体规格版本 | 总部定规格，门店定运营 | 中式足疗[60分钟][重力]、薰衣草精油[10ml] | 门店实际交易、库存/时间计算 |
| **总部主数据** | 全门店通用的标准化基础数据，含版本信息、生效时间 | 总部（唯一维护） | SPU名称、SKU规格、基准时长、官方指导价 | 保障全品牌基础信息一致性 |
| **门店运营数据** | 门店基于总部主数据生成的个性化运营数据 | 各门店（独立维护） | 门店售价、会员价、库存、闲时优惠价 | 实现千店千价、个性化运营 |
| **套餐** | 由多个服务/实物SPU/SKU组合而成的复合品项 | 总部定组合，门店定价格 | 养生套餐A=60分钟足疗+养生茶+艾草贴 | 满足组合售卖，提升客单价 |
| **闲时优惠** | 门店针对非高峰时段设置的SKU优惠价 | 门店（独立维护） | 中式足疗[60分钟]14:00-16:00 ¥98 | 提升非高峰时段预约率 |
| **虚拟积分（露珠）** | 虚拟农场场景下的用户资产，可兑换指定SKU | 总部定规则，门店定兑换SKU | 100露珠兑换基础油加项 | 提升用户粘性、日活 |

## 🏗️ 整体架构分层

### 三层架构 + 单向数据流

```
【总部主数据层（全局唯一）】
├─ SPU主表（eb_service）：服务/套餐的标准化基础信息（含版本、生效时间）
├─ SKU规格表（eb_service_sku）：服务/实物的具体规格信息（含版本、生效时间）
├─ 套餐关联表（eb_service_combo_relation）：套餐子项绑定规则（全局）
├─ 总部实物SPU/SKU表（原有电商表）：实物商品的总部基础信息
├─ 积分兑换规则表（eb_points_exchange）：积分兑换基础规则（全局）
└─ 服务物料BOM表（eb_service_bom）：服务消耗的实物物料清单 ← 社区小店版新增

        ↓（单向同步：总部新增/修改→自动同步至所有门店，支持版本生效时间）

【门店运营数据层（各门店独立）】
├─ 门店SPU引用表（eb_store_service）：门店选择上架的SPU，记录上下架/排序
├─ 门店SKU定价表（eb_store_service_sku）：门店对SKU的定价/库存/会员价/闲时优惠
├─ 原有电商实物门店表（eb_store_product*）：复用原有结构，存储实物商品运营数据
├─ 门店积分兑换表（eb_store_points_exchange）：门店选择的可兑换SKU及所需积分
├─ 技师排班表（eb_technician_schedule）：服务类SKU的技师时间槽
├─ 技师技能矩阵表（eb_technician_skill）：技师可承接的服务SPU ← 社区小店版新增
└─ 定价策略表（eb_pricing_strategy）：价格计算流水线、营销互斥规则 ← 社区小店版新增

【技术落地支撑层（全局/门店共用）】
├─ 订单主表（eb_order）：存储订单整体信息，关联门店、用户、套餐/单品SKU
├─ 订单子项明细表（eb_order_item）：套餐拆解子项、单品明细，关联核销状态
│   └─ 新增字段：real_split_price（价格分摊）、refunded_amount（退款金额） ← 社区小店版新增
├─ 操作日志表（eb_operation_log）：记录门店/总部的所有修改操作，用于溯源
├─ 库存流水表（eb_stock_flow）：实物类SKU的库存变动记录，用于对账
│   └─ 新增字段：flow_type（区分销售/服务消耗出库） ← 社区小店版新增
└─ 用户积分表（eb_user_points）：用户虚拟农场露珠（积分）记录
```

## 🔄 数据流向规则（含版本控制）

### 1. 总部新增 SPU/SKU
- 系统自动为所有门店生成默认的门店SPU/SKU记录
- 默认状态：下架、价格为总部指导价、库存为默认值、is_custom_price=0
- 关联初始版本（version=1）

### 2. 总部修改 SPU/SKU 基础信息
- 生成新的版本（version+1）
- 设置生效时间（默认次日0点）
- 所有门店的关联基础信息在生效时间后同步更新
- **历史订单/排班关联旧版本**（关键！）

### 3. 总部停用 SPU/SKU
- 系统强制将所有门店的对应SPU/SKU设为下架
- 门店无法手动恢复
- 标记版本为"停用"，禁止新订单关联

### 4. 门店操作权限
- 仅能对自身的门店SPU/SKU记录做：上下架、改价、改库存、调排序、设置闲时优惠
- 无法修改关联的总部ID、版本信息

### 5. 门店设置闲时优惠
- 关联技师排班表的时间槽
- 标记闲时时段（如14:00-16:00）
- 前端技师排班页自动展示优惠价格

### 6. 门店配置积分兑换
- 从总部积分兑换规则中选择SKU
- 设置所需露珠数量
- 前端虚拟农场展示兑换提示

### 7. 操作日志与溯源
- 所有数据同步/修改操作自动记录操作日志表
- 总部可追溯所有门店的操作
- 实物库存变动同时记录库存流水表

## 💰 社区小店版补全：四大核心盲区

### 盲区1: 财务维度 - 价格分摊与逆向退款

#### 核心痛点
套餐打折后，子项部分核销的退款金额无明确依据，导致小店财务对账混乱。

#### 解决方案
在 `eb_order_item` 表新增3个关键字段：

| 字段名 | 类型 | 说明 | 技术适配点 |
|--------|------|------|-----------|
| `real_split_price` | decimal(10,2) | 子项实付分摊金额（核心） | 退款金额的核心依据，财务对账主键 |
| `split_weight` | decimal(10,4) | 子项价格权重 | 分摊金额的计算依据 |
| `refunded_amount` | decimal(10,2) | 该子项已退款金额 | 记录子项退款明细，避免重复退款 |

#### 分摊算法（极简权重算法）
```
步骤1：计算套餐所有子项的原价总和 = ∑(子项SKU的门店售价/指导价)
步骤2：单个子项的权重 split_weight = 该子项原价 / 套餐子项原价总和
步骤3：单个子项的实付分摊金额 real_split_price = 套餐实际支付金额 × 该子项权重
步骤4：尾差处理：将尾差计入最后一个子项的real_split_price
```

#### 退款逻辑
- 仅退未核销子项的 `real_split_price`
- 已核销子项不退款
- 退款总金额 = 所有未核销子项的 `real_split_price` 总和 - 已退款金额

---

### 盲区2: 履约维度 - 服务物料BOM消耗

#### 核心痛点
服务项目（如精油开背）伴随内部实物物料消耗（如10ml精油），当前设计无自动扣减逻辑，导致小店实物库存账实不符。

#### 解决方案
新增 `eb_service_bom` 表（服务物料清单表）：

| 字段名 | 类型 | 说明 | 技术适配点 |
|--------|------|------|-----------|
| `service_id` | int(11) | 关联总部服务SPU ID | 仅服务项目有物料消耗 |
| `product_sku_id` | int(11) | 关联实物SKU ID | 消耗的实物SKU |
| `quantity` | decimal(6,2) | 单位服务消耗的实物数量 | 库存自动扣减的数量依据 |
| `is_mandatory` | tinyint(1) | 是否为必耗物料 | 区分核心消耗和可选消耗 |

#### 履约逻辑
```
步骤1：技师标记「服务完成」（用户核销服务，verify_status=1）
步骤2：系统自动查询该服务SPU对应的 eb_service_bom 表，获取必耗物料清单
步骤3：调用实物库存扣减策略，按 quantity 扣减对应门店的实物SKU库存
步骤4：在库存流水表 eb_stock_flow 中记录，出库类型标记为「服务消耗」
步骤5：可选物料由技师手动勾选后，系统执行上述扣减逻辑
```

#### 库存流水表适配
在 `eb_stock_flow` 表新增字段：
- `flow_type`: 1=销售出库 2=服务消耗出库 3=采购入库 4=退货入库

---

### 盲区3: 营销维度 - 价格优先级与多重营销冲突处理

#### 核心痛点
闲时优惠、会员价、优惠券叠加时，价格计算无明确规则，导致用户实付价格争议。

#### 解决方案
新增 `eb_pricing_strategy` 表（定价策略表）：

| 字段名 | 类型 | 说明 | 默认配置 |
|--------|------|------|---------|
| `level` | tinyint(1) | 策略级别：1=总部全局 2=门店自定义 | 1 |
| `price_calc_pipeline` | varchar(128) | 价格计算流水线 | "base->time->member->coupon" |
| `is_time_member_overlay` | tinyint(1) | 闲时价是否与会员价叠加 | 0（不叠加） |
| `coupon_deduct_base` | tinyint(1) | 优惠券抵扣基数 | 2（按最终价抵扣） |

#### 价格计算流水线
```
base（门店SKU基础价）
  ↓
time（闲时优惠价，如有则覆盖基础价，不叠加）
  ↓
member（会员价，基于上一步价格计算折扣）
  ↓
coupon（优惠券，基于上一步最终价抵扣）
```

#### 门店SKU定价表适配
在 `eb_store_service_sku` 表新增字段：
- `is_time_discount`: 是否参与闲时优惠
- `time_discount_price`: 闲时优惠价
- `time_discount_period`: 闲时优惠时段（如14:00-16:00,22:00-24:00）

---

### 盲区4: 技师维度 - 技师技能矩阵

#### 核心痛点
技师技能有边界（如初级技师仅能做基础足疗，高级技师可做正骨），当前设计无技能限制，可能导致错派订单。

#### 解决方案
新增 `eb_technician_skill` 表（技师技能关联表）：

| 字段名 | 类型 | 说明 | 技术适配点 |
|--------|------|------|-----------|
| `technician_id` | int(11) | 关联技师ID | 小店技师ID |
| `service_id` | int(11) | 关联总部服务SPU ID | 仅服务项目有技师技能绑定 |
| `is_main_skill` | tinyint(1) | 是否为技师主打技能 | 前端技师列表标记「主打技能」 |
| `sort` | int(11) | 技能排序 | 前端技师技能展示排序 |

#### 预约逻辑
```
步骤1：用户选择服务SPU（如正骨），进入「选择技师」页面
步骤2：系统自动查询 eb_technician_skill 表，获取可承接该服务SPU的技师列表
步骤3：系统再关联 eb_technician_schedule 表，过滤出该时段空闲的技师
步骤4：前端仅展示「可承接+空闲」的技师，按 sort 和 is_main_skill 排序
步骤5：用户选定技师后，系统锁定该技师的时间槽，完成预约
```

## 💡 关键技术要点

### 1. 版本控制机制
- 总部修改SPU/SKU基础信息时，version+1
- 设置生效时间（effective_time）
- 历史订单/排班关联旧版本，避免影响已有业务

### 2. 价格冗余策略
- 门店SKU定价表采用「写入时冗余」方案
- 避免N+1查询，提升性能
- 价格修改时更新冗余字段
