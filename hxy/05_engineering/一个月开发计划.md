## Week 1：安全修复 + 预约系统完善

### 1.1 安全修复（P0）

**目标：** 消除所有硬编码密钥和安全配置缺陷

| 任务 | 文件 | 说明 |
|------|------|------|
| API 密钥迁移 | `application.yaml` | 所有 AI/支付/地图密钥改为 `${ENV_VAR}` 占位 |
| SMS 验证码 | `application.yaml` | begin-code/end-code 改为 `0000`/`9999` 随机范围 |
| SMS 固定值 | `application-local.yaml` | 仅 local profile 保留 `9999` 固定值 |
| BCrypt 强度 | `SecurityProperties.java` | `passwordEncoderLength` 默认值从 4 改为 10 |
| XSS 过滤器 | `application.yaml` | `yudao.xss.enable` 改为 `true` |
| Druid 认证 | `application-local.yaml` | 配置 `login-username`/`login-password` |
| mock-enable | `SecurityProperties.java` | 确认默认 `false`，仅 local 覆盖为 `true` |
| MyBatis 加密 | `application.yaml` | 密钥迁移到环境变量 |

## Week 2：派单模式 + 加钟体系

### 2.1 派单模式

**BookingOrderDO 新增字段：**
- `dispatchMode` TINYINT — 1=点钟（用户指定技师），2=排钟（系统分配）

**新增服务：**
- `TechnicianDispatchService` — 排钟算法接口
- `TechnicianDispatchServiceImpl` — 按当日已接单数升序 + 评分降序选择最优技师

**修改：**
- `BookingOrderServiceImpl.createOrder` — 支持 dispatchMode 参数，排钟模式自动分配技师
- `AppBookingOrderController` — 新增 dispatchMode 入参

### 2.2 加钟/升级体系

**BookingOrderDO 新增字段：**
- `parentOrderId` BIGINT — 父订单 ID（加钟/升级时关联原订单）
- `isAddon` TINYINT — 0=正常订单，1=加钟子订单
- `addonType` TINYINT — 1=加钟，2=升级，3=加项目

**新增服务：**
- `BookingAddonService` — 加钟/升级业务接口
- `BookingAddonServiceImpl` — 实现逻辑：
  - 加钟：延长当前技师时间段，锁定下一个可用 slot
  - 升级：差价计算，更新 SKU 和价格
  - 加项目：基于 hxy_service_addon 表创建附加订单

**新增 Controller：**
- `AppBookingAddonController` — 用户端加钟/升级 API

### 2.3 SQL 迁移

```sql
ALTER TABLE booking_order ADD COLUMN dispatch_mode TINYINT DEFAULT 1 COMMENT '派单模式 1=点钟 2=排钟';
ALTER TABLE booking_order ADD COLUMN parent_order_id BIGINT DEFAULT NULL COMMENT '父订单ID';
ALTER TABLE booking_order ADD COLUMN is_addon TINYINT DEFAULT 0 COMMENT '是否加钟子订单';
ALTER TABLE booking_order ADD COLUMN addon_type TINYINT DEFAULT NULL COMMENT '加钟类型 1=加钟 2=升级 3=加项目';
CREATE INDEX idx_booking_order_parent ON booking_order(parent_order_id);
```

## Week 3：分润体系 + 支付对接

### 3.1 技师佣金体系

**新增表：**

`technician_commission`:
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| technician_id | BIGINT | 技师 ID |
| order_id | BIGINT | 预约订单 ID |
| user_id | BIGINT | 用户 ID |
| commission_type | TINYINT | 1=基础/2=点钟/3=加钟/4=卡项销售/5=商品/6=好评 |
| base_amount | INT | 订单金额（分） |
| commission_rate | DECIMAL(5,4) | 佣金比例 |
| commission_amount | INT | 佣金金额（分） |
| status | TINYINT | 0=待结算/1=已结算/2=已取消 |
| settlement_time | DATETIME | 结算时间 |

`technician_commission_config`:
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| store_id | BIGINT | 门店 ID |
| commission_type | TINYINT | 佣金类型 |
| rate | DECIMAL(5,4) | 佣金比例 |
| fixed_amount | INT | 固定金额（分） |

**默认佣金规则：**
- 基础服务：15%
- 点钟加成：+5%（共 20%）
- 加钟服务：20%
- 卡项销售：5%
- 商品推荐：10%
- 好评奖励：¥10 固定

**集成点：**
- `BookingOrderServiceImpl.completeService()` → 触发佣金计算
- 技师提现复用 `BrokerageWithdraw` 通道

### 3.2 支付流程打通

**预约支付链路：**
```
用户下单 → BookingOrderServiceImpl.createOrder
  → PayOrderApi.createOrder（创建支付单）
  → 微信小程序支付（WxLitePayClient）
  → 支付回调 → BookingPayCallbackHandler
  → BookingOrderServiceImpl.payOrder（更新状态）
```

**新增：**
- `BookingPayCallbackHandler` — 实现支付回调处理，更新 BookingOrder 状态
- 修改 `createOrder` — 调用 `PayOrderApi.createOrder` 创建支付单

**退款链路：**
```
用户申请退款 → AfterSaleRefundDecisionService 判断
  → 自动退款（≤50元 + 非黑名单）→ PayRefundApi.createRefund
  → 人工审核 → ReviewTicket → 管理员处理 → PayRefundApi.createRefund
  → 退款回调 → 释放时间段 + 更新 BookingOrder 状态
```
