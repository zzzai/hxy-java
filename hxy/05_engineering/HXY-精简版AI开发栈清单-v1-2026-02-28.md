# HXY 精简版 AI 开发栈清单 v1（可直接落地）

## 1. 目标与边界

1. 目标：在 0-50 店阶段，用最小成本提升研发速度、回归质量、支付上线稳定性。
2. 边界：只保留“高收益、低维护”的 AI 能力，不引入重型自治平台。
3. 原则：流程先于工具，门禁先于上线，证据先于结论。

## 2. 必选栈（MVP）

### 2.1 过程层（必须启用）

1. `AGENTS.md` 作为唯一协作规则入口（已启用）。
2. Skill 流程最小集：
   - 设计变更前：`brainstorming`
   - 代码实现前：`test-driven-development`
   - 问题排查：`systematic-debugging`
   - 宣称完成前：`verification-before-completion`
3. 文档沉淀规则：所有阶段结论进入 `hxy/`，不放聊天记录里。

### 2.2 工具层（必须启用）

1. 代码代理：Codex（当前主流程）。
2. 本地命令执行：`script/dev/*` + Docker 运行栈。
3. 回归脚本：
   - 支付真链路：`ruoyi-vue-pro-master/script/dev/run_wx_partner_real_pay_refund.sh`
   - 对账门禁：`ruoyi-vue-pro-master/script/dev/check_reconcile_gate.sh`
   - 环境核验：`ruoyi-vue-pro-master/script/dev/check_wx_openid_chain_consistency.sh`

### 2.3 质量门禁（必须启用）

1. 后端构建通过：`mvn -pl yudao-server -am -DskipTests package`
2. 核心接口冒烟通过：商品、订单、优惠券 3 条主链路接口返回 `code = 0`
3. 支付上线门禁：`REQUIRE_RECONCILE=1` 执行，失败即禁止发布
4. 命名门禁：
   - 本地/流水线脚本：`ruoyi-vue-pro-master/script/dev/check_hxy_naming_guard.sh`
   - GitHub Workflow：`ruoyi-vue-pro-master/.github/workflows/hxy-naming-guard.yml`

## 3. 可选栈（第二阶段再上）

1. 向量检索记忆库（`pgvector` 或 Milvus）：仅在跨会话知识召回不足时启用。
2. 外部多代理编排（如 OpenHands）：当前阶段不推荐，运维成本高。
3. 全量社区模板包：不整包引入，只按需挑选 Hook/Command。

## 4. 项目内落地目录（统一）

1. 规范与流程：`hxy/05_engineering/`
2. 支付方案与联调：`hxy/03_payment/`
3. 路线图：`hxy/06_roadmap/`
4. 记忆归档：`hxy/07_memory_archive/`
5. 索引清单：`hxy/99_index/`

## 5. 两周落地计划（可执行）

### Week 1：先把“稳定开发闭环”做实

1. 固化命令入口：统一使用 `script/dev` 脚本，不再手工拼命令。
2. 支付门禁前置：每天联调结束跑一次对账门禁，异常当天清零。
3. 代码提交前检查：
   - 编译通过
   - 核心接口冒烟通过
   - 关键 SQL 脚本可回放
4. 文档标准化：每次改动补“交付物 + 验收 + 回滚”。

验收标准：
1. 连续 5 天无“未验证即合并”。
2. 支付链路脚本执行成功率 >= 95%（受外部商户限制除外）。

### Week 2：再补“效率与可追溯”

1. 建立发布清单模板：版本、SQL、配置、回滚步骤、责任人。
2. 关键变更引入“最小证据包”：
   - 命令
   - 输出摘要
   - 截图/日志路径
3. 新增命名门禁：禁止新增 `yudao/ruoyi/crmeb` 到新资产命名。
   - 门禁脚本：`ruoyi-vue-pro-master/script/dev/check_hxy_naming_guard.sh`

验收标准：
1. 每次发布均有证据包。
2. 新增文件命名违规为 0。

## 6. 你现在就能执行的 8 条清单

1. 所有开发任务先写入 `hxy/06_roadmap/` 对应阶段文档。
2. 代码改动前先跑一次当前分支构建，建立基线。
3. 任何支付相关改动后，固定跑 `check_wx_openid_chain_consistency.sh`。
4. 联调当天将 `REQUIRE_RECONCILE` 设置为 `1`，不通过不发版。
5. 每次新增 SQL 必须有“幂等 + 回滚说明”。
6. 小程序端改动必须保留“后端可达性”检查记录。
7. 关键业务流程（支付/退款/分账）必须保留审计日志。
8. 每周五更新一次路线图状态（完成项/风险项/下周计划）。

## 7. 最小化结论

1. 现在不需要引入重型外部 AI 平台。
2. 先把“技能流程 + 脚本门禁 + 文档证据”三件事做稳定，收益最大。
3. 等门店规模和需求复杂度上来，再逐步补向量记忆和多代理协作。
