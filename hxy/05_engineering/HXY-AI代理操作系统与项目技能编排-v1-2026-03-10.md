# HXY-AI代理操作系统与项目技能编排-v1-2026-03-10

## 1. 目标
把当前项目已经沉淀出来的文档治理、能力真值审计、发布门禁、领域收口、合规审计，固化为一套可发现、可组合、可验证、可追溯的项目专用代理操作系统，减少长提示词、减少窗口口径漂移、减少假完成。

## 2. 项目为什么需要这一层
当前仓库已经具备强文档基线和多窗口协作能力，但执行仍依赖人工下发长提示词。主要问题有：
1. 同一类任务反复重写提示词。
2. `ACTIVE / PLANNED_RESERVED / DEPRECATED` 容易脱离代码真值。
3. `Ready / Frozen` 容易被误解成“文档已写完”。
4. booking/member 域的 route、API、controller、文档经常需要人工对照。
5. 健康数据、标签、推荐类场景需要更强的合规边界。
6. A/B/C/D 窗口交接格式和联调注意点缺少固定约束。

## 3. 设计原则
### 3.1 渐进式披露
1. frontmatter 只写“什么时候用这个 skill”。
2. `SKILL.md` 只保留流程骨架与质量门槛。
3. 详细规则、模板、清单放在 `references/`。
4. 关键事实检查放进 `scripts/`，避免纯自然语言推断。

### 3.2 可组合性
不要做一个“大一统 HXY 总 skill”。
统一通过轻量路由选择最小技能链，例如：
- 能力台账审计 -> 文档冻结收口 -> 发布门禁判断
- booking 对齐 -> 能力台账回填 -> 文档冻结
- member 收口 -> Active/Planned 再判断 -> 发布门禁复核
- 健康数据合规审计 -> 冻结收口/回退 Ready

### 3.3 项目化成功标准
项目中的 skill 不是“写完文档”就算成功，必须按项目门禁判断：
1. Frozen 文档通配 API = 0
2. Frozen 文档 `TBD_*` 错误码 = 0
3. Frozen 文档原型别名 route = 0
4. `RESERVED_DISABLED` 误归类为 `ACTIVE` = 0
5. booking/member 已知真值漂移样本检出率 = 100%
6. `degraded_pool` 不得混入主 KPI/主 ROI 口径

### 3.4 关键验证步骤脚本化
以下项目事实必须优先用脚本/扫描确认：
1. uniapp route 真值
2. 前端 API path/method 真值
3. 后端 app controller 真值
4. 文档中的 canonical API / errorCode / 状态矩阵
5. `ACTIVE_SET` vs `PLANNED_RESERVED_SET`
6. Ready/Frozen 索引一致性

### 3.5 长记忆治理
所有代理规则必须能从仓库复原，不能只留在当前会话。
必需沉淀到：
1. `.codex/skills/`
2. `docs/plans/`
3. `hxy/05_engineering/`
4. `hxy/07_memory_archive/handoffs/`

## 4. 当前项目技能版图
### 4.1 核心执行技能
1. `hxy-miniapp-capability-auditor`
   - 审计小程序能力真值
   - 判断 `ACTIVE / PLANNED_RESERVED / DEPRECATED`
2. `hxy-miniapp-doc-freeze-closer`
   - 处理 Ready/Frozen 封版收口
   - 维护索引与冻结边界
3. `hxy-release-gate-decider`
   - 产出 `Go / Go with Gate / No-Go`
   - 校验 degraded、reserved-disabled、回滚条件
4. `hxy-booking-contract-alignment`
   - 对齐 booking 前端 API、后端 controller 与文档
5. `hxy-member-domain-closer`
   - 对齐 member 路由、页面、接口、PRD、contract
6. `hxy-health-data-compliance-guard`
   - 守住面诊、舌诊、体征、标签等健康数据红线

### 4.2 新增编排技能
7. `hxy-agent-workflow-router`
   - 负责选择正确 skill 与组合顺序
   - 不替代专用 skill，只负责路由与编排
8. `hxy-window-handoff-normalizer`
   - 固定 A/B/C/D 窗口回报格式
   - 固定 handoff 结构与联调注意点粒度

## 6. 统一真值读取层级
发生冲突时按以下顺序取单一真值：
1. 代码真值：真实页面、真实 `sheep/api`、真实 app controller
2. canonical contract：freeze、state machine、errorcode register、release API list
3. 产品真值：capability ledger、coverage matrix、release decision pack、ready-to-frozen review
4. 运行与治理真值：runbook、KPI 门禁、degraded_pool、reserved-disabled
5. handoff 与窗口输出：只作为执行证据，不反向覆盖前四层

## 7. A/B/C/D 协作规范
### 7.1 不按窗口做技能，按工作流做技能
窗口只是并行执行单元，skill 只围绕稳定工作流设计。

### 7.2 窗口固定回报结构
所有窗口默认使用以下 5 段固定输出：
1. `commit 列表（hash + message）`
2. `变更文件清单`
3. `验证命令与结果`
4. `handoff 文件路径`
5. `联调注意点（字段/错误码/降级行为）`

### 7.3 A 窗口职责
1. 汇总 B/C/D 输出
2. 只按 code/doc truth 收口，不按 message 意图收口
3. 维护 capability ledger、freeze 状态、release decision 单一真值

### 7.4 B/C/D 窗口职责
1. B：产品业务、字段字典、用户可见恢复动作
2. C：contract、errorCode、failureMode、fail-open/fail-close 边界
3. D：runbook、KPI、alert、灰度与回滚门禁

## 8. 协作与交互约束
1. 直接基于项目真值推进，不反复追问“是否继续”“是否需要”。
2. 不把“文档已生成”等价为“可发布”。
3. 任何完成声明之前，必须先跑验证命令。
4. 禁止用 message 分支代替 errorCode/failureMode 分支。
5. 不允许把 `PLANNED_RESERVED` 当成 `ACTIVE` 去写 Frozen 或 release decision。

## 10. 后续扩展原则
满足下列条件之一，再新增 skill：
1. 同类任务一周内反复出现 3 次以上
2. 需要复用稳定流程且包含 3 步以上顺序工作流
3. 单靠文档难以稳定执行，必须引入脚本或模板
4. 有清晰边界，不会侵蚀现有 skill 职责

## 12. 结论
HXY 当前最需要的不是再写更多长提示词，而是让代理遵循项目自己的工作流、真值边界和验证门禁工作。项目技能体系的作用，是把多轮实践中已经跑通的方法，变成以后默认执行的系统能力。
