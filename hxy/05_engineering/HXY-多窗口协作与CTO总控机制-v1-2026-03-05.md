# HXY 多窗口协作与 CTO 总控机制 V1（2026-03-05）

## 1. 目标

1. 锁定窗口协作的分支与交付纪律，避免“分支混乱/重复开发/覆盖提交”。
2. 由窗口 A 承担 CTO 总控职责，统一产品与研发节奏。
3. 支撑窗口 B 及后续新增窗口的可持续并行协作。

## 2. 分支治理（当前生效）

1. 当前迭代锁定主开发分支：`feat/ui-four-account-reconcile-ops`。
2. 窗口 A、B 及新增窗口默认以该分支为基线，不得私自切换到其他业务分支。
3. 若出现高冲突并行任务，允许创建临时分支 `wip/<window>/<topic>`（需窗口 A 审批），合入仅允许 `cherry-pick` 回锁定分支。
4. `main` 只接受经过窗口 A 最终验收的 PR，不允许窗口直接推送上线分支。

## 3. 角色职责

### 3.1 窗口 A（CTO / 集成窗口）

1. 负责产品方案一致性、架构边界与优先级决策。
2. 维护全局 Roadmap、里程碑、风险清单与发布门禁。
3. 负责跨窗口集成、冲突裁决、最终验收与提交策略。
4. 对外输出统一状态：完成项、阻塞项、下一批任务。

### 3.2 窗口 B..N（执行窗口）

1. 严格按窗口 A 分配的任务边界开发，不跨域扩写。
2. 每次交付必须提供：commit 列表、改动文件、验证结果、handoff 路径、是否可 cherry-pick。
3. 未经窗口 A 明确指令，不得改动核心治理文档与发布脚本。

## 4. 协作协议（强制）

1. 任意窗口开始开发前，先声明“任务范围 + 禁改范围 + 验证命令”。
2. 同一时间同一文件只允许一个窗口写入；冲突文件由窗口 A 重新切分。
3. 每批交付必须包含可回滚点：SQL、接口、配置至少一项可回退说明。
4. 所有关键变更同步回写三份核心记忆文档与 handoff 文档。

## 5. CTO 节奏（默认每日）

1. 09:30：确认当日 P0/P1 任务与窗口分工。
2. 14:00：中段评审（进度、质量、风险、偏差纠正）。
3. 19:00：收口评审（验收结果、遗留问题、次日计划）。
4. 每次评审后更新执行看板与风险状态。

## 6. 新窗口接入流程

1. 窗口 A 分配唯一窗口代号与任务边界。
2. 接收最新 handoff 与三份核心记忆文档。
3. 完成首个批次后，必须通过命名门禁与记忆门禁。
4. 连续两批通过后，才可承担跨模块任务。

## 7. 执行原则基线（强制）

1. 最佳实践优先：技术决策默认服从稳定性、一致性、可回滚、可审计，不以短期速度替代长期质量。
2. 客观冷静：先确认事实与证据，再推进改动与结论；禁止基于猜测同步“完成状态”。
3. 第一性原理：取舍顺序固定为 `系统稳定性 > 数据一致性 > 可回滚性 > 可审计性 > 交付速度`。
4. 真实透明：所有窗口仅报告已执行且可验证的结果，失败与阻塞必须显式披露并附定位信息。
5. 冲突裁决：当任务指令与门禁冲突时，窗口 A 以门禁和工程基线优先，不做静默放行。
