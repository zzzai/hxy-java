# HXY 并行后端开发（2/3 窗口）实操 v1

更新日期：2026-03-01

## 1. 是否使用 `dispatching-parallel-agents`（先判断）

只在任务满足“相互独立”时使用并行：

1. 改动文件边界独立（不会频繁改同一 `Service/Mapper/SQL`）
2. 数据模型不共享迁移顺序（不会互相阻塞）
3. 验证链路可拆分（每条子任务有独立测试命令）

不满足以上条件时，顺序开发更快，冲突更少。

## 2. 本项目推荐并行切分

推荐并行 3 车道（最多）：

1. 车道1：主链路能力（Service/Controller）
2. 车道2：测试与 SQL 迁移
3. 车道3：定时任务与可观测脚本

## 3. 一键打开 2/3 窗口（tmux + worktree）

脚本：

`ruoyi-vue-pro-master/script/dev/start_hxy_parallel_backend_tmux.sh`

示例：

```bash
cd /root/crmeb-java
ruoyi-vue-pro-master/script/dev/start_hxy_parallel_backend_tmux.sh --lanes 3
```

常用参数：

1. `--lanes 2`：只开 2 个窗口
2. `--session hxy-hotfix`：指定会话名
3. `--with-worktree 0`：不用 worktree（不推荐）
4. `--attach 0`：只创建会话不进入

## 4. 手工操作（不用脚本时）

```bash
tmux new -s hxy-be-dev -n lane1
tmux new-window -t hxy-be-dev -n lane2
tmux new-window -t hxy-be-dev -n lane3
tmux attach -t hxy-be-dev
```

## 5. 协作约束（必须）

1. 同一时段，同一文件只允许一个车道修改。
2. SQL 迁移统一在车道2产出，避免重复迁移脚本。
3. 每车道提交前必须运行本车道测试；合并前跑总测试。
4. 先合并数据模型，再合并业务逻辑，最后合并任务/脚本。
