# HXY 支付模式定版（双路径）V1

- 日期：2026-02-22
- 适用阶段：0-100 家门店
- 适用底座：`/root/crmeb-java/ruoyi-vue-pro-master`

## 1. 决策结论

1. 长期主路径：`服务商 + 特约商户 + 分账`（门店收款，总部分账抽佣）。
2. 早期过渡路径：`总部统一收款 + 虚拟账户台账 + 周期结算`，仅用于小规模快速起盘。
3. 预收款（次卡/储值/疗程）在 0-100 店默认“单店隔离”，不做跨店通用。
4. 100 店后再评估“预收款跨店通用”与专门合规方案（托管/备付金/法务评审）。

## 2. 路径选择规则

1. 直营同主体（同一法人）：
   可先用“总部统一收款过渡路径”快速上线，再切主路径。
2. 加盟或独立法人门店：
   优先走“服务商 + 特约商户主路径”，不建议长期总部统收后线下分拨。
3. 如需总部先收后分（品牌统收）：
   应使用微信支付官方品牌供应链分账能力，而非长期手工转账。

## 3. 资金流定版

1. 主路径（推荐）：
   用户支付 -> 子商户（门店）收款 -> 总部发起分账 -> 总部抽佣/平台费 + 技师收益 + 门店留存。
2. 过渡路径（小规模）：
   用户支付 -> 总部商户收款 -> 系统写入门店/技师应得台账 -> 周期结算出款。
3. 企业账户（B 端）：
   企业资金入总部对公账户 -> 系统授信额度 -> 员工消费按即时消费链路处理。

## 4. 官方规则约束（按 2026-02-22 核验）

1. 分账请求是异步处理，必须“受理 + 查询”收敛终态。
2. 分账回退与退款需要闭环编排，不能只做单接口成功判断。
3. 服务商模式交易必须携带并保持 `sub_mchid` 一致性。
4. 特约商户进件应 API 化（申请、状态查询、审核回执）。
5. 品牌供应链分账支持“门店收款抽佣”和“品牌统收后分给门店”两类场景。

## 5. 风险与边界

1. 预收款跨店是“清分规则 + 合规约束”问题，不是纯支付接口问题。
2. 技师“实时到个人”能力受微信支付产品规则和账户状态约束，不能写死阈值。
3. 过渡路径若用于加盟场景，存在长期财税和清结算争议风险，必须设置退出时限。

## 6. 系统落地要求

1. 订单模型增加 `fund_mode`：
   - `DIRECT_PAY_STORE`（门店收款）
   - `DIRECT_PAY_HQ`（总部统收）
   - `PREPAID_STORE_LOCKED`（预收款单店隔离）
2. 账务台账最小集合：
   - 门店应得台账（交易明细、已结算、待结算）
   - 技师应得台账（提成明细、已结算、待结算）
   - 结算记录（结算批次、出款流水、复核人）
3. 收银模式默认用“先建订单再支付码”，避免“支付成功后反匹配订单”。
4. 退款链路固定为：
   退款申请 -> 分账回退（如适用） -> 渠道退款 -> 业务逆向补偿 -> 对账收敛。
5. 上线拦截规则增加：
   - 分账超时率
   - 分账回退失败积压
   - 退款终态超时积压
   - 结算批次未复核出款

## 7. 分阶段实施（执行版）

1. 0-10 家：
   允许“总部统收过渡路径”，快速跑通订单/支付/台账/手工结算；预收款单店隔离。
2. 10-30 家：
   切换主路径，完成特约商户进件和分账自动化；保留过渡路径仅作兜底。
3. 30-100 家：
   完成分账/退款补偿自动化，建立技师收益策略和灰度压测。
4. 100 家+：
   决策是否启动预收款跨店模型（含法务税务评审与资金方案）。

## 8. 对当前项目的直接影响

1. 本周优先级不变：支付稳定性、自动化和可审计优先，非支付后置。
2. 当前版本不做预收款跨店；需求统一标记“战略待评估”。
3. 门店保持履约职责，总部负责资金动作、规则执行和审计。
