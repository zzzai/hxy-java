# HXY 支付场景适配评估 V1（2026-02-22）

## 1. 项目背景匹配结论

1. 当前“总部统一支付中台 + 门店履约”方向与荷小悦连锁模式匹配。
2. 当前支付链路（下单/回调/退款/对账/补偿）已具备从单店到连锁扩展的基础形态。
3. 关键缺口在“服务商 + 特约商户 + 分账结算 + 门店经营账”联动完整性。
4. 0-10 家阶段可采用“总部统收 + 虚拟账户台账”过渡，但需设置退出到主路径的条件。

## 2. 现状符合项

1. 具备支付状态机与幂等治理。
2. 具备异常补偿机制（回调丢失、重试、超时）。
3. 具备上线拦截规则与回滚约束。

## 3. 万店场景必须补齐项

1. 服务商模式全链路压测（含高峰回调风暴场景）。
2. 总部结算账与门店经营账分层（避免“交易账替代结算账”）。
3. 退款策略模型分层（秒退、复核、人工升级）与 SLA 量化。
4. 统一账单中心：支付账、交易账、结算账、税务口径可对齐。
5. 预收款跨店结算单独建模，不与即时消费分账混做。

## 4. 技术落地建议

1. 将支付通知与对账任务从“脚本主导”升级为“服务主导 + 任务编排”。
2. 以 K8s CronJob + 队列消费替代部分单机 cron，提升可观测和可恢复能力。
3. 增加支付 SLO：成功率、终态收敛时延、异常积压、对账差异率。
4. 在 0-100 店阶段，预收款默认单店隔离，跨店策略延期到独立合规评审后启用。
5. 个人收益实时分发能力不得写死固定阈值，必须以商户平台实时规则校验。

## 5. API verification log（2026-02-22）

- Trigger area(s): `pay`（并覆盖 login/phone/profile 交叉边界）
- Official URL(s) checked:
  - https://developers.weixin.qq.com/miniprogram/dev/api/payment/wx.requestPayment.html
  - https://developers.weixin.qq.com/miniprogram/dev/api/open-api/login/wx.login.html
  - https://developers.weixin.qq.com/miniprogram/dev/api-backend/open-api/login/auth.code2Session.html
  - https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/getPhoneNumber.html
  - https://developers.weixin.qq.com/miniprogram/dev/api-backend/open-api/phonenumber/phonenumber.getPhoneNumber.html
  - https://developers.weixin.qq.com/miniprogram/dev/api/open-api/user-info/wx.getUserInfo.html
  - https://developers.weixin.qq.com/miniprogram/dev/api/open-api/user-info/wx.getUserProfile.html
- Constraints relevant to implementation:
  - 小程序支付前端必须通过 `wx.requestPayment` 拉起，不应改为扫码收款链路。
  - 登录态建立与手机号能力仍应走官方推荐链路，后端接口必须与前端能力匹配。
  - 支付结果不能仅信任前端回调，必须以后端支付回调与查询收敛为准。
  - 敏感信息与支付参数必须在服务端签名生成，前端只消费最小必要参数。

## 6. API verification log（2026-02-22，门禁脚本收口补充）

- Trigger area(s): `pay`（支付门禁脚本与回放脚本）
- Official URL(s) checked:
  - https://developers.weixin.qq.com/miniprogram/dev/api/payment/wx.requestPayment.html
  - https://developers.weixin.qq.com/miniprogram/dev/api/open-api/login/wx.login.html
  - https://developers.weixin.qq.com/miniprogram/dev/OpenApiDoc/user-login/code2Session.html
- Constraints relevant to implementation:
  - 支付能力仍以 `wx.requestPayment` 为前端唯一拉起入口，门禁脚本不能引导为非官方收银调用链。
  - 登录换取 `openid/session_key` 的服务端链路仍需基于 `wx.login + code2Session`，脚本检查不得偏离该约束。
  - 发布门禁放行仅代表“工程链路健康”，不替代真实商户参数与支付回调验证。
