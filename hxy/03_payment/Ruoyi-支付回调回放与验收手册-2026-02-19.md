# Ruoyi 支付回调回放与验收手册（2026-02-19）

## 1. 目标

- 复现并验证 3 类关键异常场景：
  - 重复回调（duplicate）
  - 乱序回调（out_of_order）
  - 延迟回调（delayed）
- 通过 SQL 验收项判断链路是否可收敛（`BLOCK/WARN`）。

## 3. 准备样本

### 3.1 请求体样本

- 从微信支付实际回调日志中提取原始 body，保存为：
  - `/tmp/pay_notify_success.json`
  - `/tmp/pay_notify_closed.json`（乱序场景可选）

### 3.2 请求头样本

- 逐行保存到 `/tmp/pay_notify_headers.txt`，格式：
```txt
Wechatpay-Timestamp: 1700000000
Wechatpay-Nonce: xxxxx
Wechatpay-Signature: xxxxx
Wechatpay-Serial: xxxxx
```

### 3.3 Query 参数（可选）

- 保存到 `/tmp/pay_notify_query.txt`，格式：
```txt
foo=bar
baz=qux
```

## 4. 执行回放

### 4.1 重复回调

```bash
bash ruoyi-vue-pro-master/script/dev/replay_pay_notify.sh \
  --base-url https://api.hexiaoyue.com/admin-api/pay/notify \
  --biz-type order \
  --channel-id 18 \
  --scenario duplicate \
  --repeat-times 3 \
  --first-body-file /tmp/pay_notify_success.json \
  --header-file /tmp/pay_notify_headers.txt
```

### 4.2 乱序回调

```bash
bash ruoyi-vue-pro-master/script/dev/replay_pay_notify.sh \
  --base-url https://api.hexiaoyue.com/admin-api/pay/notify \
  --biz-type order \
  --channel-id 18 \
  --scenario out_of_order \
  --first-body-file /tmp/pay_notify_success.json \
  --second-body-file /tmp/pay_notify_closed.json \
  --header-file /tmp/pay_notify_headers.txt
```

### 4.3 延迟回调

```bash
bash ruoyi-vue-pro-master/script/dev/replay_pay_notify.sh \
  --base-url https://api.hexiaoyue.com/admin-api/pay/notify \
  --biz-type order \
  --channel-id 18 \
  --scenario delayed \
  --delay-seconds 180 \
  --first-body-file /tmp/pay_notify_success.json \
  --header-file /tmp/pay_notify_headers.txt
```

## 6. 结果判定

- `BLOCK`：必须先修复，不可放行。
- `WARN`：可继续观察，但建议先收敛后放行。
- 无输出：本次回放验收通过（当前规则集下无异常）。

## 7. 典型阻断项（BLOCK）说明

- `B03_ORDER_CLOSED_BUT_SUCCESS_EXTENSION`：
  - 订单关闭态与成功扩展态冲突，属于状态机逆流。
- `B04_MULTI_SUCCESS_EXTENSION`：
  - 同一订单出现多个成功扩展，存在幂等风险。
- `B06_NOTIFY_TASK_STUCK`：
  - 通知任务未在宽限期内收敛。
- `B07_NOTIFY_TIMES_EXCEEDED`：
  - 重试次数越上限，需检查任务调度/去重逻辑。

## 8. CI 触发方式（可选）

1. 打开 GitHub Actions，选择 `pay-notify-replay-acceptance`。
2. 填写 `merchant_order_id`，选择是否 `run_replay`。
3. 若 `run_replay=true`，请把请求体/请求头换成真实样本（默认 sample 仅作模板）。
4. 运行结束后下载 artifact，优先查看：
   - `artifact_index.md`
   - `summary.txt`
   - `pay_notify_replay_report.md`
   - `tickets/tickets.tsv`（若有 BLOCK 或开启 WARN 工单）
   - `tickets/notify_summary.txt`（通知结果）
   - `logs/final_gate.log`（最终阻断判定日志）
   - `logs/acceptance.log`
5. `artifact_index.md` 已包含 report/tickets/final_gate 的文件入口，可直接点击定位。
6. Actions 页面 `Step Summary` 已展示关键结论（订单号、BLOCK/WARN、通知状态）。

## 9. 通知配置（可选）

可在 GitHub Actions Secrets 配置以下键：
1. `PAY_REPLAY_TICKET_FEISHU_WEBHOOK`：飞书机器人 webhook。
2. `PAY_REPLAY_TICKET_WEBHOOK`：通用 webhook（可由中转服务接 Jira）。

说明：
1. 若上述 secrets 未配置，通知步骤会 `SKIP`，不会阻断回放验收。
2. 通知执行结果会落到：
   - `tickets/notify_summary.txt`
   - `logs/ticket_notify.log`
